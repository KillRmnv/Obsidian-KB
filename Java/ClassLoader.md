**ClassLoader** (загрузчик классов) — это специальный компонент среды выполнения Java (JRE), который отвечает за динамическую загрузку классов в память JVM во время работы программы.

В Java классы не загружаются все сразу при старте. Вместо этого ClassLoader находит нужный `.class` файл на диске (или в сети), считывает его байт-код и создает экземпляр объекта `java.lang.Class`.

---

### 1. Иерархия загрузчиков

В стандартной Java используется трехуровневая модель загрузки. Каждый загрузчик имеет своего «родителя»:

1. **Bootstrap ClassLoader:** «Прародитель». Написан на нативном коде (C++). Загружает основные классы JDK (например, `java.lang.*`, `java.util.*`) из внутренней папки `/lib`.
    
2. **Platform ClassLoader** (ранее Extension): Загружает расширения Java и вспомогательные модули из `/lib/ext`.
    
3. **System/Application ClassLoader:** Самый используемый. Загружает классы, которые написали вы, и библиотеки, указанные в вашем **Classpath**.
    

---

### 2. Принцип делегирования (Delegation Model)

Это ключевое правило безопасности и стабильности Java. Когда загрузчику нужно найти класс, он действует по алгоритму:

- **Спросить родителя:** «Ты уже загружал этот класс или можешь его загрузить?»
    
- **Если родитель не смог:** Только тогда загрузчик пытается найти класс сам.
    

**Зачем это нужно?** Чтобы вы не могли «подменить» системные классы. Если вы создадите свой класс `java.lang.String`, System ClassLoader сначала спросит Bootstrap. Тот ответит: «Я уже загрузил настоящую строку из JDK», и ваша версия будет проигнорирована.

---

### 3. Три фазы работы ClassLoader

Процесс превращения файла в объект класса в памяти делится на этапы:

1. **Loading (Загрузка):** Поиск файла и создание бинарного представления.
    
2. **Linking (Связывание):**
    
    - _Verification:_ Проверка байт-кода на валидность (безопасность).
        
    - _Preparation:_ Выделение памяти под статические поля и их инициализация значениями по умолчанию.
        
    - _Resolution:_ Замена символьных ссылок на прямые адреса в памяти.
        
3. **Initialization (Инициализация):** Выполнение статических блоков инициализации и присвоение значений статическим полям.
    

---

### 4. Зачем создавать свой ClassLoader?

Обычно стандартных механизмов хватает, но кастомные загрузчики нужны для:

- **Горячей замены кода (HotSwap):** Загрузка новой версии класса без перезагрузки сервера.
    
- **Загрузки из необычных источников:** Например, получение байт-кода из базы данных или расшифровка его «на лету» (если он зашифрован для защиты интеллектуальной собственности).
    
- **Изоляции библиотек:** В серверах приложений (Tomcat, WildFly) разные приложения могут использовать разные версии одной и той же библиотеки, не конфликтуя друг с другом.
    

---

Написание собственного загрузчика классов обычно требуется, когда стандартный `AppClassLoader` не справляется — например, если нужно загрузить `.class` файл из специфической папки, зашифрованного архива или даже из базы данных.

Ниже приведен пример **кастомного ClassLoader**, который считывает байт-код из файла в произвольной директории.

---

### 1. Создание своего ClassLoader

Чтобы создать свой загрузчик, нужно наследоваться от `java.lang.ClassLoader` и переопределить метод `findClass()`.

> **Важно:** Мы переопределяем именно `findClass`, а не `loadClass`, чтобы сохранить **принцип делегирования**. Таким образом, системные классы по-прежнему будет загружать родитель.



```
import java.io.*;
import java.nio.file.*;

public class MyCustomLoader extends ClassLoader {
    private String classPath;

    public MyCustomLoader(String classPath) {
        this.classPath = classPath;
    }

    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        try {
            // Превращаем имя пакета (com.example.Test) в путь (com/example/Test.class)
            String fileName = name.replace('.', File.separatorChar) + ".class";
            byte[] classBytes = Files.readAllBytes(Paths.get(classPath, fileName));
            
            // Превращаем массив байтов в объект Class
            return defineClass(name, classBytes, 0, classBytes.length);
        } catch (IOException e) {
            throw new ClassNotFoundException("Не удалось найти класс: " + name, e);
        }
    }
}
```

---

### 2. Использование загрузчика

Допустим, у нас есть скомпилированный файл `TestClass.class` в папке `C:/my_classes/`. Вот как мы его загрузим и вызовем метод:

Java

```
public class LoaderDemo {
    public static void main(String[] args) {
        try {
            // 1. Создаем экземпляр нашего загрузчика
            MyCustomLoader loader = new MyCustomLoader("C:/my_classes/");

            // 2. Загружаем класс по имени
            Class<?> clazz = loader.loadClass("com.example.TestClass");

            // 3. Создаем объект (через рефлексию)
            Object obj = clazz.getDeclaredConstructor().newInstance();

            // 4. Выводим, кто загрузил этот класс
            System.out.println("Класс загружен через: " + clazz.getClassLoader());
            
            // Вызов метода, если он есть
            clazz.getMethod("sayHello").invoke(obj);

        } catch (Exception e) {
            e.printStackTrace();
        }
    }
}
```

---

### Что здесь происходит?

1. **`loadClass()`**: Вызывается первым. Он сначала идет к родителю (System ClassLoader). Если тот не находит класс (а в обычном `classpath` его нет), вызывается наш `findClass()`.
    
2. **`defineClass()`**: Это «магический» нативный метод внутри JVM. Он берет массив байтов и превращает его во внутреннюю структуру класса в памяти.
    
3. **Изоляция**: Если вы создадите два экземпляра `MyCustomLoader`, они смогут загрузить один и тот же класс `com.example.TestClass` независимо. Для JVM это будут **два разных класса**, даже если их байт-код идентичен.
