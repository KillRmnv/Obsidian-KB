**FunctionalInterface** (функциональный интерфейс) — это интерфейс, который содержит **ровно один абстрактный метод**.

Основная цель таких интерфейсов в Java — поддержка **лямбда-выражений** и **ссылок на методы (method references)**. Они позволяют передавать поведение (логику) как данные.

---

### 1. Ключевые отличия

|**Характеристика**|**Обычный интерфейс**|**Функциональный интерфейс**|
|---|---|---|
|**Количество абстрактных методов**|Любое количество (0, 1, 10...)|**Строго один**|
|**Лямбда-выражения**|Нельзя использовать (если методов > 1)|Идеально подходит для лямбд|
|**Аннотация `@FunctionalInterface`**|Обычно не используется|Настоятельно рекомендуется|
|**Назначение**|Описание контракта поведения объекта|Описание конкретного действия/функции|

---

### 2. Правила и структура

Несмотря на требование «один метод», функциональный интерфейс может содержать и другие элементы, которые **не считаются** абстрактными:

1. **Default методы:** Реализованные методы с ключевым словом `default`.
    
2. **Static методы:** Статические методы с реализацией.
    
3. **Методы класса Object:** Если вы переопределяете методы типа `public boolean equals(Object obj)`, они не идут в зачет «единственного» метода.
    

---

### 3. Аннотация `@FunctionalInterface`

Эта аннотация не является обязательной, но она выполняет роль «предохранителя»:

- Если вы пометите интерфейс этой аннотацией и случайно добавите второй абстрактный метод, **компилятор выдаст ошибку**.
    
- Она сообщает другим разработчикам, что этот интерфейс предназначен для использования в лямбдах.
    

---

### 4. Пример кода

**Обычный интерфейс:**

Java

```
interface MultimediaPlayer {
    void play();
    void stop(); // Два метода — лямбду использовать нельзя
}
```

**Функциональный интерфейс:**

Java

```
@FunctionalInterface
interface Validator {
    boolean isValid(String value); // Ровно один абстрактный метод
}

// Теперь мы можем использовать лямбду:
Validator emailValidator = (str) -> str.contains("@");
```

---

### 5. Встроенные функциональные интерфейсы

В Java (пакет `java.util.function`) уже есть набор стандартных интерфейсов для 99% задач:

- **`Predicate<T>`**: Принимает `T`, возвращает `boolean` (тестирование условия).
    
- **`Function<T, R>`**: Принимает `T`, возвращает `R` (преобразование).
    
- **`Consumer<T>`**: Принимает `T`, ничего не возвращает (потребление/действие).
    
- **`Supplier<T>`**: Ничего не принимает, возвращает `T` (поставка данных).
    

---

### Зачем это нужно?

Без функциональных интерфейсов нам приходилось писать громоздкие **анонимные классы**:

Java

```
// Старый стиль
button.addActionListener(new ActionListener() {
    @Override
    public void actionPerformed(ActionEvent e) {
        System.out.println("Клик!");
    }
});

// Новый стиль (с использованием функционального интерфейса ActionListener)
button.addActionListener(e -> System.out.println("Клик!"));
```
