
## 1. Концептуальные основы ООП

**Определение (расширенное):**  
Объектно-ориентированное программирование — это парадигма проектирования и реализации систем, основанная на представлении вычислений как совокупности взаимодействующих сущностей (объектов), каждая из которых инкапсулирует:

- **Состояние** (данные, атрибуты) — текущее значение характеристик объекта
    
- **Поведение** (методы) — операции, которые объект может выполнить или применить к себе
    
- **Идентичность** (уникальный идентификатор) — возможность различия объектов даже при одинаковом состоянии
    

**Центральная идея:** Моделирование программной системы как экосистемы автономных, взаимодействующих сущностей, каждая из которых отвечает за определённый аспект системы.

## 2. Три столпа организации в ООП

## 2.1 Единство данных и методов

**В процедурном подходе (антипаттерн для моделирования):**

text

`Данные (отдельно):        Функции (отдельно): - balance: 1000          - deposit(amount) - accountId: 12345       - withdraw(amount) - owner: "John"          - calculateInterest()`

Данные и логика разнесены. Функция может нарушить инварианты данных (например, `balance` станет отрицательным).

**В ООП (правильный подход для моделирования):**

text

`Объект BankAccount:   Состояние:  - balance: 1000  - accountId: 12345  - owner: "John"     Поведение:  - deposit(amount)      [гарантирует: balance ≥ 0]  - withdraw(amount)     [гарантирует: balance ≥ 0]  - getBalance()         [защищает данные]`

**Смысл для моделирования:** Объект — это **контейнер инвариантов**. Объект гарантирует, что его данные всегда в консистентном состоянии, потому что только его методы могут их менять.

## 2.2 Иерархическая организация (классификация)

Объекты организованы в иерархии на основе отношения «является» (is-a).

**Пример иерархии животных:**

text

`Animal (суперкласс)   ├─ Mammal (млекопитающее)  │   ├─ Dog  │   ├─ Cat  │   └─ Human  └─ Bird (птица)      ├─ Eagle      └─ Penguin`

**Смысл для моделирования:** Позволяет структурировать знание о системе от общего к частному. Все млекопитающие имеют молочные железы (общее свойство), но собака лает, а кошка мяукает (частное поведение).

## 2.3 Полиморфизм через единый интерфейс

Разные объекты в иерархии реагируют на одно и то же сообщение по-разному.

text

`для каждого животного в списке:   животное.издать_звук()`

Вызов один, а результаты разные:

- Dog издаст: "Woof"
    
- Cat издаст: "Meow"
    
- Bird издаст: "Tweet"
    

**Смысл для моделирования:** Уменьшает связанность между компонентами системы. Главный цикл не знает, какие конкретно животные в списке — он просто просит их издать звук.

## 3. Класс и объект: формальное различие

## Класс (Class)

**Определение:** Шаблон (тип) данных, описывающий структуру и поведение совокупности объектов. Класс — это **абстракция, концепция, идея**.

**Компоненты класса:**

- Атрибуты (члены данных) — переменные, которые хранят состояние
    
- Методы — функции, которые определяют поведение
    
- Модификаторы доступа (`public`, `private`, `protected`) — правила видимости
    

## Объект (Object)

**Определение:** Конкретный экземпляр класса в памяти программы. Объект — это **реализация, воплощение, конкретная вещь**.

**Аналогия:**

- Класс — это чертёж дома
    
- Объект — это построенный дом
    

Из одного чертежа (класса `House`) можно построить много домов (объектов `house1`, `house2`, `house3`), и каждый будет иметь своё состояние (в доме 1 живёт семья из 4 человек, в доме 2 — из 2 человек).

## 4. Идентичность, состояние и поведение

Три фундаментальных аспекта объекта:

## Идентичность (Identity)

**Определение:** Уникальный идентификатор объекта, который остаётся неизменным на протяжении его жизненного цикла.

**Свойство:** Два объекта считаются **разными**, даже если у них одинаковое состояние, если у них разные идентичности.

**Пример:**

text

`Объект user1: name="Alice", age=25 Объект user2: name="Alice", age=25 user1 ≠ user2 (разные идентичности, хотя данные одинаковые) user1 == user1 (одна и та же идентичность)`

## Состояние (State)

**Определение:** Совокупность значений всех атрибутов объекта в данный момент времени.

**Свойство:** Состояние изменяется при вызове методов объекта.

**Пример:**

text

`Начальное состояние account: balance=1000, transactions=[] После вызова account.withdraw(100): Новое состояние account: balance=900, transactions=[100]`

## Поведение (Behavior)

**Определение:** Набор операций (методов), которые объект может выполнить и которые изменяют его состояние или вычисляют значения на основе состояния.

**Характеристика:** Поведение определяется **классом**, а не объектом. Все объекты одного класса имеют одинаковое поведение (тот же набор методов).

**Пример:**

text

`Класс BankAccount определяет методы:   - deposit(amount)  - withdraw(amount)  - getBalance() Объект myAccount может выполнить эти методы Объект yourAccount может выполнить эти же методы`

## 5. Жизненный цикл объекта

Объект существует через несколько фаз:

## Фаза 1: Создание (Instantiation)

Объект создаётся через конструктор класса. Начальное состояние устанавливается.

text

`Создание: account = new BankAccount("John", 1000)`

## Фаза 2: Использование (Active Life)

Объект взаимодействует с другими объектами, его состояние изменяется.

text

`account.deposit(500) account.withdraw(200) balance = account.getBalance()`

## Фаза 3: Уничтожение (Destruction/Garbage Collection)

Объект больше не используется, память, которую он занимал, освобождается.

## 6. Взаимодействие объектов в системе

## 6.1 Концепция сообщений

В ООП объекты не просто «вызывают функции» — они **отправляют сообщения** друг другу.

**Определение:** Сообщение — это запрос на выполнение операции, отправленный одним объектом другому.

**Парадигма:**

text

`Объект A отправляет сообщение объекту B:   A.method(parameters) B выполняет операцию и может вернуть результат.`

**Смысл для моделирования:** Объекты взаимодействуют через чётко определённые интерфейсы, что повышает модульность.

## 6.2 Ассоциации между объектами

**Композиция** (Composition): объект содержит другие объекты как части.

text

`Объект Car содержит объекты Engine, Wheel, Transmission Car "состоит из" этих компонентов`

**Агрегация** (Aggregation): объект содержит ссылки на другие объекты, но не "владеет" ими полностью.

text

`Объект Team содержит ссылки на объекты Player Team "содержит" игроков, но игроки могут существовать независимо`

**Ассоциация** (Association): простая связь между двумя объектами.

text

`Объект Person "знает" объект Company (работает в ней)`

## 7. Инкапсуляция как основной механизм управления сложностью

**Эффект инкапсуляции:**

|Без инкапсуляции|С инкапсуляцией|
|---|---|
|Внешний код может изменять данные напрямую: `account.balance = -1000` (нарушение логики)|Внешний код может только вызывать методы: `account.withdraw(100)` (логика гарантирует корректность)|
|Трудно менять внутреннюю реализацию (много кода зависит от структуры данных)|Легко менять реализацию метода, не меняя интерфейс|
|Высокая связанность (coupling) между компонентами|Низкая связанность; компоненты почти независимы|

## 8. Наследование как механизм переиспользования и иерархизации

## Цели наследования:

1. **Переиспользование кода:** Подкласс наследует методы суперкласса, не переписывая их
    
2. **Специализация:** Подкласс переопределяет методы суперкласса для более специфичного поведения
    
3. **Полиморфизм:** Объекты разных подклассов могут обрабатываться единообразно через суперкласс
    

## Отношение "является"

text

`Dog является Animal Circle является Shape Student является Person`

Если B является A, то B наследуется от A, и B получает все характеристики A, добавляя свои.

## 9. Полиморфизм: единство и многообразие

**Определение:** Способность объектов разных типов обрабатывать одно и то же сообщение (вызов метода) с разными реализациями.

**Виды полиморфизма:**

## Полиморфизм подтипов (Subtype Polymorphism)

Основан на наследовании. Подкласс переопределяет методы суперкласса.

text

`Shape shape = new Circle();  // shape указывает на Circle shape.draw();               // вызывается метод Circle.draw() shape = new Square();       // shape указывает на Square shape.draw();               // вызывается метод Square.draw()`

## Параметрический полиморфизм (Parametric Polymorphism)

Метод или класс работает с параметрами типа (generics в Java).

text

`List<Integer> numbers = new ArrayList<Integer>(); List<String> names = new ArrayList<String>(); // Один класс List работает с разными типами данных`

## 10. Преимущества ООП для моделирования систем

1. **Адекватность модели:** Объекты соответствуют реальным сущностям системы
    
2. **Управляемость сложности:** Инкапсуляция разбивает большую систему на управляемые части
    
3. **Переиспользование:** Иерархия классов позволяет использовать общий код множество раз
    
4. **Гибкость:** Полиморфизм позволяет добавлять новые типы объектов без изменения существующего кода
    
5. **Тестируемость:** Объекты изолированы и могут тестироваться независимо
    
6. **Эволюция системы:** Легче добавлять новые функции и менять старые
    

---

## 11. Ограничения ООП

1. **Усложнение простых задач:** Для простых скриптов ООП может быть избыточным
        
2. **Производительность:** В некоторых случаях сложность объектной модели замедляет выполнение
    
3. **Глубокие иерархии:** Если иерархия классов становится слишком глубокой, система становится сложнее, а не проще
