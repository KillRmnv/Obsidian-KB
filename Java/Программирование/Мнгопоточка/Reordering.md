## 2. Reordering (Переупорядочивание)

### Что это?
**Reordering** — изменение порядка выполнения инструкций компилятором или процессором для оптимизации, которое может нарушить многопоточную семантику.

### Почему происходит?
1. **Компилятор** переупорядочивает инструкции для оптимизации
2. **Процессор** выполняет инструкции вне порядка (out-of-order execution)
3. **Кэши процессора** могут изменять видимый порядок

### Пример
```java
class ReorderingExample {
    private int x = 0;
    private int y = 0;
    private boolean ready = false;
    
    public void writer() {
        x = 42;        // 1
        y = 100;       // 2
        ready = true;  // 3 - может выполниться ПЕРВЫМ!
    }
    
    public void reader() {
        if (ready) {   // 4
            System.out.println("x: " + x + ", y: " + y);
            // Может вывести: x: 0, y: 100  или  x: 42, y: 0
        }
    }
}
```

### Как это выглядит в байткоде
```java
// Исходный код:
public void writer() {
    x = 42;    // store
    y = 100;   // store  
    ready = true; // store
}

// После оптимизации компилятором может стать:
public void writer() {
    ready = true;  // переставлено!
    y = 100;
    x = 42;
}
```

### Happens-before гарантии
```java
// Без синхронизации нет гарантий порядка
class NoHappensBefore {
    int a = 0, b = 0;
    
    void thread1() {
        a = 1;
        b = 2;
    }
    
    void thread2() {
        int localB = b;  // может увидеть 2 ДО того как увидит a=1
        int localA = a;  // может быть 0, даже если localB=2
        System.out.println("a=" + localA + ", b=" + localB);
    }
}
```

---
Reordering (переупорядочивание) — это процесс, при котором инструкции программы выполняются в порядке, **отличном от порядка, написанного в исходном коде**, но так, чтобы однопоточное выполнение сохраняло свою семантику. Это делают компилятор (во время компиляции), CPU (во время выполнения), а также аппаратура памяти и кэшей (буферы, очереди). В однопоточной программе reordering невидим; в многопоточной — это основной источник data races и труднообнаружимых bagов. Понимание механизмов reordering, типов переупорядочивания и способов его предотвращения — критично для разработчика, создающего надежные многопоточные системы.

---

## I. Три уровня Reordering

## A. Уровень 1: Компилятор (Compiler Reordering)

**Что происходит**: Компилятор переставляет инструкции при преобразовании исходного кода в машинный код, чтобы оптимизировать выполнение.​

**Примеры переупорядочивания**:

``` java
// Исходный код
int x = 0, flag = 0;

void thread1() {
    x = 42;           // Инструкция 1
    flag = 1;         // Инструкция 2
}

void thread2() {
    while (flag == 0); // Ждём
    print(x);         // Ожидаем 42
}

```
Компилятор **может** переставить инструкции в thread1:
```java
// Компилятор переставил:
void thread1() {
    flag = 1;         // Инструкция 2 (выполнена первой)
    x = 42;           // Инструкция 1 (выполнена второй)
}

```

**Зачем компилятор это делает**:

- Минимизирует зависимости инструкций
    
- Лучше заполняет процессорный конвейер
    
- Может спеколировать (speculate), что переменная не меняется извне[](https://stackoverflow.com/questions/8742333/java-memory-model-compiler-rearranging-code-lines)​
    
- Выкидывает "лишние" обращения к памяти[](https://stackoverflow.com/questions/8742333/java-memory-model-compiler-rearranging-code-lines)​
    

**Гарантия компилятора**: Однопоточная семантика сохраняется. Если в однопоточном коде был порядок A → B → C, результат остается тем же, даже если внутри инструкции переставлены (благодаря data dependencies).[](https://stackoverflow.com/questions/8742333/java-memory-model-compiler-rearranging-code-lines)​

**Проблема**: В многопоточной программе thread2 может увидеть `flag == 1`, но так как инструкции были переставлены, значение `x` может быть еще 0!

## B. Уровень 2: CPU Out-of-Order Execution

**Что происходит**: Процессор во время выполнения может начать выполнять инструкции не в том порядке, в котором они выданы ему на выполнение, чтобы лучше использовать ресурсы и скрыть latency памяти.​​

**Как это работает**:

Современные процессоры используют **superscalar** архитектуру и **out-of-order execution engine**:​

1. **Fetch**: процессор берет инструкции из памяти в порядке программы
    
2. **Decode**: декодирует их в микро-операции (µops)
    
3. **Issue**: выдает на выполнение в **любом порядке**, где есть операнды (не по программному порядку)
    
4. **Execute**: выполняет параллельно на разных исполнительных блоках
    
5. **Retire**: **результаты выходят в исходном программном порядке** (in-order retirement)

``` 
Программный порядок:
1: Load r1, [memory_addr_1]    // может занять 100+ циклов (cache miss)
2: Load r2, [memory_addr_2]    // может быть в кэше
3: Add r3, r1, r2
4: Store r3, [memory_addr_3]

Порядок выполнения на CPU:
Fetch: 1, 2, 3, 4
Issue: 2 (быстрая), 3 (зависит от 1), 4 (зависит от 3), 1 (медленная)
Execute:
  - Инструкция 2 выполняется сразу (из кэша)
  - Инструкция 1 стоит в очереди (ждет памяти)
  - Инструкция 3 ждет r1 (от инструкции 1)
  
Retire: 1, 2, 3, 4 (в исходном порядке!)

```
**В многопоточном контексте**: Хотя результаты выходят в исходном порядке (retire в порядке), **видимость этих результатов другим потокам может быть в другом порядке**. Это потому что Load/Store операции становятся видимыми другим ядрам не в момент retire, а в момент, когда они попадают в кэш или основную память.

## C. Уровень 3: Store Buffer и Invalidate Queue

**Store Buffer**:​

Каждое ядро CPU имеет локальный буфер для записей (Store Buffer). Когда ядро выполняет инструкцию Store:

1. Значение помещается в локальный Store Buffer
    
2. Ядро сразу может читать это значение (Load from Store Buffer)
    
3. **Но другие ядра еще не видят это значение** — оно не попало в кэш/память
    
4. Store Buffer постепенно опустошается, отправляя значение в кэш/память
    

Это создает **StoreLoad reordering** даже без out-of-order execution:[](https://stackoverflow.com/questions/50307693/does-an-x86-cpu-reorder-instructions)​

``` java
Thread 1:              Thread 2:
Store x = 1           Load y    (может еще быть 0!)
Store y = 2           Load x    (может еще быть 0!)

// Store x в Buffer, еще не видна другому ядру
// Thread 2: Load y может читать старое значение из своего кэша

```

## Memory Barriers: Детальный Разбор

Барьеры памяти — это инструкции, которые предотвращают определенные типы переупорядочивания:​

## A. LoadLoad Barrier

**Предотвращает**: LoadLoad reordering
``` c
Load1();
LoadLoad_Barrier();
Load2();
// Load1 ВСЕГДА выполнится перед Load2
```
## B. LoadStore Barrier

**Предотвращает**: LoadStore reordering
``` c
Load1();
LoadStore_Barrier();
Store2();
// Load1 ВСЕГДА выполнится перед Store2
```
## C. StoreStore Barrier

**Предотвращает**: StoreStore reordering
```
Store1();
StoreStore_Barrier();
Store2();
// Store1 ВСЕГДА попадет в память перед Store2
```
