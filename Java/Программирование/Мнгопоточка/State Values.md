В данном случае каждое процессорное ядро имеет свой кэш и соответственно один поток может сохранить в своем кэше значение, а второй не увидит этого
##  State Visibility (Видимость состояния)

### Что это?
**Проблема видимости** — когда изменения, сделанные одним потоком, не видны другим потокам из-за кэширования.

### Причины
1. **Кэши процессоров** - каждый поток работает со своей копией
2. **Оптимизации компилятора** - чтение переменной может быть закешировано в регистре

### Пример
```java
class VisibilityProblem {
    private boolean running = true; // НЕТ volatile!
    
    public void start() {
        new Thread(() -> {
            while (running) {
                // Бесконечный цикл? Может быть!
            }
            System.out.println("Поток остановлен");
        }).start();
    }
    
    public void stop() {
        running = false; // Изменение может НЕ БЫТЬ ВИДНО другому потоку
    }
}

// Запуск:
VisibilityProblem vp = new VisibilityProblem();
vp.start();
Thread.sleep(1000);
vp.stop(); // Поток может продолжить работать бесконечно!
```

### Почему так происходит?
```java
// Без синхронизации компилятор может оптимизировать:
while (running) {
    // превращается в:
    if (running) {
        while (true) { // бесконечный цикл!
            // ...
        }
    }
}

// Или процессор читает running из своего кэша,
// а не из основной памяти
```

### Решение: volatile
```java
class VisibilitySolution {
    private volatile boolean running = true; // Теперь изменения видны сразу
    
    public void start() {
        new Thread(() -> {
            while (running) {
                // ...
            }
        }).start();
    }
    
    public void stop() {
        running = false; // Гарантированно будет видно
    }
}
```



Нет, так понимать **нельзя**. То, что потоки одного процесса разделяют общее _адресное пространство_, вообще не гарантирует, что второй поток сразу увидит изменения, сделанные первым.

## Почему так

- Каждый поток работает не напрямую с «общей памятью», а через **кэш и буферы записи** своего ядра процессора.studfile+1​
    
- Когда поток A изменяет переменную, новое значение может какое‑то время находиться только в его кэше или буфере и ещё не быть «протолкнутым» в общую оперативную память.examclouds+1​
    
- Поток B, даже в том же процессе, может читать **старое значение** из своего кэша, пока не сработают протокол когерентности кэшей и/или барьеры памяти, которые заставят ядра согласовать данные.stackoverflow+1​
    

То есть ситуация:

> Поток 1 записал `x = 42` → в своём кэше  
> Поток 2 читает `x` и видит старое значение

**вполне возможна**, несмотря на общее адресное пространство. Общий адрес — это логика «что именно мы читаем», а не «когда мы увидим новые данные».

## Что реально даёт гарантию

Гарантии видимости между потоками дают только **механизмы синхронизации**:struchkov+1​

- мьютексы / критические секции / `synchronized`;
    
- `volatile` (в Java) или аналогичные барьеры чтения/записи;
    
- атомарные операции (CAS и т. п.);
    
- другие примитивы, которые под капотом ставят **memory fences** и заставляют кэши/буферы согласовать данные.
    

Без них кэш + переупорядочивание инструкций легко приводят к тому, что один поток уже «изменил» переменную, а другой ещё какое‑то время видит старое состояние, даже в рамках одного процесса.struchkov+1​

Если хочешь, дальше можно разобрать конкретно на примере Java и happens-before (как именно volatile/lock заставляют кэши «увидеть» изменения).
