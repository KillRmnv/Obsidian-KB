Нужно для верификации подключения пользователя к серверу(нет необходимости запрашивать логин и пароль каждый раз при запросе)
### 1. Главный принцип REST: Statelessness (Отсутствие состояния)

Одно из ключевых ограничений архитектуры REST гласит:

> **Сервер не должен хранить состояние клиента между запросами.**

Каждый запрос от клиента к серверу должен содержать **всю необходимую информацию** для его обработки (аутентификация, параметры, данные). Сервер не должен помнить, что этот клиент делал в предыдущем запросе.

#### Почему `HttpSession` нарушает REST?

В классическом Java EE (Servlets) вы используете `HttpSession`:

1. Сервер создает объект сессии в памяти.
2. Клиенту отдается `JSESSIONID` (в cookie).
3. При следующих запросах сервер по ID находит объект в памяти.

**Проблемы для REST:**

- **Масштабируемость:** Если у вас несколько серверов (кластер), запросы клиента должны всегда попадать на тот же сервер (sticky sessions), либо нужно синхронизировать сессии между серверами (Redis, DB), что усложняет архитектуру.
- **Нагрузка на сервер:** Хранение состояния тысяч пользователей в памяти сервера потребляет ресурсы.
- **Кэширование:** Ответы зависимы от состояния сессии, их сложнее кэшировать на прокси-серверах.

### 2. Чем заменяют сессии в REST?

Вместо серверных сессий в REST используют **токены** (Token-based Authentication).

#### Самый популярный вариант: JWT (JSON Web Token)

1. **Логин:** Клиент отправляет логин/пароль.
2. **Выдача:** Сервер проверяет данные и подписывает токен (JWT), содержащий информацию о пользователе (ID, роли, срок действия). **Сервер не сохраняет токен у себя.**
3. **Использование:** Клиент отправляет этот токен в заголовке каждого последующего запроса.
4. **Проверка:** Сервер проверяет подпись токена (математически) и извлекает данные пользователя. **Запрос в базу данных сессий не нужен.**

Это делает сервер **stateless** (без состояния).

### 3. Где хранить токен на клиенте?

Здесь снова всплывает тема **Cookies** из вашего предыдущего вопроса.

|Способ|Заголовок `Authorization`|Cookie (HttpOnly)|LocalStorage|
|---|---|---|---|
|**Как выглядит**|`Authorization: Bearer <token>`|Автоматически прикрепается браузером|`localStorage.getItem('token')`|
|**Безопасность (XSS)**|✅ Безопасно (JS не видит)|✅ Безопасно (JS не видит)|❌ Уязвимо (JS имеет доступ)|
|**Безопасность (CSRF)**|✅ Безопасно (нужен ручной код)|❌ Уязвимо (нужен CSRF-токен)|✅ Безопасно|
|**REST-чистота**|✅ Высокая|⚠️ Средняя (зависимость от браузера)|⚠️ Средняя|
|**Рекомендация**|**Для API / Mobile**|**Для Веб-приложений (SPA)**|**Не рекомендуется**|

- **Для чистых API (Mobile, Backend-to-Backend):** Лучше передавать токен в заголовке `Authorization`.
- **Для Веб-приложений (React/Angular + Backend):** Часто используют Cookie с флагом `HttpOnly` для защиты от XSS, но тогда нужно защищаться от CSRF.
