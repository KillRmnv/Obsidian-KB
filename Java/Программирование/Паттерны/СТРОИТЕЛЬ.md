**Builder**
***
**Строитель** — это порождающий паттерн проектирования, который позволяет создавать объекты пошагово.

Паттерн **Строитель (Builder)** решает проблему создания сложных объектов, у которых много параметров (особенно опциональных). Его главные плюсы можно разделить на удобство использования, архитектурную чистоту и безопасность.

Вот основные преимущества:

### 1. Избавление от «телескопического конструктора»
Это самая частая причина использования Строителя. Если у объекта много полей, некоторые из которых опциональны, вам пришлось бы создавать множество перегруженных конструкторов:
```java
// Плохо: трудно понять, какой параметр за что отвечает
new User("Ivan", null, null, 25, null, "email@test.com", null, true);
```
Со Строителем код становится самодокументируемым:
```java
// Хорошо: видно, что мы устанавливаем
User user = new User.Builder()
    .setName("Ivan")
    .setAge(25)
    .setEmail("email@test.com")
    .setVerified(true)
    .build();
```

### 2. Создание неизменяемых (Immutable) объектов
Строитель идеально подходит для создания объектов, состояние которых нельзя изменить после создания (что важно для многопоточности и безопасности).
*   Все поля в самом объекте `User` могут быть `private final`.
*   Значения устанавливаются только внутри Строителя.
*   Объект создается один раз в методе `build()`.
*   После создания изменить поля невозможно, так как сеттеров в самом классе нет.

### 3. Валидация данных перед созданием
Вы можете проверить корректность данных **до** того, как объект будет создан. Если данные неверны, метод `build()` выбросит исключение, и невалидный объект никогда не появится в системе.
```java
public User build() {
    if (age < 0) throw new IllegalArgumentException("Age must be positive");
    if (name == null) throw new IllegalArgumentException("Name is required");
    return new User(this);
}
```

### 4. Пошаговое построение объекта
Строитель позволяет конструировать объект поэтапно. Вы можете вызывать методы настройки в любом порядке, пропускать ненужные шаги (если поля опциональные) или даже выполнять сложную логику между шагами.

### 5. Разделение логики создания и представления (Классический GoF)
В полной реализации паттерна (с Director и разными ConcreteBuilder) один и тот же процесс сборки может создавать разные представления объекта.
*   *Пример:* Один алгоритм сборки автомобиля может создать спортивную версию (с мощным двигателем) и семейную версию (с большим багажником), используя разных строителей, но одного директора.

### 6. Изоляция сложной логики создания
Если создание объекта требует сложных вычислений или взаимодействия с другими сервисами, эта логика инкапсулируется внутри Строителя. Клиентский код остается чистым и не знает о том, как именно объект собирается внутри.

### 7. Fluent Interface (Плавный интерфейс)
Строители часто реализуются с возвратом `this` в методах настройки, что позволяет писать код цепочкой (method chaining). Это значительно повышает читаемость кода.

---

### Сравнение: Конструктор vs Строитель

| Ситуация | Конструктор / Сеттеры | Паттерн Строитель |
| :--- | :--- | :--- |
| **Обязательных параметров мало** | ✅ Отлично подходит | ❌ Избыточен (много кода) |
| **Много опциональных параметров** | ❌ Хаос (телескопические конструкторы) | ✅ Идеально |
| **Нужна неизменяемость объекта** | ⚠️ Сложно реализовать с сеттерами | ✅ Легко (поля final) |
| **Нужна валидация при создании** | ⚠️ Возможна в конструкторе | ✅ Удобно в методе `build()` |
| **Читаемость кода клиента** | ⚠️ Зависит от порядка аргументов | ✅ Высокая (имена методов) |

### Когда Строитель НЕ нужен?
Не стоит использовать Строитель для простых объектов с 2-3 полями. Это приведет к излишней verbosity (многословности) кода. Вы будете писать больше кода для создания объекта, чем для самой бизнес-логики.
