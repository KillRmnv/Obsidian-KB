
# 1. Базовое логирование (Logback / Log4j2)

## Что это

- `Logger.info/debug/error`
    
- лог в **консоль** и/или **файл**
    
- уровни логирования
    

## Где используют

📍 **ВСЕ Spring-приложения**

## Когда

- разработка
    
- отладка
    
- локальный сервер
    
- small / medium backend
    

## Пример

```java
private static final Logger log = LoggerFactory.getLogger(MyService.class);

log.info("User {} uploaded file {}", userId, fileName);
```

## Что логируют

- старт приложения
    
- бизнес-события
    
- ошибки
    
- входные данные (частично)
    

---

# 2. Файловые логи (RollingFileAppender)

## Что

- лог пишется в файлы
    
- ротация по размеру / дате
    

## Где

📍 **серверы без централизованного логирования**

## Когда

- VPS
    
- монолит
    
- один сервер
    
- учебные и pet-проекты
    

## Почему

- просто
    
- дешево
    
- быстро поднять
    

## Минусы

- неудобно искать
    
- нет агрегации
    

---

# 3. Централизованное логирование (ELK / Graylog / Loki)

## Что

- логи отправляются **в отдельную систему**
    
- хранятся централизованно
    
- поиск по всем сервисам
    

## Где

📍 **микросервисы**  
📍 **production**

## Когда

- несколько сервисов
    
- Kubernetes
    
- Docker
    
- high-load
    

## Типовые стеки

|Стек|Когда|
|---|---|
|**ELK**|крупные системы|
|**Graylog**|проще ELK|
|**Grafana Loki**|cloud-native|

## Что логируют

- все ошибки
    
- бизнес-события
    
- correlationId
    
- userId
    
- requestId
    

---

# 4. Structured logging (JSON)

## Что

- лог не строка, а JSON
    
- легко парсить
    

```json
{
  "level": "INFO",
  "service": "file-storage",
  "userId": 42,
  "action": "UPLOAD",
  "file": "doc.pdf"
}
```

## Где

📍 production  
📍 централизованные системы

## Когда

- поиск по полям
    
- аналитика
    
- фильтрация
    

## Почему

👉 **обычные строки плохо масштабируются**

---

# 5. MDC (Mapped Diagnostic Context)

## Что

- контекст запроса
    
- автоматически добавляется в каждый лог
    

```java
MDC.put("requestId", uuid);
MDC.put("userId", userId);
```

## Где

📍 web-приложения  
📍 REST API

## Когда

- нужно отследить запрос целиком
    
- несколько сервисов
    

## Что решает

❌ «кто вызвал этот лог?»  
✅ видно **всю цепочку**

---

# 6. HTTP-логирование (Logbook и аналоги)

## Что

- логирует request/response
    
- headers
    
- body
    

## Где

📍 API  
📍 gateway  
📍 debug-режим

## Когда

- отладка интеграций
    
- аудит
    
- API-gateway
    

## ❗ Важно

- **не включать body в prod без фильтрации**
    
- скрывать пароли / токены
    

---

# 7. Error logging / monitoring (Sentry)

## Что

- автоматический сбор exceptions
    
- stacktrace
    
- контекст
    

## Где

📍 production

## Когда

- важна стабильность
    
- нужно знать о падениях сразу
    

## Почему

- логи читают ПОСЛЕ
    
- Sentry уведомляет СРАЗУ
    

---

# 8. Аудит-логирование (Business logs)

## Что

- кто что сделал
    
- когда
    
- над какими данными
    

## Где

📍 финансы  
📍 документы  
📍 админки

## Когда

- требования безопасности
    
- юридический след
    

## Где хранят

- отдельная БД
    
- неизменяемые логи
    
- иногда S3
    

---

# 9. Observability (логи + метрики + трейсы)

## Что

- **логи** → что произошло
    
- **метрики** → как часто
    
- **трейсы** → где медленно
    

## Где

📍 microservices  
📍 cloud

## Когда

- high-load
    
- SLA
    
- performance
    

## Инструменты

- OpenTelemetry
    
- Prometheus
    
- Grafana
    
- Jaeger

# 12. Если коротко (TL;DR)

> **Логирование — это не "println", это инфраструктура.**

- всегда есть базовый логгер
    
- в prod — централизованный
    
- ошибки — в мониторинг
    
- контекст — через MDC
    
- бизнес-события — отдельно
    

---

Если хочешь, следующим шагом могу:

- показать **идеальный logback-spring.xml**,
    
- разобрать **типичную ошибку "слишком много логов"**,
    
- или **подстроить стратегию логирования под твой файловый/S3-сервис**.
