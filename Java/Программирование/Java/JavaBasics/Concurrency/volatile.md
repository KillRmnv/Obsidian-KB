Ключевое слово `volatile` в Java — это «облегченный» механизм синхронизации. Оно используется для того, чтобы гарантировать, что изменения переменной, сделанные одним потоком, будут немедленно видны всем остальным потокам.

Вот главные особенности, которые нужно понимать, чтобы использовать его правильно:

---

### 1. Гарантия видимости (Visibility)

В обычных условиях потоки могут кэшировать переменные в регистрах процессора или локальном кэше ядра для скорости. Если Поток А изменит переменную, Поток Б может еще долго видеть её старое значение из своего кэша.

Когда поле помечено как `volatile`, JVM гарантирует:

- **Запись:** Каждое изменение переменной немедленно записывается в основную память (Main Memory).
    
- **Чтение:** Каждое чтение переменной происходит напрямую из основной памяти, игнорируя локальные кэши.
    

---

### 2. Запрет переупорядочивания (Instruction Reordering)

Для оптимизации компилятор и процессор могут менять порядок выполнения команд. `volatile` устанавливает **Memory Barrier** (барьер памяти). Инструкции, которые стоят _до_ записи в `volatile`-переменную, не могут быть перемещены в коде _после_ неё. Это критически важно для корректной передачи состояния между потоками.

---

### 3. Отсутствие атомарности (Самая частая ошибка)

Это то, на чем спотыкаются многие. **`volatile` не делает операции атомарными.**

Если вы делаете `volatileCount++`, это не спасет от состояния гонки. Как мы уже обсуждали в теме про Atomic types, инкремент состоит из трех стадий (чтение-изменение-запись). `volatile` гарантирует, что вы прочитаете актуальное значение, но пока вы будете его увеличивать, другой поток может успеть перезаписать это значение в памяти.

> **Правило:** Используйте `volatile`, только если новое значение переменной **не зависит** от её предыдущего значения (например, флаг `isStopped = true`). Если зависит (счетчик `count++`) — используйте `AtomicInteger` или `synchronized`.

---

### 4. Отношение Happens-before

Согласно спецификации Java Memory Model (JMM), запись в `volatile`-поле _happens-before_ любого последующего чтения этого же поля. Это создает мост видимости: если вы записали что-то в обычные поля, а затем обновили `volatile`-флаг, то другой поток, прочитав этот флаг, гарантированно увидит и все ваши предыдущие записи в обычные поля.

---

### 5. Когда использовать volatile?

Лучшие сценарии для `volatile`:

- **Флаги состояния (Status flags):** Остановка потока, завершение инициализации.
    
- **Double-Checked Locking:** В паттерне Singleton для предотвращения публикации частично сконструированного объекта.
    
- **Публикация неизменяемых объектов:** Когда вы один раз записываете ссылку на тяжелый объект, который потом только читается другими.
