### 1. Файловая структура (Папки)

После распаковки архива Tomcat вы увидите следующую иерархию:

```text
apache-tomcat-<version>/
├── bin/           # Скрипты запуска и остановки
├── conf/          # Файлы конфигурации
├── lib/           # Библиотеки (JAR), общие для всех приложений
├── logs/          # Логи работы сервера
├── webapps/       # Папка для размещения приложений (WAR)
├── work/          # Временные файлы (компиляция JSP)
├── temp/          # Временные файлы JVM
├── RUNNING.txt    # Информация о запуске
└── ...
```

#### Подробное описание папок:

| Папка | Назначение | Что важно знать |
| :--- | :--- | :--- |
| **`bin/`** | Исполняемые скрипты | Здесь лежат `startup.sh` (запуск), `shutdown.sh` (остановка), `catalina.sh` (управление). На Windows — `.bat` файлы. |
| **`conf/`** | Конфигурация | **Самая важная папка.** Здесь настраиваются порты, пользователи, пути к логам. См. ниже раздел "Конфиги". |
| **`lib/`** | Глобальные библиотеки | JAR-файлы, которые видны **всем** приложениям на этом сервере. Сюда кладут драйверы БД, если нужно共享ить их между приложениями (но лучше класть в приложение). |
| **`webapps/`** | Деплой приложений | Сюда копируют `.war` файлы. Tomcat автоматически распаковывает их в папки с тем же именем. <br> **`webapps/ROOT`** — это приложение по умолчанию (открывается при заходе на `http://localhost:8080/`). |
| **`work/`** | Рабочая директория | Здесь Tomcat хранит скомпилированные классы для **JSP**. Если вы изменили JSP, а изменения не видны — иногда помогает очистка этой папки. |
| **`logs/`** | Логи | Здесь хранятся файлы логов. Главный файл — `catalina.out` (stdout/stderr сервера). |
| **`temp/`** | Временные файлы | Используется JVM и самим Tomcat для временных данных. Можно чистить при остановленном сервере. |

### 2. Ключевые файлы конфигурации (`conf/`)

В этой папке находятся файлы, определяющие поведение сервера.

1.  **`server.xml`** — **Главный конфиг.**
    *   Определяет порты (Connector port="8080").
    *   Настраивает виртуальные хосты (VirtualHost).
    *   Задает пути к логам.
    *   *Осторожно:* Ошибка в синтаксике не даст серверу запуститься.

2.  **`web.xml`** — **Глобальные настройки веб-приложений.**
    *   Содержит настройки по умолчанию для всех WAR-файлов (таймаут сессии,_mime типы, стандартные сервлеты).
    *   Не путать с `WEB-INF/web.xml` внутри вашего приложения (который переопределяет эти настройки).

3.  **`tomcat-users.xml`** — **Пользователи и роли.**
    *   Здесь создаются пользователи для доступа к **Manager App** и **Host Manager**.
    *   Пример добавления админа:
        ```xml
        <role rolename="manager-gui"/>
        <user username="admin" password="secret" roles="manager-gui"/>
        ```

4.  **`context.xml`** — **Контекст по умолчанию.**
    *   Настройки, применяемые ко всем приложениям (например, настройки пула соединений JDBC, если они вынесены глобально).

5.  **`Catalina/localhost/`** (папка внутри conf)
    *   Сюда можно класть отдельные XML-файлы для конфигурации конкретных приложений без редактирования `server.xml`.
    *   Пример: файл `my-app.xml` опишет контекст для приложения `my-app`.

### 3. Внутренняя архитектура (Компоненты)

Если копнуть глубже папок, Tomcat состоит из нескольких ключевых компонентов (контейнеров):

1.  **Catalina (Каталина)**
    *   Это сердце Tomcat. Контейнер сервлетов.
    *   Обрабатывает жизненный цикл Servlets, управляет сессиями, контекстами.
    *   Именно она загружает ваше Spring-приложение.

2.  **Coyote (Койот)**
    *   Коннектор. Отвечает за прием HTTP-запросов.
    *   Слушает порт (например, 8080), парсит HTTP-запрос и передает его в Catalina для обработки.
    *   Поддерживает протоколы HTTP/1.1, HTTP/2, AJP (для связи с Apache HTTP Server).

3.  **Jasper (Джаспер)**
    *   Движок JSP.
    *   Превращает файлы `.jsp` в Java-классы (Servlets), компилирует их и выполняет.
    *   Работает прозрачно для разработчика.

4.  **Cluster (Кластер)**
    *   Модуль для балансировки нагрузки и репликации сессий между несколькими серверами Tomcat (используется редко, чаще вместо этого ставят балансировщик перед несколькими независимыми Tomcat).

### 4. Структура веб-приложения (Inside WAR)

Когда вы кладете WAR в `webapps`, он распаковывается в стандартную структуру Java Web App. Важно знать, где что лежит внутри вашего приложения:

```text
my-app/ (распакованный WAR)
├── META-INF/          # Метаданные (context.xml приложения)
├── WEB-INF/           # Закрытая папка (клиент не имеет доступа через URL)
│   ├── classes/       # Скомпилированные классы вашего кода (.class)
│   ├── lib/           # Библиотеки приложения (JAR)
│   └── web.xml        # Конфигурация конкретного приложения
├── index.jsp          # Публичные файлы (доступны через URL)
├── css/               # Статика
└── images/            # Статика
```

*   **Важно:** Файлы внутри `WEB-INF` недоступны напрямую через браузер (защита от скачивания классов или конфигов).
*   **Важно:** Библиотеки в `WEB-INF/lib` изолированы. Приложение А не видит библиотеки приложения Б, даже если они лежат на одном Tomcat.

### 5. Практические советы по структуре

1.  **Очистка кэша:**
    Если странное поведение (старый код выполняется после деплоя):
    *   Остановите Tomcat.
    *   Удалите содержимое папки `work/Catalina/localhost/`.
    *   Удалите распакованную папку приложения в `webapps/` (оставьте только WAR).
    *   Запустите Tomcat.

2.  **Не храните данные в `webapps`:**
    Не пишите логи или загруженные файлы пользователей внутрь папки `webapps`. При обновлении приложения (перезаписи WAR) всё содержимое папки приложения удалится.
    *   *Правильно:* Вынесите путь во внешнюю директорию (например, `/var/data/uploads`) и настройте в конфиге.

3.  **Изоляция библиотек:**
    Старайтесь не класть общие JAR в глобальную папку `lib/`. Лучше упаковывайте все зависимости внутрь `WEB-INF/lib` вашего приложения (это делает Maven/Gradle автоматически). Это предотвратит конфликты версий между разными приложениями.

4.  **Конфигурация вне кода:**
    Используйте `conf/Catalina/localhost/<имя-приложения>.xml` для вынесения настроек (например, JDBC Resource) из WAR-файла. Это позволяет менять конфиг БД без пересборки приложения.
