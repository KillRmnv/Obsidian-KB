В Hibernate (и JPA) есть **два основных способа** маппинга собственного класса на поля таблицы, в зависимости от того, как вы хотите хранить данные:

1.  **`@Embeddable` + `@Embedded`** — поля вашего класса раскладываются в **несколько колонок** той же таблицы.
2.  **`@Convert` (AttributeConverter)** — весь объект сохраняется в **одну колонку** (например, как JSON, строка или зашифрованное значение).

Разберем оба варианта.

---

## Вариант 1. `@Embeddable` (Поля класса → Несколько колонок)

Используется для **Value Objects** (объектов-значений), у которых нет своего идентификатора. Например, класс `Address` (адрес) внутри сущности `User`.

### 1. Создаем встраиваемый класс
Класс должен быть аннотирован `@Embeddable` и реализовывать `Serializable`.

```java
import jakarta.persistence.Embeddable;
import java.io.Serializable;

@Embeddable
public class Address implements Serializable {

    private String city;
    private String street;
    private String zipCode;

    // Конструкторы, геттеры, сеттеры
    public Address() {}
    
    // ...
}
```

### 2. Используем в сущности
В основной сущности используем аннотацию `@Embedded`.

```java
import jakarta.persistence.*;

@Entity
@Table(name = "users")
public class User {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String name;

    // Поля класса Address станут колонками: address_city, address_street, address_zipCode
    @Embedded
    private Address address;

    // ... геттеры и сеттеры
}
```

### 3. Переопределение имен колонок (опционально)
Если вам не нравятся стандартные имена (например, хотите просто `city` вместо `address_city`), используйте `@AttributeOverrides`.

```java
@Embedded
@AttributeOverrides({
    @AttributeOverride(name = "city", column = @Column(name = "user_city")),
    @AttributeOverride(name = "zipCode", column = @Column(name = "postal_code"))
})
private Address address;
```

**Результат в БД:**
Таблица `users` будет иметь колонки: `id`, `name`, `address_city`, `address_street`, `address_zip_code`.

## Вариант 2. `@Convert` (Объект → Одна колонка)

Используется, если вы хотите сохранить весь объект в **одном поле** (например, в колонку типа `VARCHAR`, `TEXT` или `JSON`). Для этого нужен конвертер.

### 1. Создаем конвертер
Реализуем интерфейс `AttributeConverter<X, Y>`, где `X` — ваш класс, `Y` — тип данных в БД.

```java
import jakarta.persistence.AttributeConverter;
import jakarta.persistence.Converter;
import com.fasterxml.jackson.core.JsonProcessingException;
import com.fasterxml.jackson.databind.ObjectMapper;

@Converter(autoApply = false) // false = применяем только там, где укажем @Convert
public class MoneyConverter implements AttributeConverter<Money, String> {

    private static final ObjectMapper mapper = new ObjectMapper();

    // Java -> DB
    @Override
    public String convertToDatabaseColumn(Money money) {
        if (money == null) return null;
        try {
            // Пример: сохраняем как JSON строку {"amount": 100, "currency": "USD"}
            return mapper.writeValueAsString(money);
        } catch (JsonProcessingException e) {
            throw new IllegalArgumentException("Ошибка конвертации Money", e);
        }
    }

    // DB -> Java
    @Override
    public Money convertToEntityAttribute(String dbData) {
        if (dbData == null) return null;
        try {
            return mapper.readValue(dbData, Money.class);
        } catch (JsonProcessingException e) {
            throw new IllegalArgumentException("Ошибка чтения Money", e);
        }
    }
}
```

### 2. Используем в сущности
Аннотируем поле `@Convert`.

```java
@Entity
@Table(name = "orders")
public class Order {

    @Id
    private Long id;

    // Будет сохранено в одну колонку 'price' типа VARCHAR/TEXT
    @Convert(converter = MoneyConverter.class)
    @Column(columnDefinition = "TEXT")
    private Money price;

    // ...
}
```

**Результат в БД:**
Таблица `orders` будет иметь колонку `price`, содержащую строку: `{"amount":100,"currency":"USD"}`.

## Вариант 3. Нативный JSON (Hibernate 6+ / PostgreSQL)

Если вы используете **Hibernate 6** и базу данных, поддерживающую JSON (PostgreSQL, MySQL 5.7+), можно мапить класс прямо в JSON-колонку без написания конвертера.

### 1. Класс (обычный POJO)
```java
public class Money {
    private BigDecimal amount;
    private String currency;
    // геттеры, сеттеры
}
```

### 2. Сущность с `@JdbcTypeCode`
```java
import org.hibernate.annotations.JdbcTypeCode;
import org.hibernate.type.SqlTypes;

@Entity
public class Order {

    @Id
    private Long id;

    // Hibernate сам превратит объект в JSON для БД
    @JdbcTypeCode(SqlTypes.JSON)
    @Column(columnDefinition = "jsonb") // для PostgreSQL
    private Money price;
}
```

*Примечание: Для этого способа нужно подключить зависимость `hibernate-jpamodelgen` или убедиться, что версия Hibernate не ниже 6.0.*

## Сравнение подходов

| Подход | Колонки в БД | Когда использовать | Плюсы | Минусы |
|--------|--------------|--------------------|-------|--------|
| **`@Embeddable`** | Несколько (по полю класса) | Адрес, Имя, Координаты | Можно делать запросы по полям (`WHERE address_city = 'Msk'`) | Схема БД зависит от структуры класса |
| **`@Convert`** | Одна (сериализация) | Сложные объекты, настройки, история | Гибкость, схема БД не меняется при изменении класса | Нельзя фильтровать в SQL по внутренним полям |
| **`@JdbcTypeCode` (JSON)** | Одна (JSON тип) | Документы, гибкие атрибуты | Нативная поддержка JSON в БД, удобно для поиска внутри JSON | Привязка к конкретной БД (Postgres/MySQL) |

## Что выбрать?

1.  Если вам нужно **искать/фильтровать** данные по полям этого класса в SQL → используйте **`@Embeddable`**.
2.  Если это просто "груз" данных, который вы читаете целиком и не фильтруете по нему → используйте **`@Convert`** или **JSON**.
3.  Если у вас **PostgreSQL** и современный Hibernate → **JSON (`@JdbcTypeCode`)** часто является лучшим выбором для гибкости.
