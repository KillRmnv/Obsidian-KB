Кэширование в Hibernate — это механизм оптимизации производительности, который позволяет избегать лишних обращений к базе данных, храня часто используемые данные в памяти.

В Hibernate существует три основных уровня кэширования:

1.  **Кэш первого уровня (L1 Cache)**
2.  **Кэш второго уровня (L2 Cache)**
3.  **Кэш запросов (Query Cache)**

Разберем каждый из них подробно.

---

### 1. Кэш первого уровня (L1 Cache)

Это кэш по умолчанию. Он привязан к объекту **`Session`** (или `EntityManager` в JPA).

*   **Область видимости:** В рамках одной сессии.
*   **Включен:** Всегда (отключить нельзя).
*   **Механизм:** Паттерн "Identity Map". Когда вы загружаете сущность по ID, Hibernate проверяет, есть ли она уже в текущей сессии. Если есть — возвращает из памяти, если нет — делает запрос в БД и кладет в кэш.
*   **Жизненный цикл:** Живет пока открыта сессия. При закрытии сессии (`session.close()`) кэш очищается.

**Пример:**
```java
Session session = sessionFactory.openSession();
Transaction tx = session.beginTransaction();

// Первый запрос: идет в БД
User user1 = session.get(User.class, 1L); 

// Второй запрос: берет из L1 кэша (БД не опрашивается)
User user2 = session.get(User.class, 1L); 

tx.commit();
session.close(); // Кэш очищен
```

### 2. Кэш второго уровня (L2 Cache)

Это опциональный кэш, который привязан к **`SessionFactory`**. Он общий для всех сессий в приложении.

*   **Область видимости:** Глобальная (всё приложение).
*   **Включен:** Требуется явная настройка.
*   **Что хранит:** Данные сущностей (поля), данные коллекций (связи `@OneToMany` и т.д.).
*   **Провайдеры:** Сам Hibernate не хранит данные в L2, он делегирует это внешним библиотекам. Популярные провайдеры: **Ehcache**, **Infinispan**, **Redis**, **Hazelcast**.

#### Настройка L2 (пример для `hibernate.cfg.xml` или `application.properties`):

```properties
# Включить кэш второго уровня
hibernate.cache.use_second_level_cache=true

# Выбрать провайдер (например, Ehcache)
hibernate.cache.region.factory_class=org.hibernate.cache.ehcache.EhCacheRegionFactory

# Включить кэш запросов (опционально)
hibernate.cache.use_query_cache=true
```

#### Настройка сущности для кэширования:
Нужно указать, какие сущности кэшировать и какую стратегию использовать.

```java
@Entity
@Cacheable // JPA аннотация
@org.hibernate.annotations.Cache(
    usage = CacheConcurrencyStrategy.READ_WRITE, // Стратегия
    region = "UserCache" // Имя региона (опционально)
)
public class User {
    // ...
}
```

#### Стратегии конкурентного доступа (Concurrency Strategies):
Это критически важный параметр, определяющий, как Hibernate обрабатывает одновременное изменение данных.

1.  **`READ_ONLY`**: Данные никогда не меняются. Самый быстрый вариант. Подходит для справочников (города, валюты).
2.  **`READ_WRITE`**: Использует блокировки (soft locks) для обеспечения согласованности. Подходит для данных, которые меняются, но не слишком часто.
3.  **`NONSTRICT_READ_WRITE`**: Данные кэшируются, но при обновлении кэш не блокируется сразу. Есть риск прочитать устаревшие данные в очень короткий промежуток времени. Быстрее, чем `READ_WRITE`.
4.  **`TRANSACTIONAL`**: Полная поддержка транзакций (требует JTA). Самый строгий и тяжелый вариант.

### 3. Кэш запросов (Query Cache)

Кэширует не сами объекты, а **результаты запросов** (список ID сущностей и значений примитивных типов).

*   **Зависимость:** Работает только вместе с L2 кэшем. Если в кэше запроса есть список ID `[1, 5, 10]`, то сами объекты с этими ID берутся из L2 кэша.
*   **Включение:** `hibernate.cache.use_query_cache=true`.
*   **Использование в коде:**
    ```java
    List<User> users = session.createQuery("from User where active = true")
        .setCacheable(true) // Включить кэширование этого запроса
        .list();
    ```
*   **Проблема:** Кэш запросов очень чувствителен к изменениям. Если в таблицу `User` добавится новая запись или изменится поле `active` у любой записи, кэш этого запроса будет аннулирован (инвалидирован). Поэтому его часто отключают, если запросы часто меняются или данные часто обновляются.

---

### Управление кэшем (Очистка и Инвалидация)

Иногда нужно принудительно очистить кэш, чтобы получить актуальные данные из БД.

1.  **Очистка L1:**
    ```java
    session.clear(); // Очистить всю сессию
    session.evict(user); // Удалить конкретный объект из сессии
    ```
2.  **Очистка L2:**
    ```java
    // Через SessionFactory
    sessionFactory.getCache().evict(User.class); // Очистить все сущности User
    sessionFactory.getCache().evict(User.class, 1L); // Очистить пользователя с ID=1
    
    // Очистка кэша запросов
    sessionFactory.getCache().evictQueryRegions();
    ```

### Плюсы и Минусы

| Плюсы | Минусы |
| :--- | :--- |
| **Скорость:** Значительное уменьшение нагрузки на БД и времени отклика. | **Согласованность:** Риск получить устаревшие данные (Stale Data), особенно при использовании кластеризации. |
| **Масштабируемость:** Снижение количества соединений с БД. | **Память:** Кэш потребляет оперативную память приложения. |
| **Прозрачность:** Для разработчика работа с кэшем часто незаметна. | **Сложность отладки:** Трудно понять, почему данные не обновляются или почему кэш не работает. |

---

### Рекомендации и Best Practices

1.  **Не кэшируйте всё подряд.** Включайте L2 только для сущностей, которые часто читаются и редко меняются.
2.  **Избегайте кэширования часто меняющихся данных.** Если данные обновляются каждую секунду, кэш будет постоянно инвалидироваться, что только замедлит работу (оверхед на управление кэшем).
3.  **Осторожно с Query Cache.** Включайте его только для очень тяжелых и стабильных запросов (отчеты, списки справочников). Для простых `findById` он не нужен (работает L1/L2).
4.  **Следите за размером кэша.** Настройте политики вытеснения (LRU, LFU) в провайдере кэша (например, в Ehcache), чтобы память не переполнилась.
5.  **Кластеризация.** Если приложение работает на нескольких серверах, используйте распределенный кэш (например, **Redis** или **Hazelcast**), иначе на каждом сервере будут свои данные, и они рассинхронизируются.
6.  **Spring Boot.** Если вы используете Spring Boot, подключение кэширования упрощается через зависимости `spring-boot-starter-cache` и настройку `application.properties`.

### Пример настройки в Spring Boot (`application.properties`)

```properties
# Hibernate
spring.jpa.properties.hibernate.cache.use_second_level_cache=true
spring.jpa.properties.hibernate.cache.region.factory_class=org.hibernate.cache.jcache.JCacheRegionFactory
spring.jpa.properties.hibernate.cache.use_query_cache=true

# Ehcache (путь к конфигу)
spring.jpa.properties.hibernate.javax.cache.uri=classpath:ehcache.xml
```

**Итог:** Кэш в Hibernate — мощный инструмент. L1 работает "из коробки" и безопасен. L2 и Query Cache требуют аккуратной настройки и понимания того, как данные обновляются в вашей системе, чтобы избежать проблем с целостностью данных.
