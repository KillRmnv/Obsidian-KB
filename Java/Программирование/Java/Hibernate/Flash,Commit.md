Коротко: **`flush()` отправляет SQL в базу, а `commit()` завершает транзакцию и делает изменения постоянными.**

Главное отличие: **после `flush()` можно сделать откат (`rollback`), после `commit()` — нельзя.**

### Сравнительная таблица

| Характеристика | `session.flush()` | `transaction.commit()` |
| :--- | :--- | :--- |
| **Суть** | Синхронизация памяти с БД | Завершение транзакции |
| **SQL запросы** | Отправляет `INSERT/UPDATE/DELETE` в БД | Сначала делает `flush()`, затем `COMMIT` в БД |
| **Транзакция** | Остается **открытой** | **Закрывается** |
| **Откат (Rollback)** | **Возможен** (изменения отменятся) | **Невозможен** (изменения сохранены навсегда) |
| **Видимость для других** | Не видны (транзакция не завершена) | Видны (если уровень изоляции позволяет) |
| **Когда вызывать** | Редко вручную (обычно автоматически) | Всегда в конце логической единицы работы |

---

### 1. `flush()` (Сброс)
Это команда: **«Hibernate, отправь все накопленные изменения в базу данных прямо сейчас, но не закрывай транзакцию»**.

*   **Зачем нужно:** Обычно Hibernate сам решает, когда сделать flush (перед выполнением запроса или перед коммитом). Вручную вызывают редко, например, для массовых вставок, чтобы освободить память.
*   **Что происходит:** SQL-запросы уходят в БД, но транзакция висит в состоянии «в процессе».
*   **Важно:** Если после `flush()` произойдет ошибка и вы сделаете `rollback()`, все изменения, отправленные через `flush`, **отменятся**.

### 2. `commit()` (Фиксация)
Это команда: **«Все, работа закончена, сделай flush (если нужно) и зафиксируй изменения навсегда»**.

*   **Что происходит:**
    1.  Вызывается `flush()` (если есть неотправленные изменения).
    2.  Отправляется команда `COMMIT` в базу данных.
    3.  Транзакция закрывается.
*   **Важно:** После этого данные сохранены даже если сервер упадет. Откатить уже нельзя.

### Пример на коде (Почему это важно)

```java
Session session = sessionFactory.openSession();
Transaction tx = session.beginTransaction();

User user = session.get(User.class, 1L);
user.setName("New Name");

// 1. Вызываем flush вручную
session.flush(); 
// SQL UPDATE ушел в БД. 
// Но транзакция ВСЕ ЕЩЕ ОТКРЫТА.

// 2. Происходит какая-то ошибка или логическое условие
if (someCondition) {
    tx.rollback(); 
    // !!! Изменения отменятся, несмотря на flush.
    // В базе останется старое имя.
} else {
    tx.commit(); 
    // !!! Изменения сохранятся навсегда.
}

session.close();
```

### Когда `flush()` происходит автоматически?
Вам редко нужно вызывать `flush()` вручную. Hibernate делает это сам в трех случаях:
1.  Перед выполнением любого запроса к БД (чтобы запрос видел актуальные данные).
2.  При вызове `transaction.commit()`.
3.  При явном вызове `session.flush()`.

### Итог
*   **`flush()`** = «Отправь SQL, но оставь возможность отменить».
*   **`commit()`** = «Отправь SQL и зафиксируй результат окончательно».
