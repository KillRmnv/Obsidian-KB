Жизненный цикл сущности (Entity Life Cycle) в Hibernate описывает состояния, в которых может находиться Java-объект, сопоставленный с таблицей базы данных, и его взаимосвязь с объектом `Session` (или `EntityManager` в JPA).

Понимание жизненного цикла критически важно для управления производительностью, целостностью данных и избежания ошибок (например, `LazyInitializationException`).

Существует **4 основных состояния** сущности:

---

### 1. Состояния сущности (States)

#### 1. Transient (Новый / Переходный)
*   **Описание:** Объект только что создан с помощью оператора `new`.
*   **Связь с Session:** Отсутствует.
*   **Связь с БД:** Отсутствует (в базе данных строки с таким ID еще нет).
*   **Идентификатор (ID):** Обычно отсутствует (или равен `null`/`0`), если только вы не присвоили его вручную.
*   **Судьба:** Если потерять все ссылки на объект, он будет удален сборщиком мусора (Garbage Collector).
*   **Пример:** `User user = new User();`

#### 2. Persistent (Управляемый / Постоянный)
*   **Описание:** Объект ассоциирован с открытой `Session`.
*   **Связь с Session:** Есть.
*   **Связь с БД:** Представлен в базе данных (есть соответствующая строка).
*   **Идентификатор (ID):** Присутствует.
*   **Особенность:** Любые изменения полей этого объекта **автоматически отслеживаются** Hibernate. При завершении транзакции или вызове `flush()` изменения синхронизируются с БД.
*   **Пример:** `session.get(User.class, 1L);` или `session.save(user);`

#### 3. Detached (Отсоединенный)
*   **Описание:** Объект был в состоянии *Persistent*, но `Session`, с которой он был связан, была закрыта (или объект был явно отсоединен).
*   **Связь с Session:** Отсутствует.
*   **Связь с БД:** Представлен в базе данных (строка есть).
*   **Идентификатор (ID):** Присутствует.
*   **Особенность:** Изменения полей **не отслеживаются**. Чтобы сохранить изменения в БД, объект нужно снова прикрепить к новой сессии.
*   **Типичный сценарий:** Объект сущности передается из слоя DAO в слой Controller или сериализуется в HTTP-сессию.
*   **Пример:** Объект, возвращенный из метода сервиса, после закрытия сессии.

#### 4. Removed (Удаленный)
*   **Описание:** Объект находится в состоянии *Persistent*, но был помечен на удаление.
*   **Связь с Session:** Есть (до момента `flush`).
*   **Связь с БД:** Строка будет удалена при следующем `flush()` или коммите транзакции.
*   **Особенность:** После сброса (flush) объект обычно переходит в состояние *Detached* (так как сессия перестает управлять удаленной записью).
*   **Пример:** `session.delete(user);`

---

### 2. Переходы между состояниями (Transitions)

Вот как перемещать сущность между состояниями с помощью методов Hibernate:

| Из состояния | В состояние | Метод / Действие |
| :--- | :--- | :--- |
| **Transient** | **Persistent** | `session.save()`, `session.persist()`, `session.saveOrUpdate()` |
| **Transient** | **Persistent** | `session.load()`, `session.get()` (загружает из БД сразу в Persistent) |
| **Persistent** | **Detached** | `session.close()`, `session.clear()`, `session.evict(object)` |
| **Detached** | **Persistent** | `session.update()`, `session.merge()`, `session.lock()`, `session.saveOrUpdate()` |
| **Persistent** | **Removed** | `session.delete()`, `session.remove()` |
| **Removed** | **Detached** | `session.flush()` (после выполнения SQL DELETE) |

#### Важные нюансы методов перехода:

1.  **`save()` vs `persist()`**:
    *   `save()`: Возвращает сгенерированный ID. Может выполнить `INSERT` немедленно (в зависимости от стратегии генерации ID).
    *   `persist()`: Ничего не возвращает. Гарантирует, что `INSERT` не будет выполнен до `flush()`. Предпочтительнее в JPA.
2.  **`update()` vs `merge()`** (для перехода Detached -> Persistent):
    *   `update()`: Пытается прикрепить существующий объект. Если в текущей сессии уже есть объект с таким же ID, выбросит исключение `NonUniqueObjectException`.
    *   `merge()`: Копирует состояние отсоединенного объекта в новый управляемый объект (который возвращает). Безопаснее, но создает лишний объект в памяти.

---

### 3. Механизм Flush (Сброс)

Переход в состояние *Persistent* не означает, что данные мгновенно записаны в БД. Они записываются при **Flush**.

*   **Что такое Flush:** Процесс синхронизации памяти (кэша первого уровня) с базой данных. Hibernate генерирует SQL-запросы (`INSERT`, `UPDATE`, `DELETE`).
*   **Когда происходит:**
    1.  Перед выполнением любого запроса к БД (чтобы запрос видел актуальные данные).
    2.  При коммите транзакции (`transaction.commit()`).
    3.  При явном вызове `session.flush()`.
*   **Режимы (`FlushMode`):**
    *   `AUTO` (по умолчанию): Сброс перед запросами и при коммите.
    *   `COMMIT`: Сброс только при коммите (может ускорить работу, но запросы внутри транзакции не увидят изменений).
    *   `MANUAL`: Сброс только по явному вызову `flush()`.

### 4. Callbacks (Жизненные циклы через аннотации)

Hibernate позволяет встраивать логику в определенные моменты жизненного цикла сущности с помощью аннотаций (JPA или Hibernate native).

| Аннотация | Когда срабатывает |
| :--- | :--- |
| `@PrePersist` | Перед сохранением новой сущности (перед `INSERT`). |
| `@PostPersist` | Сразу после сохранения в БД. |
| `@PreUpdate` | Перед обновлением сущности (перед `UPDATE`). |
| `@PostUpdate` | Сразу после обновления. |
| `@PreRemove` | Перед удалением (перед `DELETE`). |
| `@PostRemove` | Сразу после удаления. |
| `@PostLoad` | После загрузки сущности из БД (в состояние Persistent). |

**Пример:**
```java
@Entity
public class User {
    @PrePersist
    public void onCreate() {
        this.createdAt = LocalDateTime.now();
    }
}
```

### 5. Типичные проблемы и решения

#### 1. LazyInitializationException
*   **Суть:** Вы пытаетесь обратиться к лениво загружаемой коллекции (`@OneToMany(fetch = LAZY)`) в состоянии **Detached** (сессия закрыта).
*   **Решение:**
    *   Инициализировать коллекцию внутри открытой сессии (`Hibernate.initialize(user.getOrders())`).
    *   Использовать `JOIN FETCH` в запросе.
    *   Использовать паттерн "Open Session in View" (не рекомендуется в современных приложениях).
    *   Поменять стратегию загрузки на `EAGER` (не рекомендуется для коллекций).

#### 2. Проблемы с `merge()`
*   **Суть:** `merge()` возвращает *новый* объект. Если вы изменили объект, сделали `mergedUser = session.merge(user)`, а потом продолжаете работать со старым `user`, изменения не сохранятся.
*   **Решение:** Всегда работать с объектом, который вернул `merge()`.

#### 3. Версионирование (Optimistic Locking)
*   Для контроля целостности при работе с Detached объектами используют поле `@Version`.
*   Если два пользователя загрузили одну сущность (стала Detached), изменили её и пытаются сохранить, второй получит `OptimisticLockException`, так как версия в БД изменилась.

### 6. Сводная схема (Текстовая)

```text
       new User()
         (Transient)
            |
            | save(), persist(), get(), load()
            v
       (Persistent) <-----> [ База Данных ]
            |                     ^
            | close(), clear()    | delete()
            v                     |
       (Detached)                 |
            |                     |
            | update(), merge()   |
            +---------------------+
```

### Резюме

1.  **Transient:** Просто объект в куче, БД о нем не знает.
2.  **Persistent:** Объект под присмотром Hibernate. Изменения магически улетают в БД.
3.  **Detached:** Объект с ID, но без присмотра. Нужно "вернуть" его в сессию для сохранения.
4.  **Removed:** Объект, который скоро исчезнет из БД.
5.  **Flush:** Момент истины, когда память становится данными в БД.

Понимание этих состояний помогает писать эффективный код, избегать лишних запросов и понимать, когда данные реально сохраняются.
