 **Изменения отслеживаются автоматически, но записываются в базу данных не мгновенно, а только при синхронизации (flush) и коммите транзакции.**

Давайте разберем это подробно, чтобы избежать ошибок.

### 1. Автоматическое отслеживание (Dirty Checking)
Когда объект находится в состоянии **Persistent** (привязан к открытой сессии), Hibernate действительно **автоматически отслеживает** любые изменения его полей.

*   Вам **НЕ нужно** вызывать `session.update(obj)` или `session.save(obj)` для уже существующего объекта.
*   Достаточно просто вызвать сеттер: `user.setName("New Name")`.

Hibernate хранит "слепок" (snapshot) состояния объекта, который был загружен из БД. Перед отправкой запросов он сравнивает текущее состояние объекта со слепком. Если видит разницу — помечает объект как "грязный" (dirty).

### 2. Момент записи в БД (Flush & Commit)
Хотя Hibernate *знает* об изменениях сразу, он **не отправляет** SQL-запрос `UPDATE` в базу данных в ту же миллисекунду, когда вы вызвали сеттер.

Запись происходит в два этапа:

1.  **Flush (Сброс):** Hibernate генерирует и отправляет SQL-запросы в БД.
    *   Может произойти автоматически перед выполнением другого запроса.
    *   Может произойти автоматически перед `commit()`.
    *   Можно вызвать вручную: `session.flush()`.
2.  **Commit (Фиксация):** Транзакция подтверждается.
    *   Только после `transaction.commit()` изменения становятся **видимы для других транзакций** и окончательно сохраняются.
    *   Если сделать `flush()`, но не сделать `commit()`, изменения можно откатить (`rollback()`).

### 3. Пример на коде

```java
Session session = sessionFactory.openSession();
Transaction tx = session.beginTransaction();

// 1. Загружаем объект (теперь он Persistent)
User user = session.get(User.class, 1L); 

// 2. Меняем поле
user.setName("Ivan"); 
// !!! В этот момент в БД еще ничего не изменилось. 
// Hibernate просто запомнил, что имя стало "Ivan".

// 3. Другие запросы в этой же транзакции
// Если здесь сделать запрос к User, Hibernate увидит "Ivan" в своем кэше, 
// даже если в БД еще лежит "OldName".

// 4. Конец транзакции
tx.commit(); 
// !!! ТОЛЬКО ЗДЕСЬ происходит:
//    а) session.flush() -> отправка UPDATE в БД
//    б) фиксация транзакции -> данные сохранены навсегда

session.close();
```

### 4. Что будет, если нарушить правила?

| Сценарий | Результат |
| :--- | :--- |
| **Изменили поле, но забыли `commit()`** | Изменения **не сохранятся**. При закрытии сессии транзакция откатится. |
| **Изменили поле, сделали `commit()`, но сессия уже закрыта** | Объект перешел в состояние **Detached**. Изменения **не сохранятся**, так как никто не отслеживал объект в момент изменения. |
| **Изменили поле, сессия открыта, но нет транзакции** | Зависит от настроек (autocommit), но в современном Hibernate **требуется активная транзакция** для записи. Без `beginTransaction()` изменения скорее всего не применятся или вызовут ошибку. |
