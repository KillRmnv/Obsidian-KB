## 1. Packet Tracer - Who Hears the Broadcast? (3.1.1.5)

### Цель
Наблюдать за распространением broadcast трафика в сети с VLANами и понять, как VLANы изолируют broadcast домены.

### Основные концепции

**Broadcast Domain:**
- Область сети, в которой broadcast пакеты доставляются всем устройствам
- Без VLANов: весь коммутатор = 1 broadcast домен
- С VLANами: каждый VLAN = отдельный broadcast домен

**Collision Domain:**
- Область сети, где могут происходить коллизии
- На современных коммутаторах: каждый порт = отдельный collision домен

### Алгоритм выполнения

#### Часть 1: Генерация трафика с помощью ping

**Шаг 1: Обычный ping (unicast)**
```
1. Кликните на PC0
2. Desktop → Command Prompt
3. Введите команду: ping 192.168.1.8
4. Ping должен быть успешным
```

*Теория:* PC0 и PC8 находятся в одном VLANе (VLAN 10), поэтому они могут общаться.

#### Часть 2: Генерация и анализ broadcast трафика

**Шаг 1: Переключение в Simulation режим**
```
1. Кликните на вкладку Simulation (правый нижний угол)
2. Или нажмите Shift+S
```

**Шаг 2: Настройка фильтров**
```
1. В Simulation Panel кликните Edit Filters
2. Снимите галочку Show All/None
3. Поставьте галочку ICMP
```

**Шаг 3: Создание broadcast пакета**
```
1. Кликните на иконку Add Complex PDU (открытый конверт)
2. Кликните на PC0 (источник)
3. В диалоговом окне Create Complex PDU введите:
   - Destination IP Address: 255.255.255.255  (broadcast адрес)
   - Sequence Number: 1
   - One Shot Time: 0
4. Кликните Create PDU
```

*Теория:* **255.255.255.255** - это специальный broadcast адрес, который означает "отправить всем в локальной сети".

**Шаг 4: Наблюдение за распространением пакета**
```
1. Кликните Capture/Forward дважды
2. Наблюдайте за пакетом
```

**Вопрос**: Что произошло с пакетом?

**Ответ**: Пакет отправлен на коммутатор, затем коммутатор разослал его ТОЛЬКО на те порты, которые принадлежат VLAN 10 (кроме порта-источника).

**Шаг 5: Повторение для других VLANов**
```
1. Повторите процесс для PC8 (VLAN 10)
2. Повторите процесс для PC16 (VLAN 20)
```

#### Часть 3: Ответы на вопросы для размышления

**Вопрос 1**: Если ПК в VLAN 10 отправляет broadcast, какие устройства получат его?

**Ответ**: Все устройства в VLAN 10

**Вопрос 2**: Если ПК в VLAN 20 отправляет broadcast, какие устройства получат его?

**Ответ**: Все устройства в VLAN 20

**Вопрос 3**: Если ПК в VLAN 30 отправляет broadcast, какие устройства получат его?

**Ответ**: Все устройства в VLAN 30

**Вопрос 4**: Что происходит с фреймом, отправленным от ПК в VLAN 10 к ПК в VLAN 30?

**Ответ**: Фрейм будет отброшен (dropped), потому что они не находятся в одном VLAN.

**Вопрос 5**: Какие порты загорятся, если ПК на порту 11 отправляет unicast ПК на порту 13?

**Ответ**: Порты 11 и 13 (оба в одном VLANе)

**Вопрос 6**: Какие порты загорятся, если ПК на порту 2 отправляет unicast ПК на порту 23?

**Ответ**: Пакет будет отброшен (разные VLANы)

**Вопрос 7**: Что является collision доменами на коммутаторе?

**Ответ**: Каждый порт - свой собственный collision домен

**Вопрос 8**: Что является broadcast доменами на коммутаторе?

**Ответ**: Каждый VLAN - свой собственный broadcast домен

### Ключевые выводы

```
БЕЗ VLANов:
- Broadcast идет на ВСЕ порты коммутатора
- Один большой broadcast домен
- Много лишнего трафика

С VLANами:
- Broadcast идет только на порты того же VLANа
- Каждый VLAN = отдельный broadcast домен
- Изоляция трафика
- Лучшая производительность и безопасность
```

## 2. Packet Tracer - Investigating a VLAN Implementation (3.1.2.7)

### Цель
Сравнить поведение broadcast трафика с VLANами и без них, понять важность VLAN конфигурации.

### Алгоритм выполнения

#### Часть 1: Наблюдение за broadcast трафиком С VLANами

**Шаг 1: Ping от PC1 к PC6**

```
1. Ждите, пока все индикаторы соединений станут зелеными
   (или кликните Fast Forward Time несколько раз)
2. Переключитесь в Simulation режим
3. Используйте Add Simple PDU tool
4. Кликните на PC1, затем на PC6
5. Кликните Capture/Forward для пошагового просмотра
```

**Вопрос**: Были ли пинги успешны? Почему?

**Ответ**: Нет, пинги НЕ успешны, потому что PC1 находится в VLAN 10, а PC6 - в VLAN 30. Они логически изолированы.

**Анализ процесса:**
```
1. PC1 отправляет ARP запрос (broadcast)
2. S1 получает ARP запрос на порт Fa0/1 (VLAN 10)
3. S1 пересылает ARP только на порты VLAN 10:
   - S2 через Fa0/1
   - PC4 через Fa0/11
4. PC6 НИКОГДА не получает ARP запрос
5. PC1 не получает ARP ответ
6. Ping проваливается
```

**Шаг 2: Ping от PC1 к PC4**

```
1. Кликните New под Scenario 0
2. Используйте Add Simple PDU
3. Кликните на PC1, затем на PC4
4. Кликните Capture/Forward
```

**Вопрос**: Были ли пинги успешны? Почему?

**Ответ**: Да, пинги успешны, потому что PC1 и PC4 оба принадлежат VLAN 10.

**Вопрос**: Почему S1 также пересылает пакет на PC7?

**Ответ**: Потому что PC7 также принадлежит VLAN 10, а ARP запрос - это broadcast в пределах VLAN.

#### Часть 2: Наблюдение за broadcast трафиком БЕЗ VLANов

**Шаг 1: Очистка конфигурации на всех коммутаторах**

```
На каждом коммутаторе (S1, S2, S3):

Switch> enable
Switch# erase startup-config
Erasing the nvram filesystem will remove all configuration files! Continue? [confirm] [ENTER]
[OK]
Erase of nvram: complete

Switch# delete vlan.dat
Delete filename [vlan.dat]? [ENTER]
Delete flash:vlan.dat? [confirm] [ENTER]

Switch# reload
Proceed with reload? [confirm] [ENTER]
```

*ВАЖНО:* 
- `erase startup-config` удаляет конфигурацию
- `delete vlan.dat` удаляет VLAN базу данных (хранится отдельно!)
- `reload` перезагружает коммутатор

**Вопрос**: Где хранится VLAN файл на коммутаторе?

**Ответ**: flash:vlan.dat

**Шаг 2: Ping после удаления VLANов**

```
1. Дождитесь зеленых индикаторов
2. Simulation режим
3. Add Simple PDU от PC1 к PC6
4. Кликните Capture/Forward
```

**Наблюдение**: Теперь коммутаторы пересылают ARP запросы на ВСЕ порты (кроме порта-источника).

**Вывод**: Это стандартное поведение коммутаторов БЕЗ VLANов. Вот почему VLANы улучшают производительность сети - broadcast трафик ограничен в пределах каждого VLANа.

### Сравнительная таблица

```
Параметр                  | С VLANами                    | Без VLANов
--------------------------|------------------------------|----------------------------
Broadcast домен           | Каждый VLAN отдельно         | Весь коммутатор
ARP запросы               | Только в пределах VLAN       | На все порты
Безопасность              | Изоляция между VLANами       | Нет изоляции
Производительность        | Лучше (меньше broadcast)     | Хуже (много broadcast)
Связь между сетями        | Требуется маршрутизатор/L3   | Прямая связь
```

**Шаг 3: Проверка потери связности**


**Вопрос**: Хотя access порты назначены соответствующим VLANам, были ли пинги успешны? Почему?

**Ответ**: Нет, пинги провалились. Хотя PC1 и PC4 находятся в одном VLANе (VLAN 10), trunk порты между коммутаторами НЕ настроены. Трафик VLANов не может проходить между коммутаторами.

**Вопрос**: Что можно сделать для решения проблемы?

**Ответ**: Настроить trunk порты между коммутаторами:
```
На S1:
S1(config)# interface GigabitEthernet 0/1
S1(config-if)# switchport mode trunk

На S2:
S2(config)# interface GigabitEthernet 0/1
S2(config-if)# switchport mode trunk
```

## 4. Packet Tracer - Configuring Router-on-a-Stick Inter-VLAN Routing (5.1.3.6)

### Цель
Настроить Router-on-a-Stick для обеспечения связности между VLANами.

### Основные концепции

**Router-on-a-Stick:**
- Один физический интерфейс маршрутизатора
- Несколько логических субинтерфейсов (subinterfaces)
- Каждый субинтерфейс соответствует одному VLANу
- Использует 802.1Q encapsulation (VLAN tagging)

**Зачем нужен?**
- VLANы изолированы друг от друга на Layer 2
- Для связи между VLANами нужен Layer 3 (маршрутизатор)
- Router-on-a-Stick - экономичное решение (один интерфейс вместо многих)

### Алгоритм выполнения

#### Часть 1: Тестирование связности БЕЗ Inter-VLAN маршрутизации

**Шаг 1: Ping между PC1 и PC3**

```
1. Дождитесь зеленых индикаторов или Fast Forward Time
2. PC1 → Desktop → Command Prompt
3. ping 172.17.30.10   (IP адрес PC3)
```

**Результат**: Ping провалился

**Причина**: PC1 в VLAN 10 (172.17.10.0/24), PC3 в VLAN 30 (172.17.30.0/24). Без маршрутизатора они не могут общаться.

**Шаг 2: Simulation режим для мониторинга**

```
1. Переключитесь в Simulation режим
2. Кликните Capture/Forward
3. Наблюдайте, как ping никогда не покидает PC1
```

**Вопрос**: Какой процесс провалился и почему?

**Ответ**: ARP процесс провалился. ARP запрос никогда не достиг PC3. PC1 и PC3 не в одной сети, поэтому S1 не пересылает ARP запрос PC1 (broadcast пакет) в VLAN 30. PC1 никогда не получает MAC адрес PC3. Без MAC адреса PC1 не может создать ICMP echo request.

#### Часть 2: Добавление VLANов на коммутатор

**Шаг 1: Создание VLANов на S1**

```
S1# configure terminal
S1(config)# vlan 10
S1(config-vlan)# vlan 30
S1(config-vlan)# exit
```

**Шаг 2: Назначение VLANов портам**

```
# PC1 → VLAN 10
S1(config)# interface FastEthernet 0/11
S1(config-if)# switchport mode access
S1(config-if)# switchport access vlan 10
S1(config-if)# exit

# PC3 → VLAN 30
S1(config)# interface FastEthernet 0/6
S1(config-if)# switchport mode access
S1(config-if)# switchport access vlan 30
S1(config-if)# exit
```

**Шаг 3: Проверка конфигурации**

```
S1# show vlan brief

VLAN Name                             Status    Ports
---- -------------------------------- --------- -------------------------------
1    default                          active    Fa0/1, Fa0/2, Fa0/3, Fa0/4,
                                                Fa0/5, Fa0/7, Fa0/8, Fa0/9,
                                                Fa0/10, Fa0/12, Fa0/13, Fa0/14,
                                                ..., Gig0/1, Gig0/2
10   VLAN0010                         active    Fa0/11
30   VLAN0030                         active    Fa0/6
```

**Шаг 4: Тестирование связности между PC1 и PC3**

```
От PC1: ping 172.17.30.10
```

**Вопрос**: Были ли пинги успешны?

**Ответ**: Нет, потому что каждый VLAN - это отдельная сеть, и требуется маршрутизатор или Layer 3 коммутатор для связи между ними.

#### Часть 3: Конфигурация субинтерфейсов

**Шаг 1: Конфигурация субинтерфейса для VLAN 10**

```
R1# configure terminal

# Создание субинтерфейса G0/0.10
R1(config)# interface GigabitEthernet 0/0.10

# Установка encapsulation 802.1Q с VLAN ID 10
R1(config-subif)# encapsulation dot1Q 10

# Назначение IP адреса (шлюз для VLAN 10)
R1(config-subif)# ip address 172.17.10.1 255.255.255.0

R1(config-subif)# exit
```

*Теория:*
- **G0/0.10** - субинтерфейс (номер .10 обычно соответствует VLAN 10)
- **encapsulation dot1Q 10** - VLAN tagging, связывает субинтерфейс с VLAN 10
- **172.17.10.1** - это IP адрес, который будет default gateway для устройств в VLAN 10

**Шаг 2: Конфигурация субинтерфейса для VLAN 30**

```
R1(config)# interface GigabitEthernet 0/0.30
R1(config-subif)# encapsulation dot1Q 30
R1(config-subif)# ip address 172.17.30.1 255.255.255.0
R1(config-subif)# exit
```

**Шаг 3: Проверка конфигурации**

```
R1# show ip interface brief

Interface              IP-Address      OK? Method Status                Protocol
GigabitEthernet0/0     unassigned      YES unset  administratively down down
GigabitEthernet0/0.10  172.17.10.1     YES manual administratively down down
GigabitEthernet0/0.30  172.17.30.1     YES manual administratively down down
```

**Проблема**: Оба субинтерфейса показывают "down"!

**Причина**: Субинтерфейсы - это виртуальные интерфейсы, связанные с физическим интерфейсом. Для включения субинтерфейсов нужно включить физический интерфейс!

**Решение**:
```
R1(config)# interface GigabitEthernet 0/0
R1(config-if)# no shutdown
R1(config-if)# exit
```

**Проверка**:
```
R1# show ip interface brief

Interface              IP-Address      OK? Method Status                Protocol
GigabitEthernet0/0     unassigned      YES unset  up                    up
GigabitEthernet0/0.10  172.17.10.1     YES manual up                    up
GigabitEthernet0/0.30  172.17.30.1     YES manual up                    up
```

Теперь субинтерфейсы активны!

#### Часть 4: Тестирование связности С Inter-VLAN маршрутизацией

**Шаг 1: Ping между PC1 и PC3**

```
От PC1: ping 172.17.30.10
```

**Результат**: Ping всё ещё провалился!

**Почему?**: Коммутатор S1 ещё не настроен для работы с VLANами через trunk.

**Шаг 2: Включение trunking**

```
S1# show vlan

Вопрос: Какому VLANу назначен порт G0/1?
Ответ: VLAN 1 (по умолчанию)
```

**Проблема**: Маршрутизатор R1 настроен с несколькими субинтерфейсами для разных VLANов, но порт коммутатора, подключенный к маршрутизатору, находится в режиме access (VLAN 1).

```

S1(config)# interface GigabitEthernet 0/1

S1(config-if)# exit
```

*Теория:* **Trunk** порт может передавать трафик для ВСЕХ VLANов, используя VLAN tagging (802.1Q).

**Проверка**:
```
S1# show vlan

Порт G0/1 больше НЕ указан в VLAN 1
```

**Альтернативная проверка**:
```

Gig0/1      on           802.1q         trunking      1

Port        Vlans allowed on trunk
Gig0/1      1-1005

Port        Vlans allowed and active in management domain
Gig0/1      1,10,30

Port        Vlans in spanning tree forwarding state and not pruned
Gig0/1      1,10,30
```

**Шаг 3: Simulation режим для мониторинга**

```
1. Переключитесь в Simulation режим
2. Отправьте ping от PC1 к PC3
3. Кликните Capture/Forward
```

**Процесс:**
```

2. ARP запросы и ответы между R1 и S3
3. PC1 может инкапсулировать ICMP echo request с правильной data-link информацией
4. R1 маршрутизирует запрос к PC3

```

### Основные концепции

**DHCP (Dynamic Host Configuration Protocol):**
- Автоматически назначает IP адреса клиентам
- Предоставляет: IP адрес, маску подсети, default gateway, DNS сервер
- Экономит время администрирования

**DNS (Domain Name System):**
- Преобразует имена (example.com) в IP адреса (192.168.1.1)
- Упрощает доступ к ресурсам
- Иерархическая система

### Процесс работы DHCP

```
1. DHCP Discover  (Client broadcast: "Кто-нибудь есть?")
   ↓
2. DHCP Offer     (Server: "Я предлагаю тебе 192.168.0.100")
   ↓
3. DHCP Request   (Client: "Я хочу 192.168.0.100")
   ↓
4. DHCP Acknowledge (Server: "Подтверждаю, 192.168.0.100 твой на 24 часа")
```

### Процесс работы DNS

```
1. User вводит: www.example.com
   ↓
2. Компьютер проверяет локальный DNS cache
   ↓
3. Если не найдено, отправляет запрос DNS серверу (64.100.8.8)
   ↓
4. DNS сервер отвечает: "www.example.com = 192.168.1.1"
   ↓
5. Компьютер подключается к 192.168.1.1
```

### Важные команды

```
Команда                  | Назначение
-------------------------|------------------------------------------
ipconfig /all            | Просмотр полной IP конфигурации (Windows)
ipconfig /release        | Освободить DHCP адрес
ipconfig /renew          | Запросить новый DHCP адрес
nslookup [domain]        | Запросить DNS сервер
ping [IP/domain]         | Проверить связность
```

## Общие советы по экзамену

### Ключевые концепции VLANs

```
VLAN = Virtual Local Area Network
- Логическая сегментация сети на Layer 2
- Каждый VLAN = отдельный broadcast домен
- Изоляция трафика и улучшение безопасности
```

### Типы портов коммутатора

```
Access Port:
- Принадлежит одному VLANу
- Подключается к конечным устройствам (ПК, принтеры)
- Команды: switchport mode access
          switchport access vlan [number]

Trunk Port:
- Передает трафик ВСЕХ VLANов
- Подключается к другим коммутаторам или маршрутизаторам
- Использует VLAN tagging (802.1Q)
- Команды: switchport mode trunk
          switchport trunk native vlan [number]
```

### Inter-VLAN Routing методы

```
1. Router-on-a-Stick:
   - Один физический интерфейс
   - Множество субинтерфейсов (по одному на VLAN)
   - Trunk порт между коммутатором и маршрутизатором
   - Команды: interface g0/0.[vlan-id]
             encapsulation dot1Q [vlan-id]

2. Layer 3 Switch:
   - Коммутатор с возможностями маршрутизации
   - SVIs (Switched Virtual Interfaces)
   - Быстрее, чем Router-on-a-Stick
```

### Типичные ошибки

```
❌ Забывают включить trunk между коммутаторами
   ✓ switchport mode trunk на обоих концах

❌ Забывают включить физический интерфейс для субинтерфейсов
   ✓ no shutdown на G0/0

❌ Не настраивают encapsulation на субинтерфейсах
   ✓ encapsulation dot1Q [vlan-id]

❌ Забывают команду login на line console/vty
   ✓ login или login local

❌ Путают wildcard маску с subnet маской
   ✓ Wildcard = инверсия маски

❌ Не сохраняют конфигурацию
   ✓ copy running-config startup-config
```
