## 1. Packet Tracer - Configure Initial Router Settings (6.4.1.2)

### Цель
Выполнить базовую конфигурацию маршрутизатора: пароли, MOTD баннер, сохранение конфигурации.

### Алгоритм выполнения

#### Часть 1: Проверка конфигурации маршрутизатора

**Шаг 1: Подключение консольным кабелем**
```
1. Кликните на Connections (молния)
2. Выберите Console кабель
3. Кликните на ПК → выберите RS 232
4. Кликните на маршрутизатор → выберите Console порт
5. Откройте Terminal на ПК
```

**Шаг 2: Просмотр текущей конфигурации**
```
Router> enable
Router# show running-config
```

**Вопросы для проверки**:
- Имя хоста маршрутизатора? 
- Сколько Fast Ethernet интерфейсов?
- Сколько Gigabit Ethernet интерфейсов?
- Сколько Serial интерфейсов?
- Диапазон VTY линий?

#### Часть 2: Конфигурация маршрутизатора

**Шаг 1: Задание имени маршрутизатора**
```
Router> enable
Router# configure terminal
Router(config)# hostname R1
R1(config)#
```

**Шаг 2: Защита консольной линии**
```
R1(config)# line console 0
R1(config-line)# password letmein
R1(config-line)# login
R1(config-line)# exit
```

**Шаг 3: Защита VTY линий (удаленный доступ)**
```
R1(config)# line vty 0 4
R1(config-line)# password letmein
R1(config-line)# login
R1(config-line)# exit
```

**Шаг 4: Защита привилегированного режима (неэшифрованный пароль)**
```
R1(config)# enable password cisco
```

**Шаг 5: Защита привилегированного режима (зашифрованный пароль)**
```
R1(config)# enable secret itsasecret
```

*Теория:* `enable secret` имеет приоритет над `enable password`. Используется команда `enable secret` для более высокого уровня безопасности.

**Шаг 6: Шифрование всех паролей**
```
R1(config)# service password-encryption
```

**Шаг 7: Конфигурация MOTD баннера**
```
R1(config)# banner motd #Unauthorized access is strictly prohibited.#
# или:
R1(config)# banner motd $Unauthorized access is strictly prohibited.$
```

*Важно:* Баннер ДОЛЖЕН содержать слово "access" (по условию некоторых заданий).

**Шаг 8: Сохранение конфигурации**
```
R1(config)# exit
R1# copy running-config startup-config
или:
R1# write memory
```

#### Часть 3: Проверка конфигурации

**Шаг 1: Просмотр конфигурации**
```
R1# show running-config
```

**Шаг 2: Выход и повторный вход для проверки паролей**
```
R1# exit
R1 con0 is now available
Press RETURN to get started.

[Нажимаете ENTER]

Unauthorized access is strictly prohibited.
User Access Verification
Password: [введите letmein для консоли]
```

### Полная последовательность команд

```
enable
configure terminal
hostname R1
line console 0
password letmein
login
exit
line vty 0 4
password letmein
login
exit
enable password cisco
enable secret itsasecret
service password-encryption
banner motd $Unauthorized access is strictly prohibited.$
exit
copy running-config startup-config
```

## 2. Packet Tracer - Connect a Router to a LAN (6.4.3.3)

### Цель
Конфигурировать интерфейсы маршрутизатора, подключить маршрутизатор к локальной сети.

### Алгоритм выполнения

#### Часть 1: Изучение топологии сети

**Шаг 1: Просмотр информации о маршрутизаторе**
```
R1# show ip interface brief
```

Этот вывод показывает:
- Все интерфейсы маршрутизатора
- IP адреса (если уже настроены)
- Статус интерфейсов (up/down)

#### Часть 2: Конфигурация интерфейсов маршрутизатора

**Шаг 1: Конфигурация GigabitEthernet 0/0**

Пример адресации:
```
Device      | Interface | IP Address      | Subnet Mask       | Default Gateway
------------|-----------|-----------------|-------------------|----------------
R1          | G0/0      | 192.168.10.1    | 255.255.255.0     | N/A
R1          | G0/1      | 192.168.11.1    | 255.255.255.0     | N/A
S1          | VLAN 1    | 192.168.10.2    | 255.255.255.0     | 192.168.10.1
PC1         | NIC       | 192.168.10.100  | 255.255.255.0     | 192.168.10.1
```

**Конфигурация**:
```
R1# configure terminal
R1(config)# interface gigabitEthernet 0/0
R1(config-if)# description LAN connection to S1
R1(config-if)# ip address 192.168.10.1 255.255.255.0
R1(config-if)# no shutdown    ← ОБЯЗАТЕЛЬНО! Включает интерфейс
R1(config-if)# exit
```

**Шаг 2: Конфигурация GigabitEthernet 0/1**
```
R1(config)# interface gigabitEthernet 0/1
R1(config-if)# description LAN connection to S2
R1(config-if)# ip address 192.168.11.1 255.255.255.0
R1(config-if)# no shutdown
R1(config-if)# exit
```

**Шаг 3: Сохранение конфигурации**
```
R1(config)# exit
R1# copy running-config startup-config
```

#### Часть 3: Проверка конфигурации

**Шаг 1: Проверка интерфейсов**
```
R1# show ip interface brief

Результат должен показать:
GigabitEthernet0/0 192.168.10.1    YES manual up      up
GigabitEthernet0/1 192.168.11.1    YES manual up      up
```

*ВАЖНО:* Если интерфейс показывает "down down", значит вы забыли `no shutdown`.

**Шаг 2: Проверка маршрутизационной таблицы**
```
R1# show ip route

Результат должен показать:
C    192.168.10.0/24 is directly connected, GigabitEthernet0/0
C    192.168.11.0/24 is directly connected, GigabitEthernet0/1
L    192.168.10.1/32 is directly connected, GigabitEthernet0/0
L    192.168.11.1/32 is directly connected, GigabitEthernet0/1

C = Connected (прямо подключено)
L = Local (локальный адрес интерфейса)
```

#### Часть 4: Тестирование связности

**Шаг 1: Проверка ping с ПК на маршрутизатор**
```
На PC1:
1. Command Prompt
2. ping 192.168.10.1      (должен пройти)

На PC4:
1. Command Prompt
2. ping 192.168.11.1      (должен пройти)
```

**Шаг 2: Проверка ping между ПК разных сетей (если есть маршрут)**
```
На PC1:
ping 192.168.11.1         (может не пройти, если R2 не настроен)
```

### Полная последовательность команд

```
enable
configure terminal
interface gigabitEthernet 0/0
description LAN connection to S1
ip address 192.168.10.1 255.255.255.0
no shutdown
exit
interface gigabitEthernet 0/1
description LAN connection to S2
ip address 192.168.11.1 255.255.255.0
no shutdown
exit
exit
copy running-config startup-config
```

### Ключевые моменты
- **no shutdown** - ОБЯЗАТЕЛЬНАЯ команда для включения интерфейса
- **description** - рекомендуется для документации
- Маршрутизатор автоматически добавляет подключенные маршруты в таблицу маршрутизации

## 3. Packet Tracer - Configuring IPv4 Static and Default Routes (6.2.2.4)

### Цель
Настроить статические маршруты и маршруты по умолчанию для направления трафика в несмежные сети.

### Алгоритм выполнения

#### Часть 1: Анализ сети

**Вопрос**: Сколько статических маршрутов требуется каждому маршрутизатору?
```
Ответ: 
- R1 нужно 3 статических маршрута (в несмежные сети)
- R2 нужно 2 статических маршрута
- R3 нужно 3 статических маршрута
```

*Формула:* 
```
Количество статических маршрутов = (Всего сетей) - (Прямо подключенных сетей)
```

#### Часть 2: Типы статических маршрутов

**Тип 1: Рекурсивный статический маршрут (uses next hop IP)**

Требует ДВА поиска в таблице маршрутизации:
1. Найти маршрут к next hop адресу
2. Найти exit интерфейс для next hop

```
R1(config)# ip route 172.31.0.0 255.255.255.0 172.31.1.193
             [сеть назначения]  [маска]      [next hop IP]
```

**Синтаксис:**
```
ip route [destination network] [subnet mask] [next hop IP address]
```

**Пример - конфигурация R1:**
```
R1# configure terminal
R1(config)# ip route 172.31.0.0 255.255.255.0 172.31.1.193
R1(config)# ip route 172.31.1.196 255.255.255.252 172.31.1.193
R1(config)# ip route 172.31.1.128 255.255.255.192 172.31.1.193
R1(config)# exit
```

**Тип 2: Прямо подключенный статический маршрут (uses exit interface)**

Требует ОДИН поиск в таблице маршрутизации.

```
R2(config)# ip route 172.31.1.0 255.255.255.128 Serial0/0/0
             [сеть назначения]  [маска]         [exit interface]
```

**Синтаксис:**
```
ip route [destination network] [subnet mask] [exit interface]
```

**Пример - конфигурация R2:**
```
R2# configure terminal
R2(config)# ip route 172.31.1.0 255.255.255.128 Serial0/0/0
R2(config)# ip route 172.31.1.128 255.255.255.192 Serial0/0/1
R2(config)# exit
```

**Тип 3: Маршрут по умолчанию (Default Route)**

Используется как "шлюз последней надежды" (gateway of last resort) для всех неизвестных сетей.

```
R3(config)# ip route 0.0.0.0 0.0.0.0 Serial0/0/1
             [всем сетям] [маска] [next hop или interface]
```

**Синтаксис:**
```
ip route 0.0.0.0 0.0.0.0 [next hop IP or exit interface]
```

*Теория:* 0.0.0.0/0 означает "все возможные сети". Это маршрут, используемый маршрутизатором, когда он не знает, куда отправить пакет.

**Пример - конфигурация R3:**
```
R3# configure terminal
R3(config)# ip route 0.0.0.0 0.0.0.0 Serial0/0/1
R3(config)# exit
```

**Тип 4: Полностью определенный маршрут (Fully Specified Route) - ДЛЯ ДОКУМЕНТАЦИИ**

```
R3(config)# ip route 172.31.0.0 255.255.255.0 Serial0/0/1 172.31.1.197
             [сеть назначения]  [маска]    [exit interface] [next hop IP]
```

*Примечание:* Packet Tracer может не поддерживать это. Используется для документации.

#### Часть 3: Проверка маршрутов

**Команда 1: Просмотр всей таблицы маршрутизации**
```
R1# show ip route

Результат:
C    172.31.1.0/25 is directly connected, GigabitEthernet0/0
S    172.31.0.0/24 via 172.31.1.193, 00:00:05, Serial0/0/0
S    172.31.1.192/30 via 172.31.1.193, 00:00:05, Serial0/0/0
S    172.31.1.128/26 via 172.31.1.193, 00:00:05, Serial0/0/0

C = Connected (прямо подключено)
S = Static (статический маршрут)
```

**Команда 2: Просмотр только статических маршрутов**
```
R1# show ip route static

Результат:
S    172.31.0.0/24 [1/0] via 172.31.1.193
S    172.31.1.192/30 [1/0] via 172.31.1.193
S    172.31.1.128/26 [1/0] via 172.31.1.193
```

**Команда 3: Просмотр маршрута к конкретной сети**
```
R1# show ip route 172.31.0.0

Результат:
Routing entry for 172.31.0.0/24
  Known via "static", distance 1, metric 0
  Routing Descriptor Blocks:
  * 172.31.1.193
      Route metric is 0, traffic share count is 1
```

#### Часть 4: Тестирование связности

**Шаг 1: Ping между PC в разных сетях**
```
На PC1 (в сети R1):
ping 172.31.0.254    (PC2 в сети R2)
ping 172.31.1.190    (PC3 в сети R3)

Оба должны быть успешны
```

**Шаг 2: Ping между маршрутизаторами**
```
На R1:
ping 172.31.0.1      (LAN интерфейс R2)
ping 172.31.1.129    (LAN интерфейс R3)
```

### Таблица статических маршрутов

```
Router | Destination Network | Type             | Next Hop IP / Exit Interface
-------|---------------------|------------------|----------------------------
R1     | 172.31.0.0/24       | Recursive        | 172.31.1.193 (next hop)
R1     | 172.31.1.192/30     | Recursive        | 172.31.1.193 (WAN link)
R1     | 172.31.1.128/26     | Recursive        | 172.31.1.193 (next hop)
R2     | 172.31.1.0/25       | Directly attached| Serial0/0/0 (exit interface)
R2     | 172.31.1.128/26     | Directly attached| Serial0/0/1 (exit interface)
R3     | (all networks)      | Default route    | Serial0/0/1 (exit interface)
```

### Ключевые различия

```
Рекурсивный              | Прямо подключенный        | По умолчанию
------------------------|--------------------------|-----------------------
ip route ... [next hop IP] | ip route ... [interface] | ip route 0.0.0.0 0.0.0.0
Требует 2 поиска        | Требует 1 поиск          | Требует 1 поиск
Медленнее                | Быстрее                  | Быстро (catch-all)
Используется обычно     | Используется в pt        | Для неизвестных сетей
```

## 4. Packet Tracer - Configuring a Floating Static Route (6.4.3.4)

### Цель
Настроить резервный маршрут с бОльшим административным расстоянием для обеспечения отказоустойчивости.

### Алгоритм выполнения

#### Часть 1: Основные концепции

**Административное расстояние (Administrative Distance)**

```
Протокол                         | AD (Administrative Distance)
---------------------------------|---------------------------
Прямо подключенный интерфейс    | 0 (наиболее надежный)
Статический маршрут (directly)  | 0
Статический маршрут (recursive)  | 1
EIGRP                           | 90
OSPF                            | 110
RIP                             | 120
Неизвестный маршрут             | 255 (не используется)
```

*Теория:* Маршрутизатор выбирает маршрут с НАИМЕНЬШИМ AD.

**Floating Static Route:**
- Это маршрут с ПОВЫШЕННЫМ AD (например, 5 вместо 1)
- Он НЕ видно в таблице маршрутизации, пока первичный маршрут работает
- Когда первичный маршрут падает, floating маршрут становится активным

#### Часть 2: Конфигурация

**Шаг 1: Конфигурация первичного статического маршрута (AD = 0 или 1)**

```
Edge_Router> enable
Edge_Router# configure terminal
Edge_Router(config)# ip route 0.0.0.0 0.0.0.0 Serial0/0/0
                    [default route]              [primary ISP]
```

*Примечание:* Directly attached маршрут имеет AD = 0

**Шаг 2: Проверка маршрута в таблице**
```
Edge_Router# show ip route

Результат:
S* 0.0.0.0/0 is directly connected, Serial0/0/0
```

**Шаг 3: Тестирование первичного маршрута**
```
На PC-A:
tracert 198.0.0.10   (Web Server)

Результат должен быть:
1 ms  192.168.10.1    (default gateway)
2 ms  10.10.10.1      (ISP1)
3 ms  198.0.0.10      (Web Server)
```

**Шаг 4: Конфигурация резервного маршрута (AD = 5 или выше)**

```
Edge_Router(config)# ip route 0.0.0.0 0.0.0.0 Serial0/0/1 5
                    [default route]              [backup ISP] [AD=5]
```

*ВАЖНО:* Последний параметр (5) это административное расстояние!

**Шаг 5: Проверка running-config**
```
Edge_Router# show running-config | include ip route

Результат:
ip route 0.0.0.0 0.0.0.0 Serial0/0/0
ip route 0.0.0.0 0.0.0.0 Serial0/0/1 5
```

**Шаг 6: Проверка таблицы маршрутизации**
```
Edge_Router# show ip route

Результат:
S* 0.0.0.0/0 is directly connected, Serial0/0/0
```

*ВАЖНО:* Floating маршрут НЕ видно в таблице! Это нормально.

#### Часть 3: Тестирование failover

**Шаг 1: Отключение первичного маршрута**
```
Edge_Router(config)# interface Serial0/0/0
Edge_Router(config-if)# shutdown
Edge_Router(config-if)# exit
```

*ВАЖНО:* Выключение интерфейса эмулирует отказ ISP1

**Шаг 2: Проверка таблицы маршрутизации**
```
Edge_Router# show ip route

Результат (ИЗМЕНИЛСЯ!):
S* 0.0.0.0/0 [5/0] via 10.10.10.5, Serial0/0/1
```

Теперь floating маршрут ВИДЕН потому что первичный вышел!

**Шаг 3: Тестирование backup маршрута**
```
На PC-A:
tracert 198.0.0.10   (Web Server)

Результат должен быть:
1 ms  192.168.10.1    (default gateway)
2 ms  10.10.10.5      (ISP2) ← ИЗМЕНИОСЬ!
3 ms  198.0.0.10      (Web Server)
```

Трафик теперь идет через ISP2!

**Шаг 4: Восстановление первичного маршрута**
```
Edge_Router(config)# interface Serial0/0/0
Edge_Router(config-if)# no shutdown
Edge_Router(config-if)# exit
```

**Шаг 5: Проверка возврата на первичный маршрут**
```
Edge_Router# show ip route

Результат:
S* 0.0.0.0/0 is directly connected, Serial0/0/0
```

Floating маршрут исчез - вернулись к первичному!

**Шаг 6: Тестирование восстановления**
```
На PC-A:
tracert 198.0.0.10   (Web Server)

Результат:
1 ms  192.168.10.1    (default gateway)
2 ms  10.10.10.1      (ISP1) ← ВЕРНУЛОСЬ!
3 ms  198.0.0.10      (Web Server)
```

### Полная конфигурация Edge_Router

```
enable
configure terminal
ip route 0.0.0.0 0.0.0.0 Serial0/0/0
ip route 0.0.0.0 0.0.0.0 Serial0/0/1 5
exit
copy running-config startup-config
```

### Ключевые команды

```
Команда                              | Назначение
-------------------------------------|------------------------------------------
ip route 0.0.0.0 0.0.0.0 s0/0/0     | Первичный маршрут (AD = 0)
ip route 0.0.0.0 0.0.0.0 s0/0/1 5   | Резервный маршрут (AD = 5)
show ip route                        | Показывает активные маршруты
show running-config | include route  | Показывает ВСЕ маршруты (включая floating)
interface Serial0/0/0                | Конфигурирование интерфейса
shutdown                             | Отключение интерфейса (эмуляция отказа)
no shutdown                          | Включение интерфейса (восстановление)
```

## 5. Packet Tracer - Troubleshooting Default Gateway Issues (6.4.3.4)

### Цель
Идентифицировать и исправить проблемы с конфигурацией Default Gateway.

### Алгоритм выполнения

#### Часть 1: Идентификация проблем

**Важные концепции:**
- **Default Gateway** - адрес маршрутизатора в локальной сети
- Для связи с устройствами в ДРУГИХ сетях нужен Default Gateway
- На коммутаторе настраивается команда `ip default-gateway`
- На ПК настраивается через GUI или `ipconfig`

**Типичные проблемы:**
```
Проблема                          | Симптом
----------------------------------|------------------------------------------
Неправильный IP на ПК             | Не может пинговать в своей сети
Неправильный Default Gateway      | Может пинговать локальных, но не удаленных
Missing IP на Switch              | Других устройств нет в VLAN, можно пинговать только в сети
Missing Default Gateway на Switch  | Удаленные ПК не могут достучаться к свичу
```

**Проверка конфигурации ПК:**
```
На Windows ПК:
1. Command Prompt
2. ipconfig /all
3. Проверить:
   - IPv4 Address
   - Subnet Mask
   - Default Gateway
```

**Проверка конфигурации Коммутатора:**
```
На коммутаторе:
1. CLI → enable
2. show running-config
3. Проверить:
   - interface vlan 1
   - ip address [IP] [маска]
   - ip default-gateway [шлюз]
```

#### Часть 2: Систематическое тестирование

**Шаг 1: Заполнить таблицу адресации**

Создайте таблицу со всеми устройствами:
```
Device | Interface | IP Address    | Subnet Mask      | Default Gateway
-------|-----------|---------------|------------------|----------------
PC1    | NIC       | ?             | ?                | ?
PC2    | NIC       | ?             | ?                | ?
S1     | VLAN 1    | ?             | ?                | ?
S2     | VLAN 1    | ?             | ?                | ?
R1     | G0/0      | 192.168.10.1  | 255.255.255.0    | N/A
R1     | G0/1      | 192.168.11.1  | 255.255.255.0    | N/A
```

*Команды для сбора информации:*
```
На ПК: ipconfig /all
На Коммутаторе: show running-config
На Маршрутизаторе: show ip interface brief
```

**Шаг 2: Локальное тестирование связности**

Тестируйте устройства в ОДНОЙ сети:
```
От PC1 (192.168.10.100) к:
✓ ping 192.168.10.1   (gateway R1)
✓ ping 192.168.10.2   (S1 VLAN)

От PC2 к:
✓ ping 192.168.11.1   (gateway R1)
? ping 192.168.11.2   (S2 VLAN)
```

*ЕСЛИ ЛОКАЛЬНЫЕ ПИНГИ НЕ РАБОТАЮТ:*
- Проверить IP адрес устройства (ipconfig)
- Проверить маску подсети
- Проверить, что устройство подключено кабелем к правильной сети

**Шаг 3: Удаленное тестирование связности**

Тестируйте устройства в РАЗНЫХ сетях:
```
От PC1 (в 192.168.10.0) к:
? ping 192.168.11.1   (R1 G0/1)
? ping 192.168.11.2   (S2 VLAN)
? ping 192.168.11.100 (PC в другой сети)
```

*ЕСЛИ УДАЛЕННЫЕ ПИНГИ НЕ РАБОТАЮТ:*
- Проверить Default Gateway на ПК
- Проверить Default Gateway на коммутаторе
- Проверить, что маршрутизатор имеет маршруты в обе сети

#### Часть 3: Исправление проблем

**Проблема 1: ПК имеет неправильный IP адрес**

*Симптом:* PC1 не может пинговать другие устройства в своей же сети

*Решение:*
```
1. Кликните на PC1
2. Desktop → IP Configuration
3. Проверить IP Address соответствует таблице
4. Исправить если неправильно
```

**Проблема 2: ПК имеет неправильный Default Gateway**

*Симптом:* PC1 может пинговать локальные устройства, но не может пинговать удаленные сети

*Решение:*
```
1. Кликните на проблемный PC
2. Desktop → IP Configuration
3. Проверить Default Gateway
4. Должен быть IP адрес LAN интерфейса маршрутизатора
5. Исправить если неправильно
```

**Проблема 3: Коммутатор не имеет IP адреса**

*Симптом:* Другие устройства не могут пинговать коммутатор

*Решение:*
```
На коммутаторе:
S1# configure terminal
S1(config)# interface vlan 1
S1(config-if)# ip address 192.168.10.2 255.255.255.0
S1(config-if)# no shutdown
S1(config-if)# exit
S1(config)# exit
S1# copy running-config startup-config
```

**Проблема 4: Коммутатор не имеет Default Gateway**

*Симптом:* Удаленные ПК не могут пинговать коммутатор, но коммутатор может пинговать удаленные сети

*Решение:*
```
На коммутаторе:
S1(config)# ip default-gateway 192.168.10.1
```

*ВАЖНО:* На коммутаторе используется `ip default-gateway`, НЕ `ip route 0.0.0.0 0.0.0.0`!

#### Часть 4: Систематизация результатов

**План тестирования:**
```
Тест                           | Результат (✓/✗) | Причина если ✗      | Решение
-------------------------------|------------------|---------------------|-------------------
PC1 → PC2 (локально)           |                  |                     |
PC1 → R1 G0/0 (локально)       |                  |                     |
PC1 → S1 (локально)            |                  |                     |
PC1 → PC4 (удаленно)           |                  |                     |
PC1 → R1 G0/1 (удаленно)       |                  |                     |
PC3 → R2 G0/0 (локально)       |                  |                     |
PC3 → S2 (локально)            |                  |                     |
```

### Методология troubleshooting

```
1. СБОР ИНФОРМАЦИИ
   - Заполнить таблицу адресации
   - Собрать текущую конфигурацию каждого устройства

2. ЛОКАЛЬНОЕ ТЕСТИРОВАНИЕ
   - Пинговать default gateway в своей сети
   - Пинговать коммутатор в своей сети
   - Пинговать другие ПК в своей сети

3. УДАЛЕННОЕ ТЕСТИРОВАНИЕ
   - Пинговать default gateway в другой сети
   - Пинговать коммутатор в другой сети
   - Пинговать ПК в другой сети

4. АНАЛИЗ РЕЗУЛЬТАТОВ
   - Где впервые начинают падать пинги?
   - Это локальная проблема или удаленная?

5. ПРОВЕРКА КОНФИГУРАЦИИ
   - Проверить IP адреса на всех устройствах
   - Проверить Default Gateway на всех устройствах
   - Проверить, что интерфейсы активны

6. ИСПРАВЛЕНИЕ
   - Исправить одну проблему за раз
   - Протестировать после каждого исправления

7. ПРОВЕРКА
   - Повторить все тесты
   - Убедиться, что удаленная связность тоже работает
```

## 6. Packet Tracer - Investigating Convergence (7.1.3.6)

### Цель
Изучить процесс сходимости сети (network convergence) при добавлении новых маршрутов.

### Алгоритм выполнения

#### Часть 1: Изучение установившейся сети

**Шаг 1: Просмотр маршрутизационной таблицы R1**
```
R1# show ip route

Результат должен показать:
C 192.168.1.0/24    [directly connected]
R 192.168.2.0/24 [120/1] via 192.168.1.2, Serial0/0/0
R 192.168.3.0/24 [120/2] via 192.168.1.2, Serial0/0/0
L 192.168.1.1/32    [directly connected - local]
```

Расшифровка:
- **C** = Connected (прямо подключено)
- **R** = RIP (выучено через RIP протокол)
- **[120/1]** = AD (120 для RIP) / Metric (1 hop)
- **L** = Local (локальный адрес)

**Шаг 2: Проверка конфигурации RIP**
```
R1# show running-config | include router rip
```

*Примечание:* Сеть должна быть уже сконфигурирована с RIPv1 или RIPv2.

#### Часть 2: Добавление новой LAN

**Шаг 1: Подключение кабеля к новому интерфейсу**

```
1. Используйте Copper Straight-Through кабель
2. Подключите S1 к R1 GigabitEthernet 0/0
3. Ждите, пока индикатор станет зеленым (может занять 5-10 сек)
```

**Шаг 2: Тестирование ping перед конфигурацией**
```
На PC1:
ping PC2    → FAIL (R1 не знает маршрут к 64.0.0.0)
```

*Причина:* R1 не объявляет сеть 64.0.0.0/8 в RIP обновлениях.

#### Часть 3: Наблюдение за конвергенцией сети

**Шаг 1: Переключение в Simulation режим**
```
1. Нижний правый угол → Simulation
2. Это позволит нам видеть RIP обновления
```

**Шаг 2: Конфигурация новой сети на R1**
```
R1# configure terminal
R1(config)# router rip
R1(config-router)# network 64.0.0.0
R1(config-router)# exit
R1(config)# exit
```

**Шаг 3: Наблюдение за процессом сходимости**

```
В Simulation режиме кликните "Capture/Forward":
```

Вы увидите:
1. **RIPv1 обновление от R1** - содержит новую сеть 64.0.0.0
2. **Приход в R2** - R2 получает обновление
3. **Обновление маршрутизационной таблицы R2** - появляется маршрут к 64.0.0.0

**Шаг 4: Проверка отладочной информации на R2**

Если вы включили отладку:
```
R2# debug ip rip
R2# debug ip routing
```

Вывод будет похож на:
```
RIPv1: received packet with version 1
RIPv1 update from 192.168.1.2 on Serial0/0/0
64.0.0.0 1 hops
```

*Расшифровка:*
- R2 получил RIPv1 обновление от R1
- Сеть 64.0.0.0 находится на 1 hop расстоянии

**Шаг 5: Просмотр обновленной таблицы маршрутизации R2**
```
R2# show ip route

Новая запись должна появиться:
R 64.0.0.0/8 [120/1] via 192.168.1.2, 00:00:23, Serial0/0/0
```

*Теория:* 
- [120/1] означает: AD=120 (RIP), Metric=1 hop
- 00:00:23 - время до следующего обновления RIP

**Шаг 6: Тестирование ping после конвергенции**
```
На PC1:
ping PC2    → SUCCESS (теперь R1 и R2 знают о маршруте)
ping PC3    → SUCCESS (R3 тоже обновилась)
```

#### Часть 4: Отключение отладки

**После завершения:**
```
R2# undebug all
или
R2# no debug all
```

### Процесс сходимости (Convergence Process)

```
Шаг 1: R1 добавляет сеть 64.0.0.0 в RIP конфигурацию
       ↓
Шаг 2: R1 включает маршрут в свою RIP базу (находится в локальной БД)
       ↓
Шаг 3: R1 отправляет RIPv1 обновление через Serial0/0/0 (к R2)
       ↓
Шаг 4: R2 получает обновление и добавляет маршрут в свою таблицу
       ↓
Шаг 5: R2 отправляет обновление дальше (к R3)
       ↓
Шаг 6: R3 получает обновление
       ↓
Шаг 7: ВСА сеть конвергирована (все маршрутизаторы знают о новой сети)
```

**Время конвергенции:**
- RIPv1/RIPv2: 90 секунд по умолчанию (может варьироваться)
- OSPF: 10-30 секунд
- EIGRP: 5-10 секунд

### Важные команды

```
Команда                        | Назначение
-------------------------------|------------------------------------------
show ip route                  | Просмотр всех маршрутов
show ip route rip              | Просмотр только RIP маршрутов
show ip route connected        | Просмотр только прямо подключенных
debug ip rip                   | Включить RIP отладку
debug ip routing               | Включить отладку маршрутизации
no debug all                   | Отключить все отладки
show running-config | include network | Просмотр объявленных в RIP сетей
```

## 7. Packet Tracer - Configuring RIPv2 (7.3.1.8)

### Цель
Настроить динамический протокол маршрутизации RIPv2 для автоматического распределения информации о маршрутах.

### Алгоритм выполнения

#### Часть 1: Конфигурация маршрутов по умолчанию

**Шаг 1: Создание маршрута по умолчанию на граничном маршрутизаторе (R1)**

```
R1# configure terminal
R1(config)# ip route 0.0.0.0 0.0.0.0 Serial0/0/1
```

*Назначение:* Этот маршрут позволит всему трафику из локальной сети идти в Интернет.

#### Часть 2: Конфигурация RIPv2

**Основные шаги для КАЖДОГО маршрутизатора:**

1. Вход в режим RIP конфигурации
2. Переключение на версию 2
3. Отключение автоматического суммирования
4. Добавление прямо подключенных сетей
5. Установка passive интерфейсов
6. Объявление маршрута по умолчанию (только R1)

**Шаг 1: Конфигурация R1**

```
R1# configure terminal

# Создание маршрута по умолчанию (уже сделано)

# Вход в RIP конфигурацию
R1(config)# router rip

# Переключение на версию 2
R1(config-router)# version 2

# Отключение автоматического суммирования
R1(config-router)# no auto-summary
```

*Теория:* 
- **version 2** - более современная версия RIPv1, поддерживает VLSM
- **no auto-summary** - показывает все маршруты без суммирования в классы

```
# Добавление сетей
R1(config-router)# network 192.168.1.0
R1(config-router)# network 192.168.2.0
```

*Примечание:* Используется КЛАССОВАЯ адресация (первые три октета), не CIDR!

```
# Установка passive интерфейса (не отправлять маршруты на ПК)
R1(config-router)# passive-interface GigabitEthernet 0/0

# Объявление маршрута по умолчанию другим маршрутизаторам
R1(config-router)# default-information originate

R1(config-router)# exit
R1(config)# exit
```

*Теория:*
- **passive-interface** - интерфейс НЕ отправляет маршруты, но получает их
- **default-information originate** - объявляет маршрут по умолчанию в RIP обновлениях

**Шаг 2: Конфигурация R2**

```
R2# configure terminal
R2(config)# router rip
R2(config-router)# version 2
R2(config-router)# no auto-summary

# R2 имеет три прямо подключенные сети
R2(config-router)# network 192.168.2.0
R2(config-router)# network 192.168.3.0
R2(config-router)# network 192.168.4.0

# Passive интерфейс (LAN, где нет маршрутизаторов)
R2(config-router)# passive-interface GigabitEthernet 0/0

R2(config-router)# exit
R2(config)# exit
```

**Шаг 3: Конфигурация R3**

```
R3# configure terminal
R3(config)# router rip
R3(config-router)# version 2
R3(config-router)# no auto-summary

# R3 имеет две прямо подключенные сети
R3(config-router)# network 192.168.4.0
R3(config-router)# network 192.168.5.0

# Passive интерфейс
R3(config-router)# passive-interface GigabitEthernet 0/0

R3(config-router)# exit
R3(config)# exit

# Сохраняем
R3# copy running-config startup-config
```

#### Часть 3: Проверка конфигурации

**Шаг 1: Просмотр маршрутизационных таблиц**

```
R1# show ip route

Результат должен быть:
C    192.168.1.0/24 is directly connected, GigabitEthernet0/0
C    192.168.2.0/24 is directly connected, Serial0/0/0
R    192.168.3.0/24 [120/1] via 192.168.2.1, 00:00:05, Serial0/0/0
R    192.168.4.0/24 [120/2] via 192.168.2.1, 00:00:05, Serial0/0/0
R    192.168.5.0/24 [120/3] via 192.168.2.1, 00:00:05, Serial0/0/0
S*   0.0.0.0/0 is directly connected, Serial0/0/1
```

Расшифровка:
- **C** = Connected (192.168.1.0 и 192.168.2.0 - прямо подключены)
- **R** = RIP (остальные выучены через RIP с metric=1,2,3 hops)
- **[120/1]** = [AD / Metric]
- **S\*** = Static default route

**Шаг 2: Проверка RIP информации**

```
R1# show ip protocols

Результат должен содержать:
Routing Protocol is "rip"
Sending updates every 30 seconds with +/-180 second holddown
Outgoing update filter list for all interfaces is not set
Incoming update filter list for all interfaces is not set
Default version control on all interfaces
Send version 2, expect version 2
Passive Interface(s): GigabitEthernet0/0
Redistributing: rip default information
Default route version control on all interfaces
Maximum path splits to 4
Minimum splits for equal-cost multipath 0 (currently 0 active)
RIP for Networks:
  192.168.1.0
  192.168.2.0
```

**Шаг 3: Просмотр соседей RIP**

```
R1# show ip rip database

Результат:
192.168.1.0/24    directly connected, GigabitEthernet0/0
192.168.2.0/24    directly connected, Serial0/0/0

192.168.5.0/24    0.0.0.0, Serial0/0/0, 3, 00:00:07
0.0.0.0/0         redistributed, R2, 1, 00:00:01
```

#### Часть 4: Тестирование связности

**Шаг 1: Проверка ping между всеми ПК**

```
На PC1:
ping 192.168.2.1    (R1 Serial, должен пройти)
ping 192.168.2.254  (R2 LAN, должен пройти)
```

## 8. Packet Tracer - Configuring OSPFv2 in a Single Area (8.2.2.7)

### Цель
Настроить динамический протокол маршрутизации OSPFv2 для более быстрой сходимости и лучшей масштабируемости.

### Алгоритм выполнения

#### Часть 1: Основные концепции OSPF

**OSPF vs RIP:**
```
Параметр                 | RIP            | OSPF
-----------------------|----------------|----------------------
Metric                 | Hop count (max 15) | Cost (based on bandwidth)
Convergence            | Slow (90+ sec) | Fast (10-30 sec)
Scalability            | Poor (max 15)  | Excellent (>1000)
Bandwidth usage        | High           | Low
Administrative Distance| 120            | 110
Complexity             | Simple         | Complex
Area support           | No             | Yes (single area or multi-area)
```

**OSPF Parameter:**
- **Process ID:** Локальный идентификатор (не нужно совпадать между маршрутизаторами)
- **Router ID:** Уникальный идентификатор маршрутизатора (формат: x.x.x.x)
- **Area:** Область (для single area обычно используется область 0)
- **Cost/Metric:** Основан на пропускной способности интерфейса

#### Часть 2: Конфигурация OSPF

**Основные шаги для КАЖДОГО маршрутизатора:**

1. Вход в режим OSPF конфигурации
2. Установка Router ID
3. Добавление сетей с использованием wildcard маски
4. Установка passive интерфейсов
5. Сохранение конфигурации

**Шаг 1: Конфигурация R1**

```
R1# configure terminal

# Вход в OSPF конфигурацию (Process ID = 10)
R1(config)# router ospf 10

# Установка Router ID
R1(config-router)# router-id 1.1.1.1

# Объявление сетей с wildcard маской
# Формат: network [IP адрес] [wildcard маска] area [area number]

# LAN сеть (172.16.1.0/24)
# Wildcard маска: 255.255.255.0 → 0.0.0.255
R1(config-router)# network 172.16.1.0 0.0.0.255 area 0

# WAN сеть к R2 (172.16.3.0/30)
# Wildcard маска: 255.255.255.252 → 0.0.0.3
R1(config-router)# network 172.16.3.0 0.0.0.3 area 0

# WAN сеть к R3 (192.168.10.4/30)
R1(config-router)# network 192.168.10.4 0.0.0.3 area 0

# Установка passive интерфейса (LAN)
R1(config-router)# passive-interface GigabitEthernet0/0

R1(config-router)# exit
R1(config)# exit
```

*Теория:*
- **Wildcard маска** - инвертированная маска подсети
  - Subnet 255.255.255.0 → Wildcard 0.0.0.255
  - Subnet 255.255.255.252 → Wildcard 0.0.0.3
- **Area 0** - backbone area (основная область)

**Шаг 2: Конфигурация R2**

```
R2# configure terminal
```

#### Часть 3: Проверка конфигурации

**Шаг 1: Просмотр маршрутизационной таблицы**

```
R1# show ip route

Результат должен быть:
O    172.16.2.0/24 [110/2] via 172.16.3.2, 00:00:05, Serial0/0/0
O    192.168.1.0/24 [110/11] via 192.168.10.9, 00:00:05, Serial0/0/1
C    172.16.1.0/24 is directly connected, GigabitEthernet0/0
C    172.16.3.0/30 is directly connected, Serial0/0/0
C    192.168.10.4/30 is directly connected, Serial0/0/1
L    172.16.1.1/32 is directly connected, GigabitEthernet0/0
L    172.16.3.1/32 is directly connected, Serial0/0/0
L    192.168.10.5/32 is directly connected, Serial0/0/1
```

Расшифровка:
- **O** = OSPF (выучено через OSPF)
- **[110/2]** = [AD / Cost] - 110 это AD для OSPF
- Cost основан на пропускной способности

**Шаг 2: Проверка OSPF соседства**

```
R1# show ip ospf neighbor

Результат:

3.3.3.3         1   FULL/DR      00:00:35 192.168.10.9    Serial0/0/1
```

*Теория:*
- **FULL** = маршрутизаторы обменялись информацией и синхронизированы
- **BDR** = Backup Designated Router
- **DR** = Designated Router (главный)

### Командная справка маршрутизации

```
СТАТИЧЕСКИЕ МАРШРУТЫ:
ip route 172.31.0.0 255.255.255.0 172.31.1.193       (Recursive)

ip route 0.0.0.0 0.0.0.0 Serial0/0/0                  (Default route)
ip route 0.0.0.0 0.0.0.0 Serial0/0/0 5                (Floating, AD=5)

RIPv2 КОНФИГУРАЦИЯ:
router rip
version 2
no auto-summary
network 192.168.1.0
network 192.168.2.0
passive-interface GigabitEthernet0/0
default-information originate

OSPFv2 КОНФИГУРАЦИЯ:
router ospf 10
router-id 1.1.1.1
network 172.16.1.0 0.0.0.255 area 0
passive-interface GigabitEthernet0/0

ПРОВЕРКА:
show ip route                       (Таблица маршрутизации)
show ip route static                (Только статические маршруты)
show ip route connected             (Только прямо подключенные)
show ip protocols                   (Информация о протоколе маршрутизации)
show running-config | include route (Все маршруты в конфигурации)
```

### Типичные ошибки на экзамене

```
❌ ОШИБКА: Забывают "no shutdown" на интерфейсе
   ✓ ПРАВИЛЬНО: Всегда добавляйте "no shutdown"

❌ ОШИБКА: Используют IP адрес вместо wildcard маски в OSPF
   ✓ ПРАВИЛЬНО: network 172.16.1.0 [0.0.0.255] area 0

❌ ОШИБКА: Забывают "passive-interface" для LAN интерфейсов
   ✓ ПРАВИЛЬНО: passive-interface GigabitEthernet0/0

❌ ОШИБКА: Не сохраняют конфигурацию
   ✓ ПРАВИЛЬНО: copy running-config startup-config

❌ ОШИБКА: Путают AD и Metric
   ✓ ПРАВИЛЬНО: [110/2] = [AD (110 для OSPF) / Metric (2 cost)]

❌ ОШИБКА: Забывают "version 2" для RIP
   ✓ ПРАВИЛЬНО: version 2 (нужна для VLSM поддержки)

❌ ОШИБКА: Используют маску подсети вместо wildcard для OSPF
```
