OSPF (Open Shortest Path First) — это открытый стандартный протокол маршрутизации, относящийся к категории link-state (состояние канала), который обеспечивает динамическое заполнение таблиц маршрутизации на основе полной топологической карты сети, рассчитываемой каждым маршрутизатором независимо с помощью алгоритма Dijkstra, и поддерживает иерархическую организацию сети через разделение на зоны (области). OSPF широко используется в мультивендорных корпоративных сетях и провайдерских сетях благодаря открытому стандарту и быстрой сходимости при изменении топологии.[wikipedia](https://ru.wikipedia.org/wiki/OSPF)​Lektsiia-8-11.-Marshrutizatsiia.-Protokoly-dinamicheskoi-marshrutizatsii.pdf​

## Основные концепции OSPF

OSPF отличается от distance-vector протоколов (как RIP) тем, что каждый маршрутизатор строит полную топологическую карту сети и самостоятельно вычисляет оптимальные маршруты, а не полагается на информацию от соседей.[xgu+1](http://xgu.ru/wiki/OSPF)​

**Ключевые понятия:**Lektsiia-8-11.-Marshrutizatsiia.-Protokoly-dinamicheskoi-marshrutizatsii.pdf​[xgu](http://xgu.ru/wiki/OSPF)​

- **LSA (Link-State Advertisements)** — объявления о состоянии каналов и соседях; основная единица информации, распространяемая по сети.
    
- **LSDB (Link-State Database)** — синхронизированная база данных всех LSA, содержащих полную информацию о топологии сети; одна LSDB для каждой зоны.
    
- **SPF (Shortest Path First)** — алгоритм Dijkstra, который каждый маршрутизатор использует для независимого расчёта дерева кратчайших путей от себя ко всем целевым сетям.
    
- **Area (зона)** — логическое разделение сети для снижения объёма обмена служебной информацией и нагрузки на процессор маршрутизаторов.
    

## Зоны (Areas) в OSPF

OSPF обязательно использует иерархию зон для масштабируемости и эффективности.[xgu](http://xgu.ru/wiki/OSPF)​Lektsiia-8-11.-Marshrutizatsiia.-Protokoly-dinamicheskoi-marshrutizatsii.pdf​

**Правила зон:**[wikipedia+1](https://ru.wikipedia.org/wiki/OSPF)​

- **Area 0 (Backbone)** — обязательная магистральная (костяковая) зона.
    
- Все остальные зоны должны быть соединены с Area 0 через маршрутизаторы на границе зон (ABR).
    
- Нельзя переходить из зоны 1 в зону 2 напрямую; обязательно через Area 0.
    
- Исключение: Virtual Link (виртуальная линия) может обойти это правило в специфических сценариях.
    

**Конфигурация включения OSPF в зоне:**Lektsiia-8-11.-Marshrutizatsiia.-Protokoly-dinamicheskoi-marshrutizatsii.pdf​

text

`router ospf <процесс>  network <адрес> <wildcard_маска> area <номер_зоны>`

Пример:

text

`router ospf 1  network 192.168.1.0 0.0.0.255 area 0   # зона 0 network 192.168.2.0 0.0.0.255 area 1   # зона 1`

## Типы LSA (Link-State Advertisements)

LSA — это пакеты, содержащие информацию о состоянии сети.[telecom-sales+1](https://blog.telecom-sales.ru/ospf-tipy-lsa-i-ih-opisanie/)​

|Тип|Описание|Генерирует|Диапазон|
|---|---|---|---|
|**Type 1**|Router LSA — описывает маршрутизатор и его связи с соседями|Каждый маршрутизатор|Внутри одной зоны|
|**Type 2**|Network LSA — информация о сегменте множественного доступа (например, Ethernet с несколькими маршрутизаторами); генерирует DR|DR маршрутизатор|Внутри одной зоны|
|**Type 3**|Summary LSA — суммаризация маршрутов зоны для других зон|ABR|От одной зоны в другую|
|**Type 4**|ASBR Summary LSA — объявление о присутствии ASBR в других зонах|ABR|От одной зоны в другую|
|**Type 5**|External LSA — маршруты, перераспределённые из других протоколов маршрутизации (например, из BGP или RIP)|ASBR|По всей AS|

## LSDB (Link-State Database)

LSDB — единая синхронизированная база всех известных LSA в пределах зоны.[xgu+1](http://xgu.ru/wiki/OSPF)​

Все маршрутизаторы в одной зоне имеют **идентичные копии LSDB** для этой зоны, что гарантирует согласованность расчётов маршрутов. После синхронизации LSDB каждый маршрутизатор независимо запускает алгоритм SPF и вычисляет оптимальные маршруты.Lektsiia-8-11.-Marshrutizatsiia.-Protokoly-dinamicheskoi-marshrutizatsii.pdf​[xgu](http://xgu.ru/wiki/OSPF)​

**Просмотр LSDB на маршрутизаторе:**Lektsiia-8-11.-Marshrutizatsiia.-Protokoly-dinamicheskoi-marshrutizatsii.pdf​

text

`Router#show ip ospf database              # общий обзор Router#show ip ospf database router       # Type 1 Router#show ip ospf database network      # Type 2 Router#show ip ospf database summary      # Type 3`

## Типы пакетов OSPF

OSPF использует пять основных типов пакетов для обмена информацией и синхронизации:[selectel+1](https://selectel.ru/blog/ospf/)​

## 1. Hello

Пакеты для обнаружения соседей и мониторинга их доступности.[xgu](http://xgu.ru/wiki/OSPF)​Lektsiia-8-11.-Marshrutizatsiia.-Protokoly-dinamicheskoi-marshrutizatsii.pdf​

- Отправляются каждые **10 секунд** по мультикасту (224.0.0.5 в Area 0, 224.0.0.6 для ограниченной части).
    
- Содержат список известных соседей.
    
- Если Hello не приходит в течение **40 секунд** (по умолчанию Dead interval = 4 × Hello interval), соседство разрывается.Lektsiia-8-11.-Marshrutizatsiia.-Protokoly-dinamicheskoi-marshrutizatsii.pdf​
    

## 2. DBD (Database Description)

Краткое описание содержимого LSDB.[selectel+1](https://selectel.ru/blog/ospf/)​

- Маршрутизаторы обмениваются заголовками LSA для определения различий в базах.
    
- Помогает ускорить синхронизацию, так как вместо полных LSA передаются только идентификаторы.
    
- Содержит последовательные номера (sequence numbers) для определения актуальности информации.
    

## 3. LSR (Link-State Request)

Запрос недостающей информации после анализа DBD.[selectel+1](https://selectel.ru/blog/ospf/)​

- После обмена DBD один маршрутизатор запрашивает у соседа полную информацию об LSA, которых у него нет в LSDB.
    
- Содержит тип запрашиваемого LSA и идентификатор маршрутизатора, который его генерировал.
    

## 4. LSU (Link-State Update)

Передача полной информации об LSA.[selectel+1](https://selectel.ru/blog/ospf/)​

- Ответ на LSR с полным описанием запрошенных LSA.
    
- Может содержать несколько LSA в одном пакете.
    
- При получении LSU маршрутизатор обновляет свою LSDB и отправляет подтверждение (LSAck).
    

## 5. LSAck (Link-State Acknowledgment)

Подтверждение получения других пакетов (особенно LSU).[xgu+1](http://xgu.ru/wiki/OSPF)​

- Гарантирует надёжную доставку информации.
    
- Поле данных пусто; содержит только заголовок для подтверждения.
    

## Процесс установления соседства (Neighbor Adjacency)

OSPF использует несколько состояний при установлении соседства между маршрутизаторами:[icebale.readthedocs](https://icebale.readthedocs.io/en/latest/networks/protocols-tech/OSPF/)​Lektsiia-8-11.-Marshrutizatsiia.-Protokoly-dinamicheskoi-marshrutizatsii.pdf​

1. **Down** — сосед недоступен или соседство ещё не инициализировано.
    
2. **Attempt** — попытка установить соединение (обычно в NBMA сетях).
    
3. **Init** — получены Hello-пакеты, но двусторонняя связь не установлена.
    
4. **2-Way** — маршрутизаторы получили друг друга в своих Hello-пакетах; двусторонняя связь подтверждена. На этом этапе выбираются DR и BDR.
    
5. **ExStart** — подготовка к обмену описаниями LSDB; определение ведущего и ведомого маршрутизаторов.
    
6. **Exchange** — обмен DBD-пакетами с описанием LSDB.
    
7. **Loading** — запрос недостающих LSA через LSR/LSU.
    
8. **Full** — соседство полностью установлено; LSDB синхронизирована; маршруты вычислены и добавлены в таблицу маршрутизации.
    

На практике маршрутизаторы быстро проходят через эти состояния (обычно за секунды), поэтому в нормальной работающей сети видны в основном состояния **2-Way** (для DROTHER) и **Full** (для DR/BDR).

## Роли маршрутизаторов в OSPF

## Internal Router (IR)

Маршрутизатор, все интерфейсы которого находятся в одной зоне.[nag+1](https://nag.wiki/pages/viewpage.action?pageId=25107723)​

- Имеет одну LSDB для этой зоны.
    
- Обменивается информацией только с соседями в своей зоне.
    
- Не выполняет функции шлюза между зонами.
    

## Area Border Router (ABR)

Маршрутизатор на границе между зонами, соединяющий обычные зоны с магистральной зоной (Area 0).[nag+1](https://nag.wiki/pages/viewpage.action?pageId=25107723)​

- Имеет интерфейсы в двух и более зонах, из которых **обязательно одна — Area 0**.
    
- Хранит отдельную LSDB для каждой зоны, в которой участвует.
    
- Генерирует **Type 3 (Summary) LSA** для анонса маршрутов из одной зоны в другую.
    
- Может суммаризировать маршруты для снижения размера объявлений (например, анонсировать 10.0.0.0/8 вместо каждой /24 подсети).
    
- Является маршрутизатором магистрали (Backbone Router) по определению.
    

## Autonomous System Boundary Router (ASBR)

Маршрутизатор на границе автономной системы OSPF и других систем маршрутизации или внешних сетей.[habr+1](https://habr.com/ru/companies/otus/articles/921336/)​

- Обменивается маршрутной информацией с маршрутизаторами других AS (например, через BGP).
    
- Может перераспределять маршруты из других протоколов (RIP, EIGRP, BGP, статические маршруты) в OSPF.
    
- Генерирует **Type 5 (External) LSA** для объявления внешних маршрутов.
    
- Может находиться в любом месте AS (в обычной зоне или в зоне 0); не обязательно на границе зон.
    

## DR (Designated Router) и BDR (Backup Designated Router)

На сегментах множественного доступа (например, Ethernet с несколькими маршрутизаторами):[xgu](http://xgu.ru/wiki/OSPF)​Lektsiia-8-11.-Marshrutizatsiia.-Protokoly-dinamicheskoi-marshrutizatsii.pdf​

- **DR** — основной маршрутизатор, который собирает LSA от всех остальных маршрутизаторов на сегменте и рассылает их дальше. Генерирует **Type 2 LSA**.
    
- **BDR** — резервный маршрутизатор; берёт на себя роль DR при его отказе.
    
- **DROTHER** — остальные маршрутизаторы на сегменте; отправляют LSA только DR и BDR, но не друг другу напрямую.
    

**Выбор DR/BDR:**[xgu](http://xgu.ru/wiki/OSPF)​

1. Приоритет интерфейса OSPF (по умолчанию 1); выше приоритет — выбирают DR.
    
2. Если приоритеты равны, выбирают маршрутизатор с наибольшим Router ID (обычно наибольший Loopback адрес).
    

## Метрика OSPF (Cost)

Метрика OSPF называется **Cost** (стоимость) и зависит от пропускной способности интерфейса.[twistedminds+1](http://twistedminds.ru/2012/09/ccnp-preparation-ospf-2/)​

**Формула расчёта Cost:**[twistedminds+1](http://twistedminds.ru/2012/09/ccnp-preparation-ospf-2/)​

text

`Cost = Reference Bandwidth / Interface Bandwidth`

**Стандартные значения:**[twistedminds](http://twistedminds.ru/2012/09/ccnp-preparation-ospf-2/)​

- Reference Bandwidth: 100 Мбит/с (по умолчанию).
    
- Интерфейс 100 Мбит/с (FastEthernet) → Cost = 100/100 = **1**.
    
- Интерфейс 1 Гбит/с (GigabitEthernet) → Cost = 100/1000 = **1** (округление в большую сторону; минимум 1).
    
- Интерфейс 10 Мбит/с → Cost = 100/10 = **10**.
    
- Интерфейс 64 кбит/с → Cost = 100/(0.064) ≈ **1562**.
    

**Проблема современных сетей:** с появлением интерфейсов 10 Гбит/с и 100 Гбит/с, cost всех современных высокоскоростных интерфейсов остаётся равным 1, и OSPF не может различить их по скорости.youtube​[twistedminds](http://twistedminds.ru/2012/09/ccnp-preparation-ospf-2/)​

**Решение — изменение Reference Bandwidth:**youtube​[twistedminds](http://twistedminds.ru/2012/09/ccnp-preparation-ospf-2/)​

text

`router ospf 1  auto-cost reference-bandwidth 10000    # 10 Гбит/с в Мбит/с`

Пример с reference-bandwidth = 10000:

- 10 Гбит/с → Cost = 10000/10000 = **1**.
    
- 1 Гбит/с → Cost = 10000/1000 = **10**.
    
- 100 Мбит/с → Cost = 10000/100 = **100**.
    

**Расчёт полной стоимости пути (Total Cost):**[xgu](http://xgu.ru/wiki/OSPF)​  
Суммируется Cost всех интерфейсов на пути:

text

`Total Cost = Cost(Router0→Router1) + Cost(Router1→Router2) + ... + Cost(RouterN→Network)`

**Просмотр Cost интерфейсов:**[xgu](http://xgu.ru/wiki/OSPF)​

text

`Router#show ip ospf interface brief`

## Алгоритм SPF (Shortest Path First)

После синхронизации LSDB каждый маршрутизатор независимо запускает алгоритм Dijkstra (SPF):Lektsiia-8-11.-Marshrutizatsiia.-Protokoly-dinamicheskoi-marshrutizatsii.pdf​[xgu](http://xgu.ru/wiki/OSPF)​

1. Построение дерева кратчайших путей от данного маршрутизатора ко всем целевым сетям.
    
2. Вычисление Total Cost для каждого пути.
    
3. Выбор маршрутов с наименьшей стоимостью.
    
4. Добавление лучших маршрутов в таблицу маршрутизации (RIB).
    

Маршруты, добавленные в таблицу маршрутизации, отмечаются:Lektsiia-8-11.-Marshrutizatsiia.-Protokoly-dinamicheskoi-marshrutizatsii.pdf​

- **O** — маршрут OSPF в пределах одной зоны (intra-area).
    
- **O IA** — маршрут между зонами (inter-area), генерируется ABR.
    
- **O E1 / O E2** — внешние маршруты (от ASBR), перераспределённые из других AS.
    

**Пример записи в таблице маршрутизации:**

text

`O   192.168.1.0/24 [110/2] via 192.168.1.2, FastEthernet0/0 O IA 192.168.2.0/24 [110/3] via 192.168.0.1, FastEthernet0/1 O E2 10.0.0.0/8    [110/20] via 192.168.0.1, FastEthernet0/1 Где: 110 — административная дистанция OSPF 2, 3, 20 — Total Cost метрики`

## Таймеры OSPF

OSPF использует таймеры для мониторинга соседства и обновления информации:Lektsiia-8-11.-Marshrutizatsiia.-Protokoly-dinamicheskoi-marshrutizatsii.pdf​[xgu](http://xgu.ru/wiki/OSPF)​

|Таймер|Стандартное значение|Назначение|
|---|---|---|
|**Hello Interval**|10 сек (broadcast/p2p), 30 сек (NBMA)|Периодичность отправки Hello-пакетов|
|**Dead Interval**|40 сек (broadcast), 120 сек (NBMA)|Время ожидания Hello перед разрывом соседства (обычно 4 × Hello)|
|**Retransmit Interval**|5 сек|Время ожидания подтверждения LSU перед повторной отправкой|
|**LSA Age**|Макс 3600 сек|Возраст LSA; свежие объявления переиздаются каждые 30 мин|

## Преимущества и недостатки OSPF

|Характеристика|Статус|
|---|---|
|**Открытый стандарт (RFC 2328)**|✅ Работает на оборудовании любого вендора|
|**Быстрая сходимость**|✅ Пересчёт за несколько секунд при изменении топологии|
|**Полная топологическая карта (LSDB)**|✅ Избегает проблемы count-to-infinity (как в RIP)|
|**Иерархия зон**|✅ Снижает служебный трафик и нагрузку на процессор|
|**Гибкая метрика**|✅ Можно настроить Cost на основе пропускной способности|
|**Поддержка VLSM и CIDR**|✅ Работает с современной адресацией|
|**Балансировка нагрузки**|✅ Равноценная (до 4 путей по умолчанию)|
|**Требует больше памяти**|❌ LSDB занимает память для всей зоны|
|**Больше нагрузки на процессор**|❌ Алгоритм Dijkstra требует вычислений|
|**Сложность конфигурации**|❌ Требует правильного проектирования зон|
