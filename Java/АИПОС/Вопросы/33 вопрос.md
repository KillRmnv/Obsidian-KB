# Экзаменационный билет: TCP (Transmission Control Protocol) — Протокол управления передачей

## 1. Общая характеристика и назначение

**TCP (Transmission Control Protocol)** — это основной транспортный протокол Интернета (стандарт RFC 793), который обеспечивает **надежную**, **упорядоченную** и **потоковую** доставку данных между приложениями.Lektsiia-12.-Transportnyi-uroven.pdf​[timeweb](https://timeweb.cloud/blog/transportnyj-uroven-osi)​

**Главная задача TCP:** превратить ненадежную IP-сеть (где пакеты могут теряться и путаться) в надежный виртуальный канал связи. Приложение (например, браузер) пишет данные в сокет как в файл, а TCP берет на себя всю сложную работу по их нарезке, доставке и проверке.



## 2. Структура заголовка TCP

Заголовок TCP значительно сложнее UDP, его минимальный размер составляет **20 байт**, а максимальный — **60 байт** (с опциями).

## 2.1. Таблица полей заголовка

|Поле|Размер|Описание и функции|
|---|---|---|
|**Source Port**|16 бит|Порт отправителя.|
|**Destination Port**|16 бит|Порт получателя.|
|**Sequence Number (SEQ)**|32 бита|**Порядковый номер.** Если отправляется файл, каждый байт нумеруется. Поле указывает номер _первого байта_ данных в этом сегменте. Позволяет получателю собрать данные в правильном порядке.|
|**Acknowledgment Number (ACK)**|32 бита|**Номер подтверждения.** Содержит номер _следующего_ байта, который отправитель ожидает получить (SEQ + длина полученных данных). Работает по принципу: «Я получил всё до байта X, жду байт X+1».|
|**Data Offset**|4 бита|Длина заголовка в 32-битных словах. Минимальное значение — 5 (5 * 4 = 20 байт).|
|**Reserved**|3-6 бит|Зарезервировано (нули).|
|**Flags (Control Bits)**|6-9 бит|Управляющие биты (см. раздел 2.2).|
|**Window Size**|16 бит|**Размер окна.** Количество байт, которое получатель готов принять _прямо сейчас_ без подтверждения. Используется для управления потоком (Flow Control).|
|**Checksum**|16 бит|Контрольная сумма (заголовок + данные + псевдозаголовок).|
|**Urgent Pointer**|16 бит|Указатель на «срочные данные» (используется редко, только если флаг URG=1).|
|**Options**|0-40 байт|Дополнительные параметры: MSS (максимальный размер сегмента), Window Scale, Timestamps.|

## 2.2. Ключевые флаги (Flags)

Флаги управляют жизненным циклом соединения:

1. **SYN (Synchronize):** «Хочу начать соединение». Только первые два пакета имеют этот флаг.
    
2. **ACK (Acknowledgment):** «Подтверждаю получение». Установлен во всех пакетах после начала связи.
    
3. **FIN (Finish):** «Я закончил передачу данных, хочу закрыть соединение».
    
4. **RST (Reset):** «Ошибка! Сбрось соединение немедленно» (аварийный разрыв).
    
5. **PSH (Push):** «Передай данные приложению немедленно, не буферизуй».
    
6. **URG (Urgent):** Пакет содержит срочные данные (устаревший функционал).

## 3. Управление соединением

## 3.1. Установление соединения (Three-Way Handshake)

Перед передачей данных стороны должны «договориться». Этот процесс называется **«Тройным рукопожатием»**:

1. **SYN:** Клиент отправляет сегмент `[SYN]` со случайным начальным номером последовательности `Seq = X`. (Состояние клиента: _SYN_SENT_)
    
2. **SYN-ACK:** Сервер получает запрос, запоминает `Seq` клиента, генерирует свой `Seq = Y` и отправляет ответ `[SYN, ACK]`. Поле `Ack` равно `X+1`. (Состояние сервера: _SYN_RCVD_)
    
3. **ACK:** Клиент подтверждает получение, отправляя сегмент `[ACK]`. Поле `Seq = X+1`, `Ack = Y+1`. (Состояние: _ESTABLISHED_ — соединение установлено, можно слать данные).
    

## 3.2. Разрыв соединения

TCP — дуплексный протокол, поэтому соединение разрывается в каждую сторону отдельно (хотя часто это объединяется в 4 этапа):

1. **Клиент:** Шлет `[FIN]` («Я все сказал»). Переходит в _FIN_WAIT_1_.
    
2. **Сервер:** Шлет `[ACK]` («Понял, ты закончил»). Клиент ждет, пока сервер договорит.
    
3. **Сервер:** Когда сам закончил передачу, шлет свой `[FIN]`.
    
4. **Клиент:** Шлет последний `[ACK]` и ждет тайм-аут (состояние _TIME_WAIT_), чтобы убедиться, что сервер получил ответ.

## 4. Механизмы надежности и управления

TCP — это не просто доставка, это «умная» доставка. Три главных механизма делают его таким:

## 4.1. Надежность (Reliability & ARQ)

- **Тайм-ауты и повторы (RTO):** Если отправитель не получает `ACK` в течение определенного времени (RTO), он считает пакет потерянным и отправляет его снова.
    
- **Быстрая повторная передача (Fast Retransmit):** Если отправитель получает **3 дублирующих ACK** (получатель трижды говорит «Я жду пакет №5», хотя уже пришли №6, 7, 8), отправитель понимает, что №5 потерян, и пересылает его _до_ истечения таймера.[geeksforgeeks](https://www.geeksforgeeks.org/computer-networks/tcp-congestion-control-algorithms-reno-new-reno-bic-cubic/)​
    

## 4.2. Управление потоком (Flow Control)

Этот механизм защищает **получателя**.

- Если приложение-получатель (например, веб-сервер) медленно читает данные, его буфер приема переполняется.
    
- Получатель в поле **Window Size** каждого `ACK` сообщает: «У меня осталось места на 2000 байт».
    
- Отправитель видит это и снижает скорость. Если окно станет = 0, передача временно останавливается.
    

## 4.3. Управление перегрузкой (Congestion Control)

Этот механизм защищает **сеть** (роутеры провайдеров). Если все начнут слать данные на максимальной скорости, Интернет «встанет».

- **Congestion Window (cwnd):** Внутренняя переменная отправителя (не передается в заголовке), ограничивающая объем данных в полете.
    
- **Медленный старт (Slow Start):** Начинаем передачу осторожно (1 пакет), но при каждом успешном `ACK` удваиваем скорость (экспоненциальный рост).
    
- **Congestion Avoidance:** Как только достигли порога или заметили потерю, рост становится линейным (осторожным).
    
- **Реакция на потерю:** Потеря пакета = перегрузка сети. TCP _резко_ сбрасывает скорость (обычно в 2 раза или до 1 сегмента) и начинает расти снова.[icebale.readthedocs+1](https://icebale.readthedocs.io/en/latest/networks/protocols-tech/TCP/)​

## 5. Главные выводы для ответа

- **TCP** — это гарантия доставки. Он «тяжелее» UDP из-за сложного заголовка и процедур рукопожатия.
    
- **Потоковость:** Для программы это просто труба с данными (stream), границы сообщений не сохраняются.
    
- **Интеллект:** TCP сам решает, с какой скоростью слать данные, чтобы не «уронить» сеть и не «завалить» получателя.
    
- **Применение:** Веб-сайты (HTTP/HTTPS), почта (SMTP, IMAP), передача файлов (FTP), удаленный доступ (SSH) — везде, где **точность** важнее мгновенной скорости.
