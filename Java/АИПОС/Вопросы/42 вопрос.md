## **ЭКЗАМЕНАЦИОННЫЙ ОТВЕТ: Протокол DHCP (Dynamic Host Configuration Protocol)**

## **1. ОПРЕДЕЛЕНИЕ И НАЗНАЧЕНИЕ DHCP**

**DHCP (Dynamic Host Configuration Protocol)** — это сетевой протокол прикладного уровня, обеспечивающий предоставление IP-адресов и сведений о сети для хостов, которые хотят подключиться к сети. После получения данных параметров хост становится полноправным узлом сети и может обмениваться данными в сети.Lektsiia-13-14.-Prikladnoi-uroven.pdf​

## **Основные характеристики DHCP**

|Характеристика|Описание|
|---|---|
|**Уровень OSI**|Работает на сетевом (уровень 3) и канальном (уровень 2) уровнях[cqr](https://cqr.company/ru/wiki/protocols/dynamic-host-configuration-protocol-dhcp/)​|
|**Транспортный протокол**|UDP (User Datagram Protocol)[cqr](https://cqr.company/ru/wiki/protocols/dynamic-host-configuration-protocol-dhcp/)​|
|**Порт сервера**|Порт 67 (прием DHCP-сервером)[cqr](https://cqr.company/ru/wiki/protocols/dynamic-host-configuration-protocol-dhcp/)​|
|**Порт клиента**|Порт 68 (отправка DHCP-клиентом)[cqr](https://cqr.company/ru/wiki/protocols/dynamic-host-configuration-protocol-dhcp/)​|
|**Архитектура**|Клиент-сервернаяLektsiia-13-14.-Prikladnoi-uroven.pdf​|
|**Назначение**|Автоматическое назначение сетевых параметров хостам|

## **Информация, предоставляемая DHCP**

DHCP не только выдает IP-адрес, но и предоставляет полную конфигурацию сети:[smart-soft+1](https://www.smart-soft.ru/blog/printsipy_raboty_protokola_dhcp/)​

- **IP-адрес** — основной сетевой адрес устройства
    
- **Маска подсети (Subnet Mask)** — определяет размер подсети
    
- **Шлюз (Default Gateway)** — адрес маршрутизатора для выхода в другие сети
    
- **DNS-серверы** — адреса для разрешения доменных имен
    
- **Время аренды (Lease Time)** — период, на который выдается адрес
    
- **DHCP-сервер** — адрес сервера, выдавшего конфигурацию

## **2. АРХИТЕКТУРА DHCP**

## **Модель клиент-сервер**

DHCP использует классическую архитектуру клиент-сервер:

1. **DHCP-клиент** — устройство, которому нужна сетевая конфигурация (ноутбук, смартфон, сервер)
    
2. **DHCP-сервер** — устройство, которое предоставляет IP-адреса и сетевые параметры
    
    - Может быть маршрутизатором
        
    - Может быть коммутатором
        
    - Может быть выделенным сервером[selectel](https://selectel.ru/blog/dhcp-protocol/)​

## **Magic Cookie**

Первые 4 байта поля **options** содержат значения 99, 130, 83 и 99 (в ASCII: DHCP Magic Cookie). Эта последовательность отличает сообщения DHCP от сообщений BOOTP, которые имеют идентичный формат заголовка.[habr](https://habr.com/ru/articles/825016/)​

## **5. ПРОЦЕСС DORA (Discover-Offer-Request-Acknowledge)**

DHCP использует принцип **DORA** для выдачи IP-адресов:[smart-soft+1](https://www.smart-soft.ru/blog/printsipy_raboty_protokola_dhcp/)​Lektsiia-13-14.-Prikladnoi-uroven.pdf​

## **Этап 1: DISCOVERY (Обнаружение)**

**Состояние клиента:** INITLektsiia-13-14.-Prikladnoi-uroven.pdf​

1. **Инициирование** — клиент находится в состоянии INIT и не имеет IP-адреса[smart-soft](https://www.smart-soft.ru/blog/printsipy_raboty_protokola_dhcp/)​
    
2. **Отправка DHCPDISCOVER** — клиент передает широковещательное (broadcast) сообщение DHCPDISCOVER на всех в локальной сети[selectel](https://selectel.ru/blog/dhcp-protocol/)​Lektsiia-13-14.-Prikladnoi-uroven.pdf​
    
    - Адрес источника: 0.0.0.0 (у клиента нет адреса)
        
    - Адрес назначения: 255.255.255.255 (broadcast)
        
    - MAC-адрес клиента указан в поле **chaddr**
        
3. **Переход в состояние** SELECTINGLektsiia-13-14.-Prikladnoi-uroven.pdf​
    

**Диаграмма**:[selectel](https://selectel.ru/blog/dhcp-protocol/)​Lektsiia-13-14.-Prikladnoi-uroven.pdf​

```bash
Клиент (INIT)
    ↓
Отправляет DHCPDISCOVER (broadcast)
    ↓
Переход в состояние SELECTING

```

---

## **Этап 2: OFFER (Предложение)**

**Отправитель:** DHCP-серверLektsiia-13-14.-Prikladnoi-uroven.pdf​

1. **Прием DHCPDISCOVER** — каждый DHCP-сервер в локальной сети слышит широковещательное сообщение[selectel](https://selectel.ru/blog/dhcp-protocol/)​Lektsiia-13-14.-Prikladnoi-uroven.pdf​
    
2. **Проверка адреса** — сервер проверяет, что предлагаемый адрес не занят (используя ICMP-запрос)[habr](https://habr.com/ru/articles/825016/)​
    
3. **Отправка DHCPOFFER** — сервер отправляет ответное сообщение DHCPOFFER, содержащее:[selectel](https://selectel.ru/blog/dhcp-protocol/)​Lektsiia-13-14.-Prikladnoi-uroven.pdf​
    
    - Предлагаемый IP-адрес в поле **yiaddr**
        
    - Необходимые параметры сети в поле **options** (маска подсети, шлюз, DNS)
        
    - Срок аренды IP-адреса
        
    - Идентификатор сервера в опции **Server Identifier**
        

**Диаграмма**:[selectel](https://selectel.ru/blog/dhcp-protocol/)​Lektsiia-13-14.-Prikladnoi-uroven.pdf​

```bash
DHCP-сервер
    ↓
Получает DHCPDISCOVER
    ↓
Проверяет свободность адреса
    ↓
Отправляет DHCPOFFER (может быть broadcast или unicast)

```
**Важно:** Если присутствует несколько DHCP-серверов, клиент получает несколько DHCPOFFER.Lektsiia-13-14.-Prikladnoi-uroven.pdf​[selectel](https://selectel.ru/blog/dhcp-protocol/)​

## **Этап 3: REQUEST (Запрос)**

**Состояние клиента:** REQUESTINGLektsiia-13-14.-Prikladnoi-uroven.pdf​

1. **Выбор сервера** — клиент получил одно или несколько предложений и выбирает предпочитаемый сервер на основе предложенных параметровLektsiia-13-14.-Prikladnoi-uroven.pdf​
    
2. **Отправка DHCPREQUEST** — клиент отправляет широковещательное сообщение DHCPREQUEST, указывая:[selectel](https://selectel.ru/blog/dhcp-protocol/)​Lektsiia-13-14.-Prikladnoi-uroven.pdf​
    
    - IP-адрес выбранного сервера в опции **Server Identifier**
        
    - Желаемый IP-адрес в опции **requested ip** (копия **yiaddr** из DHCPOFFER)
        
    - Свой MAC-адрес
        
3. **Уведомление серверов** — таким образом, клиент:Lektsiia-13-14.-Prikladnoi-uroven.pdf​
    
    - Подтверждает выбранному серверу, что согласен с параметрами
        
    - Уведомляет остальных серверов об отказе от их предложений
        

**Диаграмма**:[selectel](https://selectel.ru/blog/dhcp-protocol/)​Lektsiia-13-14.-Prikladnoi-uroven.pdf​

```bash
Клиент (SELECTING)
    ↓
Выбирает предпочитаемый сервер
    ↓
Отправляет DHCPREQUEST (broadcast) с Server Identifier и requested ip
    ↓
Переход в состояние REQUESTING

```

## **Этап 4: ACKNOWLEDGE (Подтверждение)**

**Состояние клиента:** BOUNDLektsiia-13-14.-Prikladnoi-uroven.pdf​

1. **Получение DHCPREQUEST** — выбранный сервер получает сообщение DHCPREQUEST[habr](https://habr.com/ru/articles/825016/)​Lektsiia-13-14.-Prikladnoi-uroven.pdf​
    
2. **Проверка адреса** — сервер проверяет в своей базе данных, может ли предоставить запрошенный адрес[habr](https://habr.com/ru/articles/825016/)​
    
3. **Отправка DHCPACK** — если адрес доступен, сервер отправляет окончательное подтверждение DHCPACK, содержащее:[habr](https://habr.com/ru/articles/825016/)​Lektsiia-13-14.-Prikladnoi-uroven.pdf​
    
    - Выделенный IP-адрес в поле **yiaddr**
        
    - Все параметры сети (маска подсети, шлюз, DNS)
        
    - Срок аренды (Lease Time)
        
    - Сохраняет запись о выделении адреса в хранилище
        
4. **Отправка DHCPNACK (если адрес недоступен)** — если сервер не может предоставить адрес (например, он уже назначен другому клиенту или клиент оказался в другой подсети), отправляется DHCPNACK[habr](https://habr.com/ru/articles/825016/)​Lektsiia-13-14.-Prikladnoi-uroven.pdf​
    
5. **Применение конфигурации** — клиент получает DHCPACK, проверяет настройки и применяет конфигурацию сетиLektsiia-13-14.-Prikladnoi-uroven.pdf​
    
6. **Переход в состояние BOUND** — клиент переходит в состояние BOUND и может использовать выданный адресLektsiia-13-14.-Prikladnoi-uroven.pdf​
    

**Диаграмма**:[selectel](https://selectel.ru/blog/dhcp-protocol/)​Lektsiia-13-14.-Prikladnoi-uroven.pdf​

```bash
DHCP-сервер
    ↓
Получает DHCPREQUEST
    ↓
Проверяет возможность выдачи адреса
    ↓
Отправляет DHCPACK (или DHCPNACK)
    ↓
Клиент получает DHCPACK
    ↓
Применяет конфигурацию
    ↓
Состояние BOUND (использование адреса)

```

## **6. СОСТОЯНИЯ DHCP КЛИЕНТА**

Клиент DHCP проходит через несколько состояний:Lektsiia-13-14.-Prikladnoi-uroven.pdf​

|Состояние|Описание|
|---|---|
|**INIT**|Начальное состояние; клиент пытается получить IP-адрес|
|**SELECTING**|Клиент получил одно или несколько DHCPOFFER и выбирает сервер|
|**REQUESTING**|Клиент отправил DHCPREQUEST и ожидает подтверждения|
|**BOUND**|Клиент получил DHCPACK и использует выданный IP-адрес|
|**RENEWING**|Клиент пытается продлить срок аренды адреса (T1 время истекло)|
|**REBINDING**|Клиент пытается продлить адрес у других серверов (T2 время истекло)|

## **7. ВРЕМЯ ЖИЗНИ IP-АДРЕСА (Lease Time)**

DHCP выдает IP-адреса на определенный период — **время аренды (Lease Time)**:[timeweb+1](https://timeweb.cloud/blog/protokol-dhcp)​

|Событие|Описание|
|---|---|
|**Выдача адреса**|DHCP-сервер указывает время аренды (обычно от нескольких часов до нескольких дней)|
|**Т1 (обновление)**|На 50% времени аренды клиент отправляет DHCPREQUEST для продления адреса[selectel](https://selectel.ru/blog/dhcp-protocol/)​|
|**Т2 (перебиндинг)**|На 87.5% времени аренды, если сервер не ответил на Т1, клиент попытается получить адрес у других серверов[smart-soft](https://www.smart-soft.ru/blog/printsipy_raboty_protokola_dhcp/)​|
|**Окончание аренды**|Если адрес не продлен, он освобождается и может быть выдан другому клиенту[selectel](https://selectel.ru/blog/dhcp-protocol/)​|

**Диаграмма жизненного цикла адреса**:
```bash
Получен DHCPACK
    ↓ (50% времени)
T1: Попытка RENEW у того же сервера (DHCPREQUEST)
    ↓ (если нет ответа)
    ↓ (87.5% времени)
T2: REBIND у других серверов (DHCPREQUEST broadcast)
    ↓ (если нет ответа)
    ↓ (100% времени)
Адрес истекает → клиент возвращается в INIT[86]

```

## **8. СПОСОБЫ РАСПРЕДЕЛЕНИЯ IP-АДРЕСОВ**

DHCP поддерживает три способа выделения IP-адресов:[wikipedia+1](https://ru.wikipedia.org/wiki/DHCP)​Lektsiia-13-14.-Prikladnoi-uroven.pdf​

## **1. Ручное распределение (Manual Allocation)**

- Сетевой администратор сопоставляет **MAC-адрес** каждого клиента определенному **IP-адресу**[wikipedia](https://ru.wikipedia.org/wiki/DHCP)​
    
- Информация хранится централизованно на DHCP-сервере[wikipedia](https://ru.wikipedia.org/wiki/DHCP)​
    
- IP-адрес остается за клиентом, даже если он выйдет из сети или отключится[selectel](https://selectel.ru/blog/dhcp-protocol/)​
    
- Удобно для критичных устройств (серверы, принтеры)
    
- Требует ручной настройки администратором
    

## **2. Динамическое распределение (Dynamic Allocation)**

- DHCP-сервер назначает **любой свободный IP-адрес из диапазона**[timeweb+1](https://timeweb.cloud/blog/protokol-dhcp)​
    
- IP-адреса не привязываются к конкретным устройствам[timeweb](https://timeweb.cloud/blog/protokol-dhcp)​
    
- Адрес предоставляется на определенное время (срок аренды)[timeweb+1](https://timeweb.cloud/blog/protokol-dhcp)​
    
- При истечении срока адрес может быть выдан другому устройству
    
- Экономит адреса и автоматизирует управление
    
- **Наиболее часто используемый метод**
    

## **3. Автоматическое распределение (Automatic Allocation)**

- DHCP-сервер назначает IP-адреса, но без срока истечения[wikipedia](https://ru.wikipedia.org/wiki/DHCP)​
    
- IP-адреса остаются за хостами постоянно
    
- Промежуточный вариант между ручным и динамическим
    
- Используется реже

## **9. СПЕЦИАЛЬНЫЕ СООБЩЕНИЯ DHCP**

## **DHCPDECLINE (Отказ клиента)**

- Клиент отправляет это сообщение, если обнаружил, что выданный адрес уже используется в сети (проверка ARP)[wikipedia](https://ru.wikipedia.org/wiki/DHCP)​
    
- После этого процедура получения IP-адреса повторяется[wikipedia](https://ru.wikipedia.org/wiki/DHCP)​
    

## **DHCPRELEASE (Освобождение адреса)**

- Клиент явно освобождает IP-адресLektsiia-13-14.-Prikladnoi-uroven.pdf​
    
- Позволяет серверу раньше вернуть адрес в пул свободных адресов
    
- Происходит при корректном выключении устройства
    

## **DHCPINFORM (Запрос параметров)**

- Клиент, который уже имеет IP-адрес, может запросить только параметры сети (маску, шлюз, DNS)[selectel](https://selectel.ru/blog/dhcp-protocol/)​Lektsiia-13-14.-Prikladnoi-uroven.pdf​
    
- Используется для получения дополнительной информации без получения нового адреса

## **10. ПАРАМЕТРЫ И ОПЦИИ DHCP**

|**IP Address Lease Time**|51|Время аренды IP-адреса|
|**Parameter Request List**|55|Список запрашиваемых опций клиентом|

**Конец опций** обозначается байтом **0xff**.[habr](https://habr.com/ru/articles/825016/)​

## **11. БЕЗОПАСНОСТЬ DHCP**

## **Уязвимости DHCP**

1. **DHCP Starvation** — атака, при которой злоумышленник отправляет множество DHCPDISCOVER сообщений, истощая пул свободных адресов[arxiv](https://arxiv.org/pdf/2502.10646.pdf)​
    
2. **Rogue DHCP Server** — кто-то включает в сеть поддельный DHCP-сервер, выдающий вредоносную конфигурацию[arxiv](https://arxiv.org/pdf/2502.10646.pdf)​
    
3. **Replay Attacks** — перехват и повторное воспроизведение DHCP-сообщений[arxiv](https://arxiv.org/pdf/2502.10646.pdf)​
    
4. **MAC Spoofing** — подделка MAC-адреса для обхода ограничений[arxiv](https://arxiv.org/abs/1410.4398v1)​
    

## **Меры безопасности**

- **DHCP Snooping** — мониторинг DHCP-трафика на коммутаторе для предотвращения поддельных серверов[jurnal.kdi](https://jurnal.kdi.or.id/index.php/bt/article/download/1174/691)​
    
- **DAI (Dynamic ARP Inspection)** — проверка соответствия IP и MAC адресов[jurnal.kdi](https://jurnal.kdi.or.id/index.php/bt/article/download/1174/691)​
    
- **DHCPv6** — улучшенная версия DHCP для IPv6 с лучшей безопасностью[ccsenet](https://ccsenet.org/journal/index.php/mas/article/download/4310/3740)​

## **12. ПРИМЕНЕНИЕ DHCP**

**Где используется DHCP:**

- **Корпоративные сети** — автоматическое назначение IP-адресов сотрудникам
    
- **Домашние сети** — маршрутизаторы выдают IP-адреса устройствам в сети
    
- **Мобильные сети** — для выдачи адресов мобильным устройствам
    
- **Сети общественных мест** — WiFi в кафе, аэропортах (гостевые сети)
    
- **Облачные среды** — для управления адресацией на виртуальных машинах

## **Заключение**

**DHCP** — это критически важный протокол для автоматического управления IP-адресацией в современных сетях. Он использует четырехэтапный процесс **DORA** (Discover-Offer-Request-Acknowledge) для выдачи динамических IP-адресов.

Ключевые преимущества DHCP:

- **Автоматизация** — избегает ручной настройки каждого устройства
    
- **Гибкость** — поддерживает ручное, динамическое и автоматическое распределение адресов
    
- **Масштабируемость** — легко управлять адресами в больших сетях
    
- **Мобильность** — устройства могут подключаться к разным сетям и автоматически получать адреса
    

Однако DHCP требует внимания к вопросам безопасности, особенно в критичных сетях, где рекомендуется использовать DHCP Snooping и другие защитные механизмы.
