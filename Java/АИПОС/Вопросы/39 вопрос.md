## **1. ПРОТОКОЛ HTTP (HyperText Transfer Protocol)**

## **Определение и назначение**

HTTP (HyperText Transfer Protocol) — это семейство протоколов запроса/ответа прикладного уровня без учета состояния, которые используют общий интерфейс, расширяемую семантику и самоописательные сообщения для обеспечения гибкого взаимодействия с сетевыми гипертекстовыми информационными системами. Это основной протокол для передачи веб-страниц и данных в интернете.Lektsiia-13-14.-Prikladnoi-uroven.pdf​

## **Основные характеристики HTTP**

|Характеристика|Описание|
|---|---|
|**Уровень сети**|Протокол прикладного уровняLektsiia-13-14.-Prikladnoi-uroven.pdf​|
|**Порт**|Порт 80Lektsiia-13-14.-Prikladnoi-uroven.pdf​|
|**Модель взаимодействия**|Клиент-серверная архитектураLektsiia-13-14.-Prikladnoi-uroven.pdf​|
|**Транспортный протокол**|TCP (в HTTP/3 используется QUIC)Lektsiia-13-14.-Prikladnoi-uroven.pdf​|
|**Состояние**|Протокол без состояния (stateless) — каждый запрос независимLektsiia-13-14.-Prikladnoi-uroven.pdf​|
|**Безопасность**|Данные передаются в открытом виде без шифрования|

## **Принцип работы HTTP**

Клиент-серверное взаимодействие происходит следующим образом:Lektsiia-13-14.-Prikladnoi-uroven.pdf​

1. Браузер (HTTP-клиент) инициализирует TCP-соединение с сервером (создает сокет)
    
2. Сервер принимает TCP-соединение от клиента
    
3. Браузер и веб-сервер обмениваются сообщениями прикладного уровня
    
4. TCP-соединение закрывается

## **2. ПРОТОКОЛ HTTPS (HyperText Transfer Protocol Secure)**

## **Определение**

HTTPS (HyperText Transfer Protocol Secure) — это расширение протокола HTTP, объединяющее HTTP и криптографические протоколы SSL/TLS. HTTPS обеспечивает защищенную передачу данных путем шифрования информации между браузером и сервером.[habr](https://habr.com/ru/articles/959544/)​Lektsiia-13-14.-Prikladnoi-uroven.pdf​

## **Как работает HTTPS**

HTTPS использует следующий механизм защиты:[webtronics+1](https://webtronics.ru/blog/chem-otlichayutsya-protokoly-http-i-https/)​

1. **Процесс рукопожатия (TLS Handshake)** — проверяется подлинность сервера и согласовываются параметры шифрования[habr](https://habr.com/ru/articles/959544/)​
    
2. **Обмен ключами** — клиент получает публичный ключ сервера и генерирует секретный ключ[webtronics](https://webtronics.ru/blog/chem-otlichayutsya-protokoly-http-i-https/)​
    
3. **Шифрование данных** — все данные, включая URL, HTTP-заголовки и тело запроса/ответа, шифруются с использованием симметричного ключа[webtronics](https://webtronics.ru/blog/chem-otlichayutsya-protokoly-http-i-https/)​
    
4. **Дешифрование** — получатель расшифровывает данные с помощью того же ключа[webtronics](https://webtronics.ru/blog/chem-otlichayutsya-protokoly-http-i-https/)​
    

## **Основные характеристики HTTPS**

|---|---|
|**Порт**|Порт 443Lektsiia-13-14.-Prikladnoi-uroven.pdf​|
|**Шифрование**|SSL/TLS протоколы обеспечивают криптографическую защитуLektsiia-13-14.-Prikladnoi-uroven.pdf​|
|**Требование TLS**|Современные браузеры требуют TLS 1.2 или TLS 1.3[proselyte](https://proselyte.net/http-evolution/)​|
|**Аутентификация**|Цифровой сертификат подтверждает подлинность сервера[webtronics](https://webtronics.ru/blog/chem-otlichayutsya-protokoly-http-i-https/)​|

## **4. МЕТОДЫ HTTP ЗАПРОСА**

HTTP-методы определяют действие, которое должна выполнить сервер. Существует несколько основных методов, которые поддерживаются в разных версиях HTTP.

## **Методы по версиям HTTP**

**HTTP/0.9:**

- **GET** — запрос ресурсаLektsiia-13-14.-Prikladnoi-uroven.pdf​
    

**HTTP/1.0:**

- **GET** — запрос ресурсаLektsiia-13-14.-Prikladnoi-uroven.pdf​
    
- **POST** — отправка данных на серверLektsiia-13-14.-Prikladnoi-uroven.pdf​
    
- **HEAD** — получение заголовков GET без тела сообщенияLektsiia-13-14.-Prikladnoi-uroven.pdf​
    

**HTTP/1.1, HTTP/2, HTTP/3:**Lektsiia-13-14.-Prikladnoi-uroven.pdf​

- **GET** — запрос ресурса
    
- **POST** — отправка данных на сервер
    
- **HEAD** — получение заголовков без тела
    
- **PUT** — создание или замена ресурса по указанному URL
    
- **DELETE** — удаление ресурса
    
- **OPTIONS** — запрос списка ожидаемых методов сервера
    
- **TRACE** — для диагностики
    
- **CONNECT** — создание TCP-туннеля
    
- **PATCH** — частичное изменение ресурса
    

## **Детальное описание основных методов**

**GET-запрос**

- Используется для запроса данных с сервера
    
- Данные передаются в URL, видны в адресной строке[hexlet](https://ru.hexlet.io/blog/posts/metody-http-get-post-put-i-drugie)​
    
- Безопасен, идемпотентен (повторные запросы дают одинаковый результат)
    
- Не должен содержать тело сообщения
    
- Кэшируется браузером
    

**POST-запрос**

- Используется для отправки данных на серверLektsiia-13-14.-Prikladnoi-uroven.pdf​
    
- Данные передаются в теле сообщения (не видны в URL)[hexlet](https://ru.hexlet.io/blog/posts/metody-http-get-post-put-i-drugie)​
    
- Не кэшируется по умолчанию
    
- Может менять состояние сервера
    
- Не идемпотентен (повторные запросы могут дать разные результаты)
    

**PUT-запрос**

- Используется для создания или замены ресурса по указанному URLLektsiia-13-14.-Prikladnoi-uroven.pdf​
    
- Идемпотентен — повторное выполнение не меняет результат[hexlet](https://ru.hexlet.io/blog/posts/metody-http-get-post-put-i-drugie)​
    
- Требует полное тело ресурса для замены
    

**DELETE-запрос**

- Используется для удаления ресурсаLektsiia-13-14.-Prikladnoi-uroven.pdf​
    
- Идемпотентен — повторное удаление уже удаленного ресурса не меняет состояние системы[sky+1](https://sky.pro/wiki/sql/metody-http-get-post-i-drugie/)​
    
- Обычно возвращает код состояния 204 (No Content)[sky](https://sky.pro/wiki/sql/metody-http-get-post-i-drugie/)​
    
- При повторном запросе может вернуть 404 (Not Found)[nodul](https://www.nodul.ru/blog/metody-http-zaprosov-get-vs-post-vs-put-i-drugie)​
    

**PATCH-запрос**

- Используется для частичного изменения ресурсаLektsiia-13-14.-Prikladnoi-uroven.pdf​
    
- Отличается от PUT тем, что требует только измененные поля, а не полный ресурс[hexlet](https://ru.hexlet.io/blog/posts/metody-http-get-post-put-i-drugie)​
    

**HEAD-запрос**

- Аналогичен GET, но сервер возвращает только заголовки без тела сообщенияLektsiia-13-14.-Prikladnoi-uroven.pdf​
    
- Используется для проверки существования ресурса или получения метаинформации
    

**OPTIONS-запрос**

- Запрашивает список ожидаемых методов сервераLektsiia-13-14.-Prikladnoi-uroven.pdf​
    
- Позволяет клиенту узнать, какие операции поддерживает сервер

## **5. ВЕРСИИ HTTP**

## **HTTP/0.9**

- Первая версия HTTP (1991)
    
- Поддерживал только метод GETLektsiia-13-14.-Prikladnoi-uroven.pdf​
    
- Очень простая структура
    
- Сейчас полностью устарел
    

## **HTTP/1.0**

- Выпущена в 1996 году
    
- Добавлены методы POST и HEADLektsiia-13-14.-Prikladnoi-uroven.pdf​
    
- Каждый запрос требует отдельного TCP-соединения
    
- Отсутствует кэширование
    
- Требует 2 RTT на один объектLektsiia-13-14.-Prikladnoi-uroven.pdf​
    
- Медленная для современных приложений
    

## **HTTP/1.1**

- Выпущена в 1997/1999 годах[proselyte](https://proselyte.net/http-evolution/)​
    
- Основная версия интернета в течение 20+ лет[proselyte](https://proselyte.net/http-evolution/)​
    
- Добавлены методы PUT, DELETE, OPTIONS, TRACE, CONNECT, PATCHLektsiia-13-14.-Prikladnoi-uroven.pdf​
    
- **Постоянные соединения** — одно соединение может использоваться для нескольких объектовLektsiia-13-14.-Prikladnoi-uroven.pdf​
    
- Требует 1 RTT на один объектLektsiia-13-14.-Prikladnoi-uroven.pdf​
    
- Текстовый формат сообщений (ASCII)[proselyte](https://proselyte.net/http-evolution/)​
    
- Множественные TCP-соединения для параллелизма (обычно 4-8 на хост)[proselyte](https://proselyte.net/http-evolution/)​
    

## **HTTP/2**

- Выпущена в 2015 году[proselyte](https://proselyte.net/http-evolution/)​
    
- Большое улучшение производительности
    
- **Ключевые особенности**:[proselyte](https://proselyte.net/http-evolution/)​
    

|Особенность|Описание|
|---|---|
|**Бинарный фрейминг**|Вместо текстового формата используется бинарный для более эффективной обработки[proselyte](https://proselyte.net/http-evolution/)​|
|**Мультиплексирование**|Полное мультиплексирование — несколько потоков запросов/ответов в одном соединении без блокировки друг друга[proselyte](https://proselyte.net/http-evolution/)​|
|**Сжатие заголовков**|Заголовки сжимаются алгоритмом HPACK; повторяющиеся поля передаются как ссылки[proselyte](https://proselyte.net/http-evolution/)​|
|**Server Push**|Сервер может отправить дополнительные ресурсы проактивно без отдельного запроса[proselyte](https://proselyte.net/http-evolution/)​|
|**Приоритизация**|Клиент может присвоить потокам веса и зависимости для оптимизации загрузки[proselyte](https://proselyte.net/http-evolution/)​|
|**Одно TCP-соединение**|Все запросы работают через одно длительное соединение[proselyte](https://proselyte.net/http-evolution/)​|
|**Требование TLS**|Почти всегда используется с TLS/HTTPS; браузеры не поддерживают незашифрованный HTTP/2[proselyte](https://proselyte.net/http-evolution/)​|
|**Требование TLS версии**|Требует TLS 1.2 или TLS 1.3[proselyte](https://proselyte.net/http-evolution/)​|

**Преимущества HTTP/2:**

- Значительное улучшение производительности[proselyte](https://proselyte.net/http-evolution/)​
    
- Уменьшение задержек при загрузке веб-страниц
    
- Меньше накладных расходов на установление соединений
    

**Недостатки HTTP/2:**

- Требует обновления инфраструктуры (прокси, балансировщиков)[proselyte](https://proselyte.net/http-evolution/)​
    
- Многие старые оптимизации HTTP/1.1 становятся неэффективными[proselyte](https://proselyte.net/http-evolution/)​
    

## **HTTP/3**

- Выпущена в 2022 году (стандартизирована в RFC 9114)[temofeev](https://temofeev.ru/info/articles/http-1-2-i-3-prosto/)​
    
- Используется с протоколом QUIC вместо TCP[temofeev](https://temofeev.ru/info/articles/http-1-2-i-3-prosto/)​
    
- **Ключевые улучшения**:[temofeev](https://temofeev.ru/info/articles/http-1-2-i-3-prosto/)​
    
    - Одно рукопожатие вместо множественных для установления соединения и шифрования[temofeev](https://temofeev.ru/info/articles/http-1-2-i-3-prosto/)​
        
    - Сохранение соединения при изменении сетей (например, при переходе с WiFi на мобильную сеть)[temofeev](https://temofeev.ru/info/articles/http-1-2-i-3-prosto/)​
        
    - Не требует повторных рукопожатий при разрыве соединения благодаря уникальному идентификатору соединения[temofeev](https://temofeev.ru/info/articles/http-1-2-i-3-prosto/)​
        
    - Улучшенная обработка потери пакетов

## **7. HTTP-СОЕДИНЕНИЯ**

Существуют два типа HTTP-соединений:Lektsiia-13-14.-Prikladnoi-uroven.pdf​

## **Постоянные соединения**

- Через одно соединение клиент-сервер может отправить несколько объектовLektsiia-13-14.-Prikladnoi-uroven.pdf​
    
- Требует 1 RTT на один объектLektsiia-13-14.-Prikladnoi-uroven.pdf​
    
- После отправки ответа сервер оставляет соединение открытымLektsiia-13-14.-Prikladnoi-uroven.pdf​
    
- Клиент отправит запрос как только встретит содержащий ссылку объектLektsiia-13-14.-Prikladnoi-uroven.pdf​
    
- Более эффективны для загрузки нескольких ресурсов
    

## **Непостоянные соединения**

- Один объект — одно соединениеLektsiia-13-14.-Prikladnoi-uroven.pdf​
    
- Требуется 2 RTT на один объектLektsiia-13-14.-Prikladnoi-uroven.pdf​
    
- Нагрузка на ОС для каждого нового соединенияLektsiia-13-14.-Prikladnoi-uroven.pdf​
    
- Браузеры часто открывают параллельные TCP-соединения для извлечения объектов по ссылкеLektsiia-13-14.-Prikladnoi-uroven.pdf​
    

_(RTT — Round Trip Time — время прохождения пути от клиента до сервера и обратно для небольшого пакета)_Lektsiia-13-14.-Prikladnoi-uroven.pdf​

**SSL (Secure Socket Layer)**

- Предшественник TLS
    
- Устарел и больше не рекомендуется использовать
    

**TLS (Transport Layer Security)**

- Современный стандарт криптографической защиты[webtronics](https://webtronics.ru/blog/chem-otlichayutsya-protokoly-http-i-https/)​Lektsiia-13-14.-Prikladnoi-uroven.pdf​
    
- Обеспечивает конфиденциальность, целостность и аутентификацию[webtronics](https://webtronics.ru/blog/chem-otlichayutsya-protokoly-http-i-https/)​
    
- Версия 1.2 или выше требуется для современных браузеров[proselyte](https://proselyte.net/http-evolution/)​

## **Заключение**

HTTP — это основной протокол веб-приложений, который претерпел значительную эволюцию от версии 0.9 к HTTP/3. HTTPS обеспечивает безопасность через шифрование SSL/TLS. Выбор версии HTTP зависит от требований приложения, но HTTP/2 и HTTP/3 предпочтительны для современных веб-приложений благодаря их производительности и встроенным механизмам безопасности.
