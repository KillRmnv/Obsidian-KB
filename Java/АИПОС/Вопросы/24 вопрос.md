ICMP (Internet Control Message Protocol) — вспомогательный протокол сетевого уровня, используемый для передачи сообщений об ошибках и диагностических сведений между узлами и маршрутизаторами в сетях IP, однако он не предусмотрен для передачи пользовательских данных. Протокол ICMP определён в RFC 792 и работает как составная часть протокола IP, помогая выявить проблемы с доставкой пакетов и диагностировать состояние сетей.[ip-calculator+1](https://ip-calculator.ru/blog/networking/icmp-error-messages-and-format-explained/)​

## Назначение и роль ICMP

ICMP решает две основные задачи:[studfile+1](https://studfile.net/preview/16868795/page:3/)​

1. **Сообщение об ошибках:** когда маршрутизатор или конечный хост не могут доставить или обработать IP‑пакет (например, сеть недоступна, хост не существует, порт закрыт), они отправляют ICMP‑сообщение об ошибке обратно источнику пакета, объясняя причину отказа.[pmn.narod+1](https://pmn.narod.ru/disciplins/net/09.pdf)​
    
2. **Диагностика и проверка связи:** ICMP используется для запроса информации о состоянии удалённых узлов (например, команда ping), для трассировки маршрутов (traceroute), определения параметров сети (маска подсети, время).[aws.amazon+1](https://aws.amazon.com/ru/what-is/icmp/)​
    

Важно отметить, что ICMP сам по себе не гарантирует доставку сообщений об ошибках; они подчиняются обычным правилам IP и могут быть потеряны или задержаны.[intuit](https://intuit.ru/studies/courses/1123/200/lecture/5179)​

## Структура ICMP‑сообщения

Все ICMP‑сообщения начинаются с унифицированного 8‑байтного заголовка:[net.academy+1](https://net.academy.lv/lection/net_LS-10RU_icmp.pdf)​

text

`0          1          2          3 +---------+---------+----------+ |  Type   |  Code   | Checksum | +---------+---------+----------+ | Identifier (2 байта) | Sequence (2 байта) | +---------------------+-------------------+ |          Data (переменная длина)         | +---------------------------------------------+`

- **Type (1 байт):** тип сообщения; определяет назначение и формат остального сообщения. Всего может быть 256 различных типов, но используется значительно меньше.[iteducate.github+1](https://iteducate.github.io/networks/icmp_prot/)​
    
- **Code (1 байт):** уточняющий код; для сообщений об ошибках конкретизирует причину; для запросов-ответов обычно равен 0. До 256 кодов на тип, но не все комбинации используются.[net.academy+1](https://net.academy.lv/lection/net_LS-10RU_icmp.pdf)​
    
- **Checksum (2 байта):** контрольная сумма для проверки целостности сообщения (вычисляется по всему сообщению, включая заголовок и данные).[ip-calculator+1](https://ip-calculator.ru/blog/networking/icmp-error-messages-and-format-explained/)​
    
- **Identifier и Sequence (для некоторых типов):** используются для связывания запроса и ответа, особенно в ping.[net.academy](https://net.academy.lv/lection/net_LS-10RU_icmp.pdf)​
    
- **Data (переменная длина):** дополнительные данные, формат которых зависит от типа сообщения. Для ошибок здесь содержится исходный IP‑заголовок (20–60 байт) и первые 8 байт данных пакета, вызвавшего ошибку.[ip-calculator+1](https://ip-calculator.ru/blog/networking/icmp-error-messages-and-format-explained/)​
    

## Типы и коды ICMP‑сообщений

ICMP‑сообщения можно разделить на две группы:[iteducate.github+1](https://iteducate.github.io/networks/icmp_prot/)​

## Сообщения об ошибках

Эти сообщения отправляются в случае проблем с доставкой пакета:

|Type|Описание|Основные коды|
|---|---|---|
|**3**|Destination Unreachable (пункт назначения недоступен)|0 — нет маршрута; 1 — сеть недоступна; 2 — хост недоступен; 3 — протокол недоступен; 4 — порт недоступен; 5 — требуется фрагментация, но установлен флаг DF; 6–13 — прочие причины.[ip-calculator](https://ip-calculator.ru/blog/networking/icmp-error-messages-and-format-explained/)​|
|**5**|Redirect (перенаправление)|Маршрутизатор рекомендует источнику использовать другой маршрутизатор для доступа к целевой сети.[iteducate.github](https://iteducate.github.io/networks/icmp_prot/)​|
|**11**|Time Exceeded (превышен лимит времени)|0 — TTL = 0 при передаче; 1 — TTL = 0 при сборке фрагментов.[net.academy](https://net.academy.lv/lection/net_LS-10RU_icmp.pdf)​|
|**12**|Parameter Problem (ошибка в заголовке)|Маршрутизатор обнаружил ошибку в полях IP‑заголовка.[net.academy](https://net.academy.lv/lection/net_LS-10RU_icmp.pdf)​|

Важное правило: **ICMP‑сообщение об ошибке никогда не отправляется в ответ на другое ICMP‑сообщение об ошибке**, чтобы избежать бесконечных циклов.[pmn.narod](https://pmn.narod.ru/disciplins/net/09.pdf)​

## Сообщения запрос-ответ

Эти сообщения используются для диагностики; обе стороны обмена имеют код 0:

|Type|Описание|Применение|
|---|---|---|
|**0 / 8**|Echo Reply (отклик эхо) / Echo Request (запрос эхо)|Команда ping; проверка доступности хоста и измерение задержки.[net.academy](https://net.academy.lv/lection/net_LS-10RU_icmp.pdf)​|
|**13 / 14**|Timestamp (запрос времени) / Timestamp Reply (отклик времени)|Синхронизация часов между узлами; содержит Sender Timestamp, Receive Timestamp, Transmit Timestamp.[net.academy](https://net.academy.lv/lection/net_LS-10RU_icmp.pdf)​|
|**17 / 18**|Address Mask Request / Address Mask Reply|Запрос маски подсети (в основном устарели).[iteducate.github](https://iteducate.github.io/networks/icmp_prot/)​|
|**15 / 16**|Information Request / Information Reply|Запрос о принадлежности хоста к сети (устарели).[iteducate.github](https://iteducate.github.io/networks/icmp_prot/)​|

## Сообщение об ошибке: Destination Unreachable (Type 3)

Это наиболее часто встречающееся ICMP‑сообщение об ошибке. Основные коды:[ip-calculator](https://ip-calculator.ru/blog/networking/icmp-error-messages-and-format-explained/)​

- **Code 0:** нет маршрута к месту назначения (в таблице маршрутизации нет записи).
    
- **Code 1:** сообщение с пунктом назначения административно запрещено (фильтр, ACL блокирует).
    
- **Code 2:** за пределами исходного адреса (rare).
    
- **Code 3:** адрес недоступен (например, неправильная маска подсети).
    
- **Code 4:** порт недоступен (хост доступен, но на этом порту никто не слушает); используется в traceroute.
    

## Сообщение Time Exceeded (Type 11)

Отправляется маршрутизатором, когда TTL в пакете обнуляется:[net.academy](https://net.academy.lv/lection/net_LS-10RU_icmp.pdf)​

- **Code 0:** TTL равно 0 при передаче (пакет отброшен маршрутизатором, так как его TTL истёк).
    
- **Code 1:** TTL равно 0 при сборке фрагментов (фрагменты собирались слишком долго).
    

Это сообщение критично для работы команды traceroute.[net.academy](https://net.academy.lv/lection/net_LS-10RU_icmp.pdf)​

## Диагностика: команда PING

PING (Packet Internet Groper) — утилита, использующая ICMP Echo Request и Echo Reply для проверки доступности узла и измерения задержки.[neoserver+1](https://neoserver.ru/help/network-monitoring-linux-ping-traceroute-mtr)​

**Процесс:**[justogroup+1](http://www.justogroup.ru/dokumentacija/cisco/kommutiruemye_seti/ispolizovanie_rasshirennih_komand_ping_i_traceroute.pdf)​

1. Источник отправляет ICMP Echo Request (Type 8, Code 0) на IP‑адрес целевого хоста.
    
2. Путём маршрутизации пакет доходит до целевого хоста.
    
3. Целевой хост получает Echo Request и генерирует Echo Reply (Type 0, Code 0) с теми же данными, отправляя его обратно к источнику.
    
4. Источник измеряет время, прошедшее от отправки запроса к получению ответа (round-trip time, RTT).
    

**Формат пакета ICMP Echo:**[net.academy](https://net.academy.lv/lection/net_LS-10RU_icmp.pdf)​

text

`Type = 8 (Echo Request) или 0 (Echo Reply) Code = 0 Checksum = [контрольная сумма] Identifier = [ID процесса или потока] Sequence Number = [номер последовательности] Data = [тестовые данные]`

**Опции Record Route и Source Route:**[cbs](https://cbs.ru/lib/technical-articles/10946/)​

- **Record Route:** маршрутизаторы на пути записывают свои IP‑адреса в заголовок IP; отвечающий хост копирует эту информацию в Echo Reply.
    
- **Strict Source Route:** источник может заранее задать список маршрутизаторов, через которые должен пройти пакет; полезно для тестирования определённых путей.
    

## Диагностика: команда TRACEROUTE

Traceroute (tracert в Windows) — утилита для отслеживания маршрута пакета от источника к узлу назначения, показывающая все промежуточные маршрутизаторы и задержки на каждом шаге.[justogroup+1](http://www.justogroup.ru/dokumentacija/cisco/kommutiruemye_seti/ispolizovanie_rasshirennih_komand_ping_i_traceroute.pdf)​

**Механизм работы (Unix traceroute):**[justogroup+1](http://www.justogroup.ru/dokumentacija/cisco/kommutiruemye_seti/ispolizovanie_rasshirennih_komand_ping_i_traceroute.pdf)​

1. Отправляется **последовательность UDP‑пакетов** на несуществующий порт (по умолчанию 33434) на целевом хосте, с **увеличивающимся значением TTL** (1, 2, 3, ...).
    
2. Первый пакет (TTL=1) достигает первого маршрутизатора, его TTL истекает → маршрутизатор отправляет ICMP Time Exceeded (Type 11, Code 0).
    
3. Второй пакет (TTL=2) проходит первый маршрутизатор, достигает второго → второй маршрутизатор отправляет ICMP Time Exceeded.
    
4. Этот процесс продолжается, пока пакет не достигнет целевого хоста.
    
5. Когда UDP‑пакет попадает на целевой хост на несуществующий порт, хост отправляет ICMP Port Unreachable (Type 3, Code 4).
    
6. Traceroute завершается после получения этого сообщения.
    

**Windows tracert:**[net.academy](https://net.academy.lv/lection/net_LS-10RU_icmp.pdf)​

- Использует ICMP Echo Request (как ping) вместо UDP.
    
- Механизм аналогичен: увеличивающийся TTL, получение Time Exceeded от каждого маршрутизатора.
    

**Вывод traceroute содержит:**[justogroup](http://www.justogroup.ru/dokumentacija/cisco/kommutiruemye_seti/ispolizovanie_rasshirennih_komand_ping_i_traceroute.pdf)​

- Номер хопа.
    
- IP‑адрес и имя маршрутизатора (если доступно).
    
- Задержку в миллисекундах (обычно 3 попытки с разными TTL).
    
- Звёздочки (*), если маршрутизатор не отвечает.
    

## Практические применения ICMP

1. **Проверка доступности сети:** ping проверяет, работает ли целевой хост и доступен ли он по IP.
    
2. **Диагностика маршрутов:** traceroute показывает, где именно прерывается связь и через какие маршрутизаторы проходит трафик.
    
3. **Определение MTU:** ICMP Fragmentation Needed (Type 3, Code 4) помогает осуществить Path MTU Discovery.
    
4. **Контроль перегрузок:** ICMP Source Quench используется для уведомления источника о перегрузке (хотя сейчас редко применяется).
    
5. **Перенаправление маршрутов:** ICMP Redirect (Type 5) позволяет маршрутизаторам рекомендовать более оптимальный путь.
    

## Ограничения и соображения безопасности

Многие администраторы блокируют ICMP на границах сетей по соображениям безопасности:[aws.amazon](https://aws.amazon.com/ru/what-is/icmp/)​

- **Ping флуд:** большое количество ICMP Echo Requests может использоваться для DDoS‑атак.
    
- **Information disclosure:** ICMP сообщения могут раскрывать информацию о топологии сети.
    

Однако полное отключение ICMP может затруднить диагностику сетевых проблем, поэтому часто блокируют только определённые типы или ограничивают скорость.
