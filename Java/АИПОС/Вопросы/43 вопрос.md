## **ЭКЗАМЕНАЦИОННЫЙ ОТВЕТ: Протоколы Telnet и SSH**

## **ВВЕДЕНИЕ**

**Telnet** и **SSH** — это два протокола прикладного уровня, предназначенные для удаленного управления компьютерами и серверами через командную строку. Оба позволяют администраторам работать с удаленными устройствами, но существенно отличаются по уровню безопасности и функциональности.Lektsiia-13-14.-Prikladnoi-uroven.pdf​

---

## **1. ПРОТОКОЛ TELNET (Teletype Network)**

## **Определение и назначение**

**Telnet** — это сетевой протокол для реализации текстового терминального интерфейса по сети при помощи TCP. Telnet позволяет подключиться к удаленной машине и работать с ней как с локальным терминалом.[dzen+1](https://dzen.ru/a/ZAdgyyZBz1wXbYLu)​Lektsiia-13-14.-Prikladnoi-uroven.pdf​

## **Основные характеристики Telnet**

|Характеристика|Описание|
|---|---|
|**Уровень OSI**|Протокол прикладного уровня|
|**Порт**|Порт 23 (по умолчанию)Lektsiia-13-14.-Prikladnoi-uroven.pdf​[lightyear](https://lightyear.ai/tips/telnet-versus-ssh)​|
|**Транспортный протокол**|TCP|
|**Архитектура**|Клиент-сервер|
|**Шифрование**|**Отсутствует** — данные передаются в открытом видеLektsiia-13-14.-Prikladnoi-uroven.pdf​[dzen](https://dzen.ru/a/ZAdgyyZBz1wXbYLu)​|
|**Аутентификация**|Простая (логин и пароль в открытом виде)[scanitex+1](https://scanitex.com/ru/resources/ports/tcp/23)​|
|**Модель связи**|Двусторонняя, реал-тайм|

## **Принцип работы Telnet**

1. **Подключение** — клиент подключается к серверу на порт 23[wiki.merionet](https://wiki.merionet.ru/articles/konsol-terminal-telnet-i-ssh-chto-eto-i-zachem-nuzhno)​
    
2. **Аутентификация** — пользователь вводит логин и пароль[dzen](https://dzen.ru/a/ZAdgyyZBz1wXbYLu)​
    
    - **Важно:** логин и пароль передаются в открытом виде (plain text)
        
3. **Виртуальный терминал** — клиент получает виртуальный доступ к терминалу удаленной машины[lightyear](https://lightyear.ai/tips/telnet-versus-ssh)​
    
4. **Выполнение команд** — пользователь вводит команды, которые выполняются на удаленной машине[lightyear](https://lightyear.ai/tips/telnet-versus-ssh)​
    
5. **Получение результатов** — результаты выполнения команд передаются обратно клиенту[lightyear](https://lightyear.ai/tips/telnet-versus-ssh)​
    
6. **Разрыв соединения** — завершение сессии путем отключения

## **2. ПРОТОКОЛ SSH (Secure Shell)**

## **Определение и назначение**

**SSH (Secure Shell)** — это сетевой протокол прикладного уровня, позволяющий производить удаленное управление операционной системой и туннелирование TCP-соединений. SSH обеспечивает безопасный, зашифрованный канал связи между клиентом и сервером.[8host+1](https://www.8host.com/blog/osnovy-shifrovaniya-i-soedineniya-ssh/)​Lektsiia-13-14.-Prikladnoi-uroven.pdf​

## **Основные характеристики SSH**

|Характеристика|Описание|
|---|---|
|**Уровень OSI**|Протокол прикладного уровня|
|**Порт**|Порт 22 (по умолчанию)Lektsiia-13-14.-Prikladnoi-uroven.pdf​[wiki.merionet](https://wiki.merionet.ru/articles/konsol-terminal-telnet-i-ssh-chto-eto-i-zachem-nuzhno)​|
|**Транспортный протокол**|TCP|
|**Архитектура**|Клиент-сервер|
|**Шифрование**|**Встроенное шифрование всего трафика**[8host+1](https://www.8host.com/blog/osnovy-shifrovaniya-i-soedineniya-ssh/)​|
|**Аутентификация**|Открытые ключи (RSA, DSA, ECC) или пароли[skyeng](https://skyeng.ru/magazine/wiki/it-industriya/chto-takoe-ssh/)​|
|**Целостность данных**|Проверка целостности с помощью хеширования[8host+1](https://www.8host.com/blog/osnovy-shifrovaniya-i-soedineniya-ssh/)​|
|**Трафик**|Весь трафик зашифрован уникальным ключом[wiki.merionet](https://wiki.merionet.ru/articles/konsol-terminal-telnet-i-ssh-chto-eto-i-zachem-nuzhno)​|

## **Принцип работы SSH**

**Этапы установления SSH-соединения**:[skyeng+1](https://skyeng.ru/magazine/wiki/it-industriya/chto-takoe-ssh/)​

1. **TCP-соединение** — клиент подключается к серверу на порт 22
    
2. **Обмен версиями** — клиент и сервер обмениваются информацией о версиях SSH
    
3. **Обмен ключами (Key Exchange)**
    
    - Используется алгоритм Диффи-Хеллмана (Diffie-Hellman)
        
    - Обе стороны генерируют временные пары ключей
        
    - Обмен открытыми ключами для создания общего секретного ключа
        
    - **Ни одна сторона не полностью контролирует ключ**[8host](https://www.8host.com/blog/osnovy-shifrovaniya-i-soedineniya-ssh/)​
        
4. **Установка симметричного шифрования**
    
    - Общий ключ используется для симметричного шифрования (AES)[8host](https://www.8host.com/blog/osnovy-shifrovaniya-i-soedineniya-ssh/)​
        
    - Все последующие данные шифруются с помощью этого ключа
        
5. **Аутентификация пользователя**
    
    - Открытый ключ сервера отправляется клиенту
        
    - Сервер отправляет зашифрованное сообщение
        
    - Клиент расшифровывает с помощью приватного ключа
        
    - Клиент подтверждает подлинность и вычисляет хеш MD5[8host](https://www.8host.com/blog/osnovy-shifrovaniya-i-soedineniya-ssh/)​
        
6. **Открытие защищенного канала** — установка безопасного соединения
    
7. **Выполнение команд** — все команды и данные передаются зашифрованными

## **3. МЕХАНИЗМ ШИФРОВАНИЯ SSH**

## **Асимметричное шифрование (для обмена ключами)**

SSH использует **асимметричное шифрование** (открытый/закрытый ключ) на этапе обмена ключами:[skyeng+1](https://skyeng.ru/magazine/wiki/it-industriya/chto-takoe-ssh/)​

- **Публичный ключ** — хранится на сервере, используется для шифрования
    
- **Приватный ключ** — хранится на клиентском устройстве, используется для дешифрования[skyeng](https://skyeng.ru/magazine/wiki/it-industriya/chto-takoe-ssh/)​
    
- **Алгоритмы:** RSA, DSA, ECC[skyeng](https://skyeng.ru/magazine/wiki/it-industriya/chto-takoe-ssh/)​
    

## **Симметричное шифрование (для сеанса)**

После установления общего секретного ключа SSH использует **симметричное шифрование**:[8host](https://www.8host.com/blog/osnovy-shifrovaniya-i-soedineniya-ssh/)​

- **Один ключ для обеих сторон** — одинаковый ключ на клиенте и сервере
    
- **Более быстрое** — симметричное шифрование быстрее асимметричного
    
- **Алгоритмы:** AES, DES, 3DES[skyeng+1](https://skyeng.ru/magazine/wiki/it-industriya/chto-takoe-ssh/)​
    

## **Алгоритм Диффи-Хеллмана (DH)**

Этот алгоритм позволяет обеим сторонам создать один и тот же секретный ключ, не передавая его по сети:[8host](https://www.8host.com/blog/osnovy-shifrovaniya-i-soedineniya-ssh/)​

1. Обе стороны выбирают большое простое число
    
2. Выбирается генератор шифрования (например, AES)
    
3. Каждая сторона придумывает свой закрытый ключ
    
4. На основе закрытого ключа и общего числа каждая сторона создает открытый ключ
    
5. Открытые ключи обмениваются
    
6. Каждая сторона использует свой закрытый ключ, открытый ключ другой стороны и общее число для создания общего секретного ключа
    
7. **Обе стороны получают один и тот же секретный ключ**[8host](https://www.8host.com/blog/osnovy-shifrovaniya-i-soedineniya-ssh/)​
    

## **Проверка целостности данных**

SSH использует **хеширование** для создания кодов аутентификации сообщений (MAC):

- **Гарантирует целостность** — данные не были изменены в пути[lightyear+1](https://lightyear.ai/tips/telnet-versus-ssh)​
    
- **Защищает от MITM-атак** — любые изменения данных будут обнаружены
    
- **Алгоритмы:** MD5, SHA-1, SHA-256

## **4. МЕТОДЫ АУТЕНТИФИКАЦИИ SSH**

SSH поддерживает несколько методов аутентификации:[skyeng](https://skyeng.ru/magazine/wiki/it-industriya/chto-takoe-ssh/)​

## **1. Аутентификация по паролю**

- Пользователь вводит пароль
    
- Пароль передается через защищенный зашифрованный канал
    
- **Более безопасно, чем Telnet**, но менее безопасно, чем ключи
    

## **2. Аутентификация по открытым ключам**

- Клиент хранит приватный ключ
    
- Сервер хранит публичный ключ
    
- Клиент доказывает свою личность с помощью приватного ключа
    
- **Наиболее безопасный метод**[skyeng](https://skyeng.ru/magazine/wiki/it-industriya/chto-takoe-ssh/)​
    
- Поддерживает разные типы ключей: RSA, DSA, ECC
    

## **3. Аутентификация по хостам**

- Основана на доверии хостам
    
- Используется реже, менее безопасна

## **7. БЕЗОПАСНОСТЬ TELNET**

## **Проблемы безопасности**

1. **Отсутствие шифрования**
    
    - Все команды передаются в открытом виде
        
    - Любой может перехватить трафик
        
2. **Видимые учетные данные**
    
    - Логин и пароль видны в сетевом трафике
        
    - Легко найти в дампе трафика
        
3. **Уязвимость MITM (Man-in-the-Middle)**
    
    - Злоумышленник может встать между клиентом и сервером
        
    - Перехватить и изменить данные
        
4. **Отсутствие аутентификации сервера**
    
    - Клиент не может убедиться в подлинности сервера
        
    - Возможно подключение к фальшивому серверу
        
5. **Подслушивание (Eavesdropping)**
    
    - На общедоступных сетях легко подслушивать[alotceriot](https://www.alotceriot.com/ru/%D0%B4%D0%B5%D0%BC%D0%B8%D1%81%D1%82%D0%B8%D1%84%D0%B8%D1%86%D0%B8%D1%80%D1%83%D1%8F-telnet-%D1%80%D1%83%D0%BA%D0%BE%D0%B2%D0%BE%D0%B4%D1%81%D1%82%D0%B2%D0%BE-%D0%BF%D0%BE-%D1%83%D0%B4%D0%B0%D0%BB/)​
        
    - Нет защиты от пассивных атак[alotceriot](https://www.alotceriot.com/ru/%D0%B4%D0%B5%D0%BC%D0%B8%D1%81%D1%82%D0%B8%D1%84%D0%B8%D1%86%D0%B8%D1%80%D1%83%D1%8F-telnet-%D1%80%D1%83%D0%BA%D0%BE%D0%B2%D0%BE%D0%B4%D1%81%D1%82%D0%B2%D0%BE-%D0%BF%D0%BE-%D1%83%D0%B4%D0%B0%D0%BB/)​
        

## **Уязвимости Telnet**

**Статистика атак на IoT:**

- Telnet атакуется в **75,4% всех атак** на IoT-устройства[habr](https://habr.com/ru/companies/kaspersky/articles/424259/)​
    
- SSH атакуется в 11,59% случаев[habr](https://habr.com/ru/companies/kaspersky/articles/424259/)​
    
- Причина: Telnet легче использовать благодаря отсутствию шифрования

## **8. БЕЗОПАСНОСТЬ SSH**

## **Защитные механизмы SSH**

1. **Шифрование данных**
    
    - Все данные, команды и вывод — зашифрованы[lightyear+1](https://lightyear.ai/tips/telnet-versus-ssh)​
        
    - Использует сильные алгоритмы (AES, RSA)[skillbox+1](https://skillbox.ru/media/code/ssh-chto-eto-i-kak-im-polzovatsya-gaid-dlya-novichkov/)​
        
2. **Аутентификация**
    
    - Верификация подлинности сервера перед передачей данных[lightyear](https://lightyear.ai/tips/telnet-versus-ssh)​
        
    - Поддерживает открытые ключи и пароли[skyeng](https://skyeng.ru/magazine/wiki/it-industriya/chto-takoe-ssh/)​
        
3. **Целостность данных**
    
    - MAC (Message Authentication Code) гарантирует, что данные не изменены[8host](https://www.8host.com/blog/osnovy-shifrovaniya-i-soedineniya-ssh/)​
        
    - Защита от подделки и модификации[lightyear](https://lightyear.ai/tips/telnet-versus-ssh)​
        
4. **Защита от MITM-атак**
    
    - Обмен ключами предотвращает перехват[8host](https://www.8host.com/blog/osnovy-shifrovaniya-i-soedineniya-ssh/)​
        
    - Аутентификация сервера защищает от фальшивых серверов[lightyear](https://lightyear.ai/tips/telnet-versus-ssh)​
        

## **Известные уязвимости SSH**

|Уязвимость|Описание|
|---|---|
|**Terrapin Attack**|Атака на бинарный протокол пакетов SSH, позволяет манипулировать номерами последовательности[arxiv](https://arxiv.org/html/2312.12422v1)​|

## **Меры безопасности SSH**

- Отключение входа по паролю (использование только ключей)
    
- Переназначение порта 22 на другой
    
- Использование ключей вместо паролей[habr](https://habr.com/ru/companies/kaspersky/articles/424259/)​
    
- Регулярное обновление SSH-сервера
    
- Ограничение доступа через firewall

## **11. РЕКОМЕНДАЦИИ**

## **ТЕЛNET**

**Используйте только если:**

- Работаете с устаревшим оборудованием, которое не поддерживает SSH
    
- Находитесь в полностью изолированной локальной сети
    
- Тестируете доступность портов (без критичных данных)
    

**Никогда не используйте Telnet для:**

- Передачи конфиденциальных данных
    
- Управления серверами через интернет
    
- Современных систем
    

## **SSH**

**Используйте всегда, когда:**

- Нужен удаленный доступ
    
- Требуется безопасность
    
- Работаете с серверами или облачными платформами
    

**Лучшие практики SSH:**

- Используйте аутентификацию по открытым ключам (отключите пароли)[habr](https://habr.com/ru/companies/kaspersky/articles/424259/)​
    
- Переназначьте порт 22 на другой
    
- Регулярно обновляйте SSH-сервер
    
- Мониторьте попытки подключения
    
- Используйте инструменты типа fail2ban для защиты

## **Заключение**

**Telnet** — это устаревший протокол, который раньше был стандартом для удаленного управления, но теперь полностью заменен на **SSH**. Основная разница заключается в **безопасности**:

- **Telnet** передает все данные в открытом виде, что делает его уязвимым для атак и перехвата.
    
- **SSH** обеспечивает полное шифрование, аутентификацию и проверку целостности данных.
    

В современном мире **SSH — это обязательный стандарт** для любого удаленного доступа. Telnet следует рассматривать только как историческая реликвия для управления древним оборудованием в полностью защищенных локальных сетях. Статистика показывает, что Telnet целевая в 75% всех атак на IoT-устройства, тогда как SSH разработан с учетом безопасности и значительно устойчивее к атакам.
