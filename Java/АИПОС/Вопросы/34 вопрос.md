# Экзаменационный билет: Состояния соединений в протоколе TCP

## 1. Введение

Соединение в TCP — это не просто передача данных. Это **сложный жизненный цикл**, в котором соединение проходит через ряд состояний в зависимости от того, какие события происходят. Каждое состояние представляет определённую точку в жизни соединения и определяет, какие операции допустимы.[](https://maxnilz.com/docs/004-network/003-tcp-connection-state/)​​

**Ключевая концепция:** Состояния TCP отражают информацию, хранящуюся в **Transmission Control Block (TCB)** — структуре ядра операционной системы, которая содержит все данные о соединении (номера портов, номера последовательности, размеры окон и т.д.).[](https://maxnilz.com/docs/004-network/003-tcp-connection-state/)​

---

## 2. Все 11 состояний TCP

Согласно RFC 793, TCP соединение проходит через следующие состояния (перечислены в порядке их появления):

## 2.1. Состояния установления соединения

| Состояние        | Сторона | Описание                                                                                                                                                                       |
| ---------------- | ------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **CLOSED**       | Обе     | **Фиктивное состояние.** Соединения нет. TCB не существует. Это начальное состояние (до `OPEN`) и финальное состояние (после закрытия).                                        |
| **LISTEN**       | Сервер  | Сервер выполнил **пассивное открытие** (passive OPEN) и ждёт входящего запроса на соединение от клиента. Сервер готов принять SYN.                                             |
| **SYN-SENT**     | Клиент  | Клиент выполнил **активное открытие** (active OPEN) и отправил SYN, но еще не получил SYN-ACK от сервера. Клиент ждет соответствующего ответа.                                 |
| **SYN-RECEIVED** | Сервер  | Сервер получил SYN от клиента и отправил свой SYN-ACK, но еще не получил финальный ACK от клиента. Соединение находится в процессе установления.                               |
| **ESTABLISHED**  | Обе     | **Нормальное состояние передачи данных.** Оба конца завершили 3-way handshake и могут обмениваться данными. Это основное состояние, в котором работает большинство приложений. |
|                  |         |                                                                                                                                                                                |

## 2.2. Состояния разрыва соединения

| Состояние      | Сторона     | Описание                                                                                                                                                                                                                  |
| -------------- | ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **FIN-WAIT-1** | Инициатор   | Сторона, которая первой захотела закрыть соединение, отправила FIN и ждет подтверждения (ACK) от другой стороны. Может одновременно получить и FIN от другой стороны.                                                     |
| **FIN-WAIT-2** | Инициатор   | Инициатор получил ACK на свой FIN, значит другая сторона **получила** сигнал закрытия. Теперь ждет получения FIN от другой стороны (которая должна перейти в CLOSE-WAIT, закончить данные и отправить свой FIN).          |
| **CLOSE-WAIT** | Респондент  | Сторона, которая **получила** FIN, отправила ACK на него и теперь ждет, пока приложение закончит отправлять свои данные и сообщит TCP о необходимости закрыться. После этого она перейдет в LAST-ACK.                     |
| **CLOSING**    | Обе (редко) | **Редкое состояние.** Обе стороны одновременно инициировали закрытие. Сторона отправила FIN и получила FIN от другой, но еще не получила ACK на свой FIN.                                                                 |
| **LAST-ACK**   | Респондент  | Сторона, которая первоначально **получила** FIN (была в CLOSE-WAIT), отправила свой FIN и ждет финального ACK. После получения ACK переходит в CLOSED.                                                                    |
| **TIME-WAIT**  | Инициатор   | **Критическое состояние очистки.** Сторона, которая первоначально инициировала закрытие, отправила финальный ACK. Теперь ждет **2 × MSL** (Maximum Segment Lifetime, обычно 60 секунд на Linux) перед переходом в CLOSED. |
## 3. Жизненный цикл соединения: На примере сервера и клиента

## 3.1. Фаза установления (3-Way Handshake)

```bash
КЛИЕНТ                              СЕРВЕР
  CLOSED                            CLOSED
    |                                 |
    | active OPEN                     | passive OPEN
    | (создать TCB)                   | (создать TCB)
    v                                 v
  SYN-SENT ----[1. SYN]---->       LISTEN
    |                                 |
    |                          rcv SYN, snd SYN,ACK
    |                                 v
    |                            SYN-RECEIVED
    |                                 |
    |<---[2. SYN,ACK]----            |
    |                                 |
    | rcv SYN,ACK
    | snd ACK
    v                                 v
ESTABLISHED <---[3. ACK]--->     ESTABLISHED

```


**Что происходит:**

1. **Клиент** → отправляет `[SYN]` с начальным номером последовательности `SEQ=X`.
    
2. **Сервер** ← получает `[SYN]`, запоминает `SEQ` клиента, выбирает свой `SEQ=Y` и отправляет `[SYN, ACK]` с `ACK=X+1`.
    
3. **Клиент** → получает `[SYN, ACK]`, отправляет `[ACK]` с `SEQ=X+1`, `ACK=Y+1`.
    
4. **Оба** → переходят в **ESTABLISHED**. Теперь можно передавать данные.
    

## 3.2. Фаза передачи данных

Обе стороны остаются в состоянии **ESTABLISHED**, пока идет обмен данными. Каждый пакет с данными содержит `[ACK]` флаг, подтверждающий получение предыдущих данных.

## 3.3. Фаза разрыва (4-Way Close) — Нормальный сценарий

```
КЛИЕНТ (закрыл первым)             СЕРВЕР
ESTABLISHED                      ESTABLISHED
    |                                 |
    | CLOSE                           |
    | snd FIN                         |
    v                                 |
FIN-WAIT-1 ----[8. FIN]---->        |
    |                                 |
    |                          rcv FIN, snd ACK
    |                                 v
    |                            CLOSE-WAIT
    |                                 |
    |<---[9. ACK]----                |
    |                                 | CLOSE
    v                                 | snd FIN
FIN-WAIT-2                           |
    |                                 v
    |                            LAST-ACK
    |<---[10. FIN]----               |
    |                                 |
    | rcv FIN, snd ACK
    v
TIME-WAIT ----[11. ACK]--->    CLOSED
    |                           (TCB удален)
    | (ждет 2×MSL = ~60 сек)
    v
  CLOSED
  (TCB удален)

```


**Пошагово:**

1. **Клиент** отправляет `[FIN]`. Переходит в **FIN-WAIT-1**. («Я всё сказал, хочу закрыться»)
    
2. **Сервер** получает `[FIN]`, отправляет `[ACK]`. Переходит в **CLOSE-WAIT**. (« Услышал, что ты закрываешься»)
    
3. **Клиент** получает `[ACK]`. Переходит в **FIN-WAIT-2**. («Ты получил мой FIN»)
    
4. **Сервер** завершает отправку своих данных и отправляет `[FIN]`. Переходит в **LAST-ACK**. («И я тоже закрываю»)
    
5. **Клиент** получает `[FIN]`, отправляет `[ACK]`. Переходит в **TIME-WAIT**. (« Подтверждаю твой FIN»)
    
6. **Сервер** получает `[ACK]` и сразу переходит в **CLOSED**. (TCB удален, соединение разорвано)
    
7. **Клиент** остается в **TIME-WAIT** ~60 секунд, затем переходит в **CLOSED**.
    

## 3.4. Редкий сценарий: Одновременное закрытие (CLOSING)

Если обе стороны попробуют закрыться одновременно:


```
СТОРОНА A                          СТОРОНА B
ESTABLISHED                      ESTABLISHED
    |                                 |
    | snd FIN                         | snd FIN
    v                                 v
FIN-WAIT-1                      FIN-WAIT-1
    |<---[FIN]-----|                 |
    |                                 |
    | rcv FIN                 rcv FIN |
    v                                 v
CLOSING                          CLOSING
    |                                 |
    |-----[ACK]---->                 |
    |<----[ACK]-----|                |
    |                                 |
    v                                 v
TIME-WAIT                        TIME-WAIT

```

Обе стороны переходят в **CLOSING**, не проходя через FIN-WAIT-2 и CLOSE-WAIT.

## 4. Состояние TIME-WAIT: Особенности и назначение

## 4.1. Почему TIME-WAIT существует?

**Проблема 1:** Задержанные сегменты. Если новое соединение откроется с той же четвёркой адресов (IP_src:port_src, IP_dst:port_dst) сразу после закрытия старого, потерянные пакеты от старого соединения могут быть ошибочно приняты новым соединением.[](https://maxnilz.com/docs/004-network/003-tcp-connection-state/)​

**Проблема 2:** Потеря последнего ACK. Если финальный ACK потеряется, удаленная сторона остается в состоянии **LAST-ACK**. Если не ждать TIME-WAIT, новое SYN будет интерпретирован как ошибка, и удаленная сторона отправит RST.[](https://maxnilz.com/docs/004-network/003-tcp-connection-state/)​

**Решение:** TIME-WAIT гарантирует:

- Все задержанные сегменты от старого соединения будут отброшены (время жизни сегмента = MSL).
    
- Удаленная сторона успеет пересмотреть свой FIN несколько раз, если нужно, и получит новый ACK.
    

## 4.2. Длительность TIME-WAIT

RFC 793 рекомендует ждать **2 × MSL**:[](https://www.ietf.org/rfc/rfc793.txt)​

- **MSL (Maximum Segment Lifetime)** — максимальное время жизни сегмента в сети (обычно 30–120 секунд).
    
- На Linux это фиксировано на **60 секунд**:[](https://maxnilz.com/docs/004-network/003-tcp-connection-state/)​
    

`#define TCP_TIMEWAIT_LEN (60*HZ) // ~60 секунд`

**Практическое следствие:** Если закрыть тысячи соединений за секунду, можно собрать до 60,000 сокетов в TIME-WAIT, что израсходует ресурсы.[](https://maxnilz.com/docs/004-network/003-tcp-connection-state/)​

## 4.3. Только инициатор закрытия в TIME-WAIT

**Важно:** TIME-WAIT состояние достигается **только** стороной, которая первой инициировала закрытие (отправила первый FIN). Респондент переходит из LAST-ACK сразу в CLOSED.[](https://maxnilz.com/docs/004-network/003-tcp-connection-state/)​

## 5. Реакция на ошибки: Флаг RST

Если во время любого состояния приходит сегмент **[RST]**, соединение **немедленно** переходит в **CLOSED** без каких-либо уведомлений приложению.[](https://www.ietf.org/rfc/rfc793.txt)​

**Примеры:**

- Клиент пытается подключиться к порту, где нет слушающего сервера → сервер отправляет [RST].
    
- Сегмент пришел с неправильным номером последовательности → можно отправить [RST].

## 8. Вывод для экзамена

1. TCP соединение имеет **11 состояний**, отражающих этапы его жизни.
    
2. **Установление:** CLOSED → LISTEN/SYN-SENT → SYN-RECEIVED → ESTABLISHED (3-way handshake).
    
3. **Передача:** ESTABLISHED (обе стороны).
    
4. **Разрыв:** Инициатор идет FIN-WAIT-1 → FIN-WAIT-2 → TIME-WAIT; Респондент идет CLOSE-WAIT → LAST-ACK → CLOSED.[](https://maxnilz.com/docs/004-network/003-tcp-connection-state/)​​
    
5. **TIME-WAIT:** Только инициатор; ждет ~60 сек, чтобы убедиться, что удаленная сторона получила ACK.
    
6. **RST:** Можно отправить из любого состояния для немедленного разрыва.
