# Экзаменационный билет: Протоколы транспортного уровня (TCP и UDP)

## 1. Введение: Место и роль в модели OSI

Транспортный уровень (Layer 4) — это «мозг» передачи данных. Если сетевой уровень (IP) занимается тем, чтобы доставить пакет по адресу (маршрутизация), то транспортный уровень отвечает за то, **что делать с этими данными на конечном устройстве**.

**Ключевые функции:**

1. **Адресация приложений (Порты):** Распределение данных между конкретными программами (браузер, Telegram, игра) через порты (16-битные числа).
    
2. **Сегментация:** Разбиение больших данных на части (сегменты) перед отправкой и их сборка обратно.
    

---

## 2. Протокол UDP (User Datagram Protocol)

**UDP** — это протокол без установления соединения. Его часто называют протоколом «ненадежной доставки» или принципом «выстрелил и забыл».

## 2.1. Особенности

- **Отсутствие «рукопожатия»:** Данные отправляются сразу, без предварительной проверки готовности получателя.
    
- **Нет гарантий:** Протокол не проверяет, дошел ли пакет. Если пакет потерялся, UDP не будет его пересылать.
    
- **Сохранение границ:** UDP передает данные «пачками» (датаграммами). Если приложение отправило 100 байт, получатель считает 100 байт (а не поток, как в TCP).
    

## 2.2. Структура заголовка UDP

Заголовок UDP предельно прост и имеет фиксированный размер — **8 байт**. Это обеспечивает минимальную задержку при обработке.

|Поле (Field)|Размер|Описание|
|---|---|---|
|**Source Port**|16 бит|Порт отправителя. Указывает, какой процесс отправил данные.|
|**Destination Port**|16 бит|Порт получателя. Указывает, какому процессу адресованы данные.|
|**Length**|16 бит|Общая длина датаграммы (Заголовок + Данные).|
|**Checksum**|16 бит|Контрольная сумма. Используется для проверки целостности заголовка и данных (защита от битовых ошибок).|

> **Где применяется:** Видеоконференции (Zoom, Skype), стриминг, онлайн-игры (где важна реакция, а не 100% точность), DNS-запросы.

## 3. Протокол TCP (Transmission Control Protocol)

**TCP** — это ориентированный на соединение протокол, обеспечивающий гарантированную доставку данных в правильном порядке. Это сложный механизм, превращающий ненадежную IP-сеть в надежный канал связи.

## 3.1. Структура заголовка TCP

Заголовок TCP намного сложнее UDP. Его минимальный размер — **20 байт**, максимальный — **60 байт** (за счет поля Options).

**Визуальная схема заголовка:**

|Поле (Field)|Размер|Описание|
|---|---|---|
|**Source Port**|16 бит|Порт отправителя.|
|**Destination Port**|16 бит|Порт получателя.|
|**Sequence Number** (SEQ)|32 бита|**Порядковый номер.** Если отправляется файл, каждый байт нумеруется. Поле указывает номер _первого_ байта данных в этом сегменте. Это позволяет собирать данные в правильном порядке.|
|**Acknowledgment Number** (ACK)|32 бита|**Номер подтверждения.** Содержит номер следующего байта, который получатель _ожидает_ получить. (Говорит: «Я получил всё до байта X, жду X+1»).|
|**Data Offset**|4 бита|Длина заголовка (указывает, где начинаются полезные данные). Измеряется в 32-битных словах.|
|**Reserved**|3-6 бит|Зарезервировано для будущего использования (всегда нули).|
|**Control Flags**|6-9 бит|**Флаги управления** (см. ниже). Определяют тип сегмента.|
|**Window Size**|16 бит|**Размер окна.** Количество байтов, которое получатель готов принять прямо сейчас без подтверждения (Flow Control).|
|**Checksum**|16 бит|Контрольная сумма заголовка и данных.|
|**Urgent Pointer**|16 бит|Указатель важности. Используется, если нужно передать экстренные данные вне очереди (используется редко).|
|**Options**|0-40 байт|Дополнительные параметры (например, максимальный размер сегмента MSS).|

## 3.2. Ключевые флаги (Control Flags)

Именно флаги управляют состоянием соединения:

- **SYN (Synchronize):** Используется для начала соединения.
    
- **ACK (Acknowledgment):** Подтверждение получения данных (присутствует почти во всех пакетах после начала связи).
    
- **FIN (Finish):** Запрос на корректное завершение соединения (нет больше данных).
    
- **RST (Reset):** Экстренный разрыв соединения (при ошибке).
    
- **PSH (Push):** Требование немедленно передать данные приложению, не буферизуя их.
    

## 3.3. Механизмы работы TCP

1. **Установление соединения (Three-Way Handshake):**
    
    - КЛИЕНТ -> `SYN` -> СЕРВЕР («Хочу подключиться»)
        
    - СЕРВЕР -> `SYN, ACK` -> КЛИЕНТ («Слышу тебя, готов подключиться»)
        
    - КЛИЕНТ -> `ACK` -> СЕРВЕР («Ок, соединение установлено»)
        
    - _Только после этого идут данные._
        
2. **Обеспечение надежности (ARQ - Automatic Repeat reQuest):**
    
    - Отправитель запускает таймер при отправке сегмента. Если `ACK` не пришел до истечения таймера, сегмент считается потерянным и отправляется повторно.
        
3. **Управление потоком (Flow Control):**
    
    - Использует поле **Window Size**. Если буфер получателя переполнен, он отправляет пакет с меньшим значением окна (вплоть до 0), прося отправителя «притормозить».
        
4. **Управление перегрузкой (Congestion Control):**
    
    - Защищает саму сеть (роутеры) от перегрузки. TCP начинает передачу медленно (_Slow Start_), экспоненциально увеличивая скорость, пока не начнутся потери пакетов. При потере пакета скорость резко сбрасывается.

## 4. Сравнение и современные альтернативы

|Критерий|UDP|TCP|
|---|---|---|
|**Надежность**|Нет|Есть (гарантия доставки, порядка, целостности)|
|**Тип соединения**|Без соединения|Установление соединения (Handshake)|
|**Скорость**|Высокая|Ниже (из-за «рукопожатий» и подтверждений)|
|**Размер заголовка**|8 байт (фиксированный)|20–60 байт|
|**Применение**|DNS, DHCP, VoIP, Игры, Streaming|Web (HTTP), Email (SMTP), FTP, SSH|

**Современные протоколы (Advanced):**  
В лекциях и современной практике упоминаются протоколы, устраняющие недостатки TCP:

- **SCTP:** Обеспечивает надежность, как TCP, но поддерживает **multihoming** (работа через несколько сетевых интерфейсов одновременно для отказоустойчивости).
    
- **QUIC (основа HTTP/3):** Работает **поверх UDP**, но реализует надежность и шифрование внутри себя. Это позволяет загружать веб-страницы быстрее, так как устраняет долгую процедуру рукопожатия TCP.

## Вывод для ответа:

Транспортный уровень балансирует между надежностью и скоростью. **TCP** — это сложный «грузовик» с накладными, который гарантирует, что груз доедет в сохранности (используя сложные заголовки с номерами последовательности и подтверждениями). **UDP** — это «мотоцикл», быстрый и легкий (всего 8 байт заголовка), но без страховки груза. Выбор протокола зависит исключительно от требований приложения.
