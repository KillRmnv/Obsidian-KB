**Ковариантность** — это способность использовать более производный тип (подтип) там, где изначально предполагался базовый тип. В Java этот термин встречается в трех основных контекстах: массивы, возвращаемые значения методов и генерики (Generics).

Простыми словами: если $B$ является подтипом $A$, то и контейнер $f(B)$ считается подтипом $f(A)$.

---

### 1. Ковариантность возвращаемых типов

Это самое частое и полезное проявление ковариантности. При переопределении метода в наследнике вы имеете право изменить возвращаемый тип на более специфичный.

Java

```
class Animal {
    Animal giveBirth() { return new Animal(); }
}

class Cat extends Animal {
    @Override
    Cat giveBirth() { return new Cat(); } // Ковариантность: Cat вместо Animal
}
```

**Почему это удобно?** Вам не нужно делать приведение типов вручную, когда вы работаете напрямую с объектом `Cat`.

---

### 2. Ковариантность массивов

В Java массивы являются ковариантными. Это означает, что если `Integer` наследуется от `Number`, то и `Integer[]` считается подтипом `Number[]`.

Java

```
Number[] numbers = new Integer[10]; // Работает
numbers[0] = 10.5; // Ошибка во время выполнения (ArrayStoreException)!
```

> **Важно:** Ковариантность массивов считается небезопасной. Компилятор разрешит вам положить `Double` в массив, который на самом деле является `Integer[]`, если ссылка на него имеет тип `Number[]`. Ошибка вылетит только в Runtime. Это одна из причин, почему появились генерики.

---

### 3. Ковариантность в Generics (`? extends T`)

По умолчанию генерики **инвариантны**. Это значит, что `List<Integer>` **не является** подтипом `List<Number>`. Чтобы разрешить ковариантность, используются вайлдкарды с верхней границей: `<? extends T>`.

Java

```
List<? extends Number> list = new ArrayList<Integer>(); // Теперь это работает
```

**Особенности `? extends T` (Upper Bounded Wildcard):**

- **Чтение:** Безопасно. Мы точно знаем, что в списке лежат объекты, которые являются `Number` (или его наследниками).
    
- **Запись:** **Запрещена** (кроме `null`). Компилятор не знает, какой именно подтип `Number` там находится. Если бы мы попытались добавить `Double` в `List<Integer>`, это нарушило бы типобезопасность.

### Сводная таблица

|**Контекст**|**Тип связи**|**Безопасность**|
|---|---|---|
|**Массивы**|Ковариантны по умолчанию|**Небезопасно** (ошибка в Runtime)|
|**Возвращаемые типы**|Ковариантны|**Безопасно**|
|**Генерики (по умолчанию)**|Инвариантны|**Безопасно**|
|**Генерики (`? extends T`)**|Ковариантны|**Безопасно** (только для чтения)|

---

### Для чего это нужно?

Ковариантность позволяет писать более гибкий и переиспользуемый код. Например, метод `sum(List<? extends Number> list)` сможет обработать и список `Integer`, и список `Double`, не требуя создания отдельных методов для каждого типа.
